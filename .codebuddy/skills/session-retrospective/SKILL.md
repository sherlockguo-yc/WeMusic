---
name: 会话复盘
description: >
  当用户说「复盘 / 总结问题 / 这次有什么问题 / 回顾一下」时触发。
  回顾当前会话的完整对话历史，从 Agent、用户、配置三个维度分析问题，
  输出结构化复盘报告，并为每个发现的问题提出可执行的改进建议。
---

## 会话复盘工作流

### 步骤 1：回顾会话历史

扫描当前会话的完整对话历史，识别以下模式：

**Agent 端问题**：
- 重复修复的模式——同一个文件/同一个功能被反复修改超过 2 次
- 猜测导致的错误——Agent 在不确定时做了假设，后续被纠正
- 违反已有 Rules——做了某条 Rule 明确禁止的事
- 低效的修改——改了又改回去了，说明方案一开始就不对
- **对工具/架构的错误假设**——对 Skill、Rule、API、构建工具等能力的判断错误，导致方案建立在错误前提上。标志：后续交互中被迫推翻之前的核心论点
- **文档/产物大幅返工**——同一产物（文档、代码块）被全量重写超过 2 次，说明前期需求理解或方案设计不充分

**用户端问题**：
- 需求不够清晰——用户多次补充/修正需求描述
- 信息缺失——Agent 反复追问同一个问题用户才给出信息
- 用户被迫纠正 Agent 的错误假设——用户花费轮次纠正 Agent 对工具/项目的错误认知

**配置端问题**：
- 缺失的 Rule——某个错误如果有一条 Rule 就能避免
- 矛盾的 Rule——两条 Rule 对同一情况给出了冲突的指引
- 失效的 Rule——有一条 Rule 应该生效但没有阻止某个错误
- **未探索的可行路径**——Agent 过早宣告某条路径不可行，但实际存在更简单的解决方案

### 步骤 2：分类并评分

对每个发现的问题标注：
- 严重程度：🔴 高（导致了功能 bug 或部署失败）/ 🟡 中（增加了交互轮次但最终修复）/ 🟢 低（轻微的不一致）
- 类型：Agent 实现 / 用户沟通 / 配置缺失

### 步骤 3：提出改进建议

对每个问题给出具体、可执行的建议：

- Agent 端问题 → "建议创建/修改 Rule：[规则文件名]，内容为：[建议内容]"
- 用户端问题 → "建议改进沟通方式：[具体建议]"
- 配置端问题 → "建议修改 [Rule/Skill 文件名]：[具体修改内容]"

### 步骤 4：输出报告

```markdown
# 会话复盘报告

> 会话时间：[自动填入]
> 主要任务：[自动填入]

## 发现的问题

| # | 严重程度 | 类型 | 问题 | 建议 |
|---|---------|------|------|------|
| 1 | 🟡 | Agent | XX 规则被违反 | 该规则已存在但描述不够清晰，建议补充具体示例 |
| 2 | 🔴 | 配置 | 缺少 XX 规则 | 建议创建 Rule：[内容] |

## 统计

- 总修改次数：N
- 重复修改次数：N（浪费约 N 轮交互）
- 可预防的问题：N（如果已有合适规则）

## 行动项

- [ ] 创建 Rule：XXX
- [ ] 修改 Rule：XXX
- [ ] 更新 Skill：XXX
```

### 步骤 5：执行行动项

将报告中的行动项列出，逐一询问用户是否现在执行。用户确认的，立即创建/修改对应的 Rule 或 Skill。

### 自我进化

复盘 Skill 本身也需要迭代。每次复盘结束后，检查：
- 本次复盘是否发现了上一次复盘没有覆盖的问题类型？
- 复盘流程是否有遗漏的检查维度？
- 如果有，更新本 Skill 的检查清单。
