## 经验沉淀工作流

### 步骤 1：识别并分类

根据用户使用的关键词，匹配分类决策表：

| 用户说 | → 分类 | → 存放位置 |
|--------|-------|-----------|
| "以后都这么写" / "这是规范" / "标准做法" | 代码规范 | `.codebuddy/rules/` |
| "记住这个坑" / "又出现了" / "上次也是" | 历史教训 | `.codebuddy/rules/` |
| "统一用" / "约定" / "大家都这么写" | 项目约定 | `.codebuddy/rules/` |
| "我习惯" / "我喜欢用" / "偏好" | 个人偏好 | Memory |
| "这个项目用的是" / "部署在" / "使用的技术栈" | 项目背景 | Memory |

### 步骤 2：重复检查（存入 Rules 前）

在创建任何新 Rule 之前，扫描已有 Rules：

1. 列出已有 Rules 的文件名和一句话描述
2. 判断新规则是：
   - **全新的** → 直接创建
   - **与 XX 规则部分重叠** → 报告："这条规则与现有的 [XX.mdc] 存在部分重叠，是否合并到已有规则中？还是单独创建？"
   - **已被 XX 规则覆盖** → 报告："这条规则已被 [XX.mdc] 覆盖了，不需要新建。是否需要更新已有规则以更明确表达？"

### 步骤 3：向用户确认

格式：
"分析结果：这属于 **[分类]**，建议「[合并到已有规则 / 新建 Rule / 存入 Memory]」。
- 如果是新建 Rule：文件名建议 `[xxx].mdc`，确认吗？
- 如果是合并到已有 Rule：将在 `[已有文件名]` 中追加以下内容：[预览]"

### 步骤 4：执行写入

用户确认后执行。如果是创建/修改 Rule，进入步骤 5。

### 步骤 5：追加决策上下文（按需）

只有当 Rule 正文**没有解释「为什么这样选」**时，才追加 HTML 注释形式的决策上下文。

判断标准：如果规则正文已经说清楚了原因（如「CSS 布局混用禁止」的背景段、「Vite URL」的错误分析），**不需要**追加。追加决策上下文只在以下情况有价值：背后的决策逻辑没有写入正文，时间久了会忘。

如果确实需要，追加以下格式：

```html
<!--
┌─ 决策上下文 ────────────────────────────┐
│ 日期: [当前日期]                         │
│ 触发: [具体的问题场景]                    │
│ 方案A: [备选方案]                        │
│ 方案B: [备选方案]                        │
│ 选择: [选择了哪个]                        │
│ 理由: [为什么选这个]                      │
│ 重评条件: [什么情况下需重新评估]           │
└──────────────────────────────────────────┘
-->
```

### 步骤 6：关联更新

存储完成后，检查是否需要同步更新：
- `docs/AI协作方法论.md` — 如果沉淀的是一个新的协作模式或教训
- `docs/人机协作深度实践.md` — 如果沉淀的是新的工作流方法
