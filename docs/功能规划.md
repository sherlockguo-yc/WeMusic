## 高优

> 最优先启动的功能，核心用户价值和产品增长驱动力。

### 3. 社交功能

> 基于 WeMusic 现有 JWT 用户体系，构建站内社交功能。无需依赖微信登录，用户通过邮箱/用户名注册即可使用全部社交能力。微信登录仅是降低注册门槛的可选增强，非社交功能的前置条件。

#### 概述

利用已有用户系统，构建站内社交以提高用户黏性：歌单分享与发现、评论互动、关注/粉丝、动态时间线、一起听。类似小红书等站内社交模型——用户体系自闭环，不依赖第三方账号。

#### 功能方向

- **歌单分享与发现**：公开分享歌单，热门歌单/编辑推荐/分类浏览。
- **评论与互动**：对歌曲/歌单评论、点赞、回复。
- **用户主页**：展示公开歌单、最近播放、收藏歌曲、关注/粉丝机制。
- **一起听**：实时同步播放，进度同步，创建听歌房间邀请好友。
- **动态/时间线**：展示好友最近听的歌、创建的歌单、分享的歌曲。

#### 技术考量

- 已有 JWT 认证 + 用户注册/登录体系，无需额外接入第三方账号系统即可启动。
- 需要数据库新增表：用户关系表（关注/粉丝）、评论表、动态表。
- WebSocket 用于「一起听」实时同步。
- 建议分阶段：歌单公开分享 + 歌曲分享 → 评论/关注/用户主页 → 动态时间线 → 一起听。
- 后续可接入微信登录（见搁置需求）降低注册门槛，但不影响社交功能本身的实现和上线。

#### 待确认

- [ ] MVP 选哪几个先上
- [ ] 关注/粉丝模型：单向关注（Twitter 模型）vs 双向好友（微信模型）
- [ ] 是否需要独立后端服务（WebSocket 部分）
- [ ] 一起听的房间人数上限
- [ ] 动态是推模式（写入时生成 feed）还是拉模式（读取时聚合）

### 3. 本地音乐 / 离线缓存

#### 概述

目前所有音频都依赖 Bilibili 在线流，每次播放都需要请求音频代理。如果 B 站视频被下架或网络不稳定，歌曲就无法播放。支持离线缓存能大幅提升可用性。

#### 功能方向

- **离线缓存**：已播放过的歌曲自动缓存音频流到本地（IndexedDB 或 OPFS）。缓存管理：查看大小、按歌曲/时间清理、一键清空。弱网/离线自动降级到缓存播放。
- **本地音乐上传**（二期）：支持上传 MP3/FLAC/WAV/AAC，自动解析 ID3 标签，与 QQ 音乐元数据匹配。

#### 思考

离线缓存更轻量：歌曲元数据已在服务端，只需缓存音频流。本地音乐上传需要处理文件上传、ID3 解析、存储空间管理，复杂度高不少。**建议：先做离线缓存，再做本地上传。**

#### 待确认

- [ ] 离线缓存的存储限制（建议 2-5GB 上限）
- [ ] 是否需要支持 PWA 的 Service Worker 缓存策略
- [ ] 本地音乐上传的文件大小限制（FLAC 单曲可达 30-50MB）

### 4. 音效增强

#### 概述

当前播放器是纯原声输出，没有任何音频后处理。加入基础音效能力可以显著提升听感，特别是对于音质差异较大的 B 站视频源。

#### 功能方向

- **淡入淡出**：切歌时平滑过渡（crossfade），可配置过渡时长（0/3/5/8/10 秒）。
- **均衡器（EQ）**：预设曲线（流行/摇滚/古典/人声/电子）+ 自定义 10 段图形 EQ。
- **音量标准化**：基于 EBU R128 或 ReplayGain 进行响度归一化，解决不同 B 站视频源音量差异巨大的问题。

#### 思考

均衡器和淡入淡出都可以用 Web Audio API（`BiquadFilterNode` + `GainNode`）在前端实现，无需后端改动。**淡入淡出对体验提升最直接、实现成本最低，建议作为第一步。**

#### 待确认

- [ ] EQ 用 Web Audio BiquadFilter 还是第三方库
- [ ] 音量标准化是实时处理还是预处理（第一次播放时分析 + 缓存响度值）
- [ ] 淡入淡出和视频源切换的交互

### 5. 智能推荐增强

#### 概述

现有的推荐系统已经用了播放历史 + 喜欢 + 歌单 + 完播率的协同思路，可以进一步挖掘场景化推荐和每日推荐。

#### 功能方向

- **每日推荐歌单（日推）**：每天自动生成 20-30 首歌，考虑最近7天未播放、偏好流派、播放时段偏好。日推历史可回溯。
- **场景/心情歌单**：预设场景标签（运动/学习/睡前/通勤），自动生成对应推荐。
- **相似歌曲**：播放页底部展示"相似歌曲"列表，一键加入队列。

#### 思考

日推是网易云音乐的核心功能，用户黏性极高。但需要足够的数据积累才有意义。实现上可复用现有推荐引擎的大部分逻辑，主要是时间窗口和新鲜度权重的调整。

#### 待确认

- [ ] 日推的生成时机：每天凌晨定时生成 vs 用户首次打开时生成
- [ ] 相似歌曲的数据来源：纯本地计算 vs 外部 API
- [ ] 场景标签：AI 模型还是简单规则匹配

### 6. 播放体验增强

#### 概述

在现有播放控制基础上增加专业用户（学歌、扒谱、练琴）和日常用户都需要的实用功能。

#### 功能方向

- **AB 循环**：在进度条上标记 A/B 点，循环播放区间。快捷键 `[` 设 A、`]` 设 B。典型场景：学歌、乐器练习跟弹。
- **变速播放**：0.5x ~ 2x，保持音调不变（pitch-preserving）。典型场景：慢速听细节、快速浏览长歌。
- **定时开机播放**：设置指定时间自动开始播放（当闹钟），音量渐强（fade in 30s/60s）。

#### 思考

AB 循环和变速播放都可以用 Web Audio API 实现。Chrome 的 `<audio>` 标签已原生支持 pitch-preserving 变速。定时开机播放作为 Web 应用有一定局限，配合 PWA + Periodic Background Sync 可能实现。

#### 待确认

- [ ] 变速播放：浏览器原生支持足够还是需要 Web Audio 额外处理
- [ ] 定时开机播放：是否需要桌面客户端才能实现
- [ ] AB 循环 UI 放哪：进度条上标记 vs 播放控制区

### 7. 音乐收藏夹增强

#### 概述

当前的"我喜欢"是扁平列表，歌曲多了之后很难管理。标签系统和智能歌单可以提升整理效率。

#### 功能方向

- **标签系统**：手动打标签（如 #摇滚 #治愈 #开车听），多标签支持，侧边栏按标签筛选，标签颜色自定义。
- **自动标签**：基于 QQ 音乐流派/风格字段自动打标签，基于播放时段打时间标签。
- **智能歌单**：基于标签+规则的动态歌单（如"标签=#摇滚 AND 时长<5:00"），实时计算无需手动维护。

#### 思考

标签系统投入产出比极高：实现简单（几个数据库字段 + UI），但能显著提升大规模收藏的可用性。智能歌单让用户从"维护歌单"变成"定义规则"。建议先做手动标签，再做自动标签和智能歌单。

#### 待确认

- [ ] 标签存储：单独 tags 表还是 JSON 字段
- [ ] 智能歌单的规则 DSL 设计
- [ ] 标签是否支持导入/导出

### 8. 歌词增强

#### 概述

当前已有 LRC 滚动歌词和换源功能，在展示质量和便捷性上可进一步提升。

#### 功能方向

- **双语歌词/翻译**：如果网易云歌词源提供了翻译，展示双语对照。允许用户手动添加/编辑翻译。
- **桌面歌词**：迷你悬浮歌词窗口，始终置顶（Web 版用迷你窗口方案，完整支持需桌面客户端）。
- **纯音乐检测**：自动判断纯音乐跳过歌词搜索，判断依据：歌名关键词 + 歌词源返回空 + 时长特征。

#### 思考

双语歌词和纯音乐检测实现成本低且不依赖打包，优先做。桌面歌词在 Web 应用中的实现受限——浏览器无法创建置顶悬浮窗，真正实现需要 Electron/Tauri。

#### 待确认

- [ ] 桌面歌词：Web 版迷你窗口 vs 未来桌面客户端
- [ ] 双语歌词翻译来源：网易云自带翻译 vs 翻译 API
- [ ] 纯音乐检测的准确率阈值

### 9. 音频可视化

#### 概述

在播放页面加入实时音频频谱/波形可视化，增强沉浸感和视觉吸引力。

#### 功能方向

- **频谱分析仪（Spectrum）**：经典柱状频谱，多种配色（跟随主题色/彩虹/单色）和样式（柱状/波浪线/圆形雷达）。
- **波形图**：整首歌的波形预览（类似 SoundCloud），可点击跳转，已播放部分变色。
- **粒子/氛围效果**：频谱驱动的动态粒子动画，可替代静态封面作为播放页背景。

#### 思考

纯前端功能，用 Web Audio API 的 `AnalyserNode` 即可实现。频谱分析仪实现最简单（几十行代码），效果直观。波形图需用 OfflineAudioContext 预分析。**建议路径：频谱 → 波形 → 粒子效果。**

#### 待确认

- [ ] 可视化组件位置：替代封面 / 封面下方 / 全屏背景
- [ ] 波形数据是否缓存到服务端
- [ ] 低端设备是否自动降级

### 10. 数据迁移与导入

#### 概述

音乐用户的歌单和收藏沉淀在多个平台，迁移壁垒是切换到新播放器的最大障碍。

#### 功能方向

- **网易云音乐导入**：解析歌单链接，导入歌曲列表。支持导入"我喜欢的音乐"（需要 cookie/token）。
- **QQ 音乐导入**：已有歌单链接解析，增强为批量导入。支持导入收藏/收听历史。
- **通用导入格式**：支持 JSON/CSV/TXT（歌名-歌手）导入。
- **迁移报告**：显示成功/失败匹配数，一键导出未能匹配的歌曲。

#### 思考

核心挑战是跨平台歌曲匹配——网易云和 QQ 音乐的元数据存在差异（繁简体、feat. 标注方式、remix 版本）。需模糊匹配 + 多种候选策略。Cookie/token 必须仅在浏览器端处理，不能上传到服务端。

#### 待确认

- [ ] Cookie/token 的处理方式：纯前端解析 vs 服务端代理
- [ ] 导入后歌曲如何标记来源
- [ ] 是否需要 Spotify/Apple Music 导入

### 11. 自动化与快捷指令

#### 概述

让用户通过 URL Scheme、PWA 快捷方式、自定义快捷键快速触发播放器操作，减少操作步骤。

#### 功能方向

- **URL Scheme**：`/play/song/123` 直接播放、`/search/晴天` 直接搜索、`/radio` 开启私人 FM。
- **PWA 快捷方式**：右键 PWA 图标 → "播放日推"/"打开我喜欢"。
- **键盘快捷键自定义**：支持多键组合，预设配置方案（Vim 风格/标准风格）。

#### 思考

实现成本低、收益大。特别是 URL Scheme 和快捷键自定义，能显著提升重度用户效率。

#### 待确认

- [ ] URL Scheme 是否需要在路由系统中做破坏性变更
- [ ] 快捷键自定义的冲突检测策略
- [ ] PWA 快捷操作数量限制（Chrome 最多 3 个）

### 12. 外部服务集成

#### 概述

和主流音乐社区/工具打通，让 WeMusic 的播放数据同步到其他平台，扩大生态影响力。

#### 功能方向

- **Last.fm Scrobbling**：播放记录自动上报 + Now Playing 状态同步。
- **Discord Rich Presence**：状态栏显示当前歌曲（歌名+歌手+封面+进度条）。
- **ListenBrainz / Libre.fm**：开源替代方案。
- **Webhook 通知**：播放事件时发 POST 请求，联动 Home Assistant/IFTTT（如播放音乐时自动调暗灯光）。

#### 思考

低成本高回报：实现简单（HTTP API 调用），但给用户"数据不锁死在 WeMusic"的信心。Last.fm 和 Discord 用户量大、API 文档完善，优先做。

#### 待确认

- [ ] Last.fm API key 的管理方式
- [ ] Discord Rich Presence 是否需要桌面客户端
- [ ] 是否支持"不 scrobble 某首歌"（私密收听）
- [ ] Webhook 签名验证

### 13. 音乐笔记 / 听歌日记

#### 概述

允许用户为歌曲写私人笔记，记录听歌时的心情、想法、场景。把听歌变成记录生活的方式。

#### 功能方向

- **歌曲笔记**：Markdown 笔记，关联时间戳 + 心情标记（开心/悲伤/平静/兴奋/怀旧），按时间/心情/歌手筛选。
- **听歌日记**：每日自动生成听歌摘要，用户可追加内容，导出为 Markdown。
- **记忆回顾**："去年的今天你听了什么"、"这首歌你第一次听是 2023 年 3 月 15 日"，随机推送旧笔记。

#### 思考

技术实现简单（文本+元数据存储），但体验价值很高。和现有播放统计互补：统计是量化数据，笔记是质化记忆。建议在播放页底部放小图标入口，hover 展开。

#### 待确认

- [ ] 笔记存储：服务端数据库 vs 纯本地 localStorage
- [ ] 笔记默认私人，是否支持公开
- [ ] Markdown 编辑器方案

### 14. 电台 / 漫游模式

#### 概述

"我不知道听什么"是音乐播放器最常见的痛点。电台模式可以在用户不主动选歌时自动播放合适的歌曲。

#### 功能方向

- **歌曲电台**：以当前歌曲为种子，基于歌手/流派/时长等维度找相似歌曲，支持点赞/点踩调整方向。
- **歌手电台**：播放该歌手及其相似歌手的歌曲，优先热门→穿插冷门→相似歌手。
- **流派/风格电台**：预设摇滚/电子/爵士/古典/说唱等频道。
- **私人 FM**：综合长期偏好的无限电台，已喜欢的歌降权，基于时段调整风格。

#### 思考

核心难点是"相似歌曲"计算。初期可做简单版本：同歌手热门歌 + 同流派热门歌混排，播放量做多样性采样。用户量大了之后协同过滤更有效。

#### 待确认

- [ ] 电台如何"无限"：预生成 50 首追加 vs 实时计算
- [ ] 点赞/点踩如何影响推荐（学习模型 vs 规则权重）
- [ ] 是否需要 skip 次数限制

### 15. 关注歌手

#### 概述

用户关注喜欢的歌手，当歌手发布新歌时收到通知。从"被动搜索"到"主动发现"。

#### 功能方向

- **关注歌手**：歌手页面一键关注，已关注列表（侧边栏入口），歌手动态（新歌/新专辑）。
- **新歌通知**：定时轮询关注歌手的最新歌曲，浏览器通知推送，支持每日/每周汇总。
- **歌手上新日历**：日历视图展示当月/当周新歌，点击直接播放。

#### 思考

价值取决于 QQ 音乐 API 是否能稳定返回"最新歌曲"数据。如果只能拿到全部歌曲列表，需本地差分计算。PWA + Service Worker 可实现离线通知。

#### 待确认

- [ ] QQ 音乐 API 是否支持按时间排序获取歌手歌曲
- [ ] 轮询频率：每小时/每天/每周
- [ ] 大量关注歌手（100+）的 API 限流问题

### 16. 协作歌单

#### 概述

多个用户共同编辑同一个歌单，实时同步所有修改。类似 Spotify Collaborative Playlist。

#### 功能方向

- **创建协作歌单**：开启协作模式，生成邀请链接/码，设置协作者权限（可编辑/仅查看/可添加不可删除）。
- **实时同步**：WebSocket/SSE 同步添加/删除/排序，冲突处理（LWW + 操作日志），"正在编辑"状态指示。
- **协作动态**：操作历史（谁在什么时候添加/删除了什么歌），支持 @某人 添加歌曲建议。

#### 思考

协作歌单是社交功能的自然延伸。可和社交功能一起规划：MVP 先做协作歌单 + 歌单公开分享，再做评论/关注。

#### 待确认

- [ ] WebSocket vs SSE 做实时同步
- [ ] 冲突处理策略：OT/CRDT/LWW
- [ ] 是否需要"锁定"功能防止多人同时编辑

### 17. 睡眠 / 专注模式

#### 概述

在定时停止基础上加入环境音、白噪音、番茄钟，让 WeMusic 帮助睡眠和专注工作。

#### 功能方向

- **环境音/白噪音**：内置雨声/海浪/篝火/森林/风扇/咖啡店，可叠加音乐同时播放（独立音量控制），无缝循环。
- **番茄钟专注模式**：25 分钟专注/5 分钟休息，自动切换专注歌单/放松音乐，完成统计。
- **睡眠增强**：定时音量渐弱（最后 5 分钟逐渐降低），睡眠专属 UI（暗色/极简/无蓝光），睡眠统计。

#### 思考

让 WeMusic 从"听歌工具"拓展为"生活方式工具"。环境音用两个 Audio 元素叠加实现。白噪音文件可从 freesound.org 获取。

#### 待确认

- [ ] 白噪音来源：内置打包（增加体积）vs 在线流（依赖网络）
- [ ] 是否和音乐共用音频播放器
- [ ] 番茄钟浏览器后台运行（Service Worker + Notification）

### 18. 无障碍与国际化

#### 概述

扩展可达用户群体：屏幕阅读器用户、非中文母语用户、移动端用户。

#### 功能方向

- **屏幕阅读器 A11y**：`aria-label`、键盘导航、`aria-live` 播报、高对比度主题。

- [ ] 移动端：单独 UI 布局 vs CSS 响应式
- [ ] 高对比度主题：独立选项 vs 跟随系统

### 19. 歌单模板与曲库健康

#### 概述

帮助用户维护干净的音乐库：检测重复歌曲、修正元数据错误、提供歌单模板。

#### 功能方向

- **重复歌曲检测**：检测同歌不同视频源、跨歌单重复，合并建议保留最佳视频源。
- **歌单模板**：预设"华语经典 100"/"欧美热单"/"日系 ACG"等，一键创建。
- **死链检测**：定期检查 B 站视频可用性，标记不可用，一键换源。
- **元数据修正**：允许用户手动修正不准确的元数据（歌手名/专辑信息）。

#### 思考

面向收藏了大量歌曲的重度用户。当歌单 500+ 首歌时，手动找重复和死链几乎不可能。元数据修正是争议功能——有维护成本但 QQ 音乐确实存在数据质量问题。

#### 待确认

- [ ] 重复检测粒度：完全匹配 vs 编辑距离
- [ ] 死链检测频率：每天/每周
- [ ] 歌单模板数据来源：硬编码 vs 实时拉取排行榜
- [ ] 元数据修正是否同步到服务端

## 管理功能

> 面向管理员和运营人员的后台管理能力。当前实现极其单薄（仅展示用户数/反馈数/总时长 + 用户列表 + 反馈删除，权限仅靠环境变量白名单），需系统性补全。本规划按模块拆分为独立的功能体系，不涉及具体实现方案。

### 现状诊断

| 维度 | 现状 | 差距 |
|------|------|------|
| 权限模型 | `ADMIN_USERNAME` 环境变量白名单，管理员/用户二元区分 | 无角色分级、无操作粒度控制 |
| 数据看板 | 三个静态数字（用户数/反馈数/总时长） | 无趋势图、无用户行为分析、无可视化 |
| 用户管理 | 列表展示（用户名/注册时间/播放时长/歌单歌曲数），只读 | 无法禁用/启用、无详情页、无批量操作 |
| 内容审核 | 仅反馈列表可删除；`blocked_sources` 表有数据但无 UI 入口 | 屏蔽源不可管理、无审核工作流 |
| 系统配置 | 所有配置依赖手动改 `.env` 重启 | 无运行时配置面板、无功能开关 |
| 操作审计 | 无 | 无操作日志、安全事件黑盒 |
| 系统监控 | 无 | 服务器/API/数据库状态完全不可见 |

### 模块 1：权限与角色管理

- **角色体系**：预置角色 — 超级管理员（Super Admin，最高权限）/ 管理员（Admin，日常运营）/ 内容审核员（Moderator，仅审核）/ 只读观察员（Viewer，仅查看）。支持自定义角色，选择可见模块与操作权限。
- **管理员管理**：从现有用户中委任/移除管理员角色。管理员列表展示角色、最后活跃时间、操作概览。支持管理员自助放弃权限。管理员会话管理：查看与撤销其他管理员的活跃会话。
- **操作级权限控制**：每个模块内区分「查看」「编辑」「删除」「导出」等操作，可针对角色精细配置。数据级权限：特定管理员只能查看自己负责范围的数据。
- **安全机制**：管理员登录可配置二次验证（TOTP 验证码）。敏感操作（删除用户、重置密码、批量操作、备份恢复）须二次确认。异常登录告警（异地 IP、非常规时段）。

---

### 模块 2：数据看板

- **实时概览卡片**：DAU / MAU / 新增注册 / 活跃播放用户数。实时播放数、当前在线用户数。歌曲/歌单/反馈/屏蔽源总量。播放分钟数 24h/7d/30d 汇总。
- **趋势图表**：用户增长（折线图，日/周/月粒度）。播放时长时段分布热力图。注册→活跃转化漏斗。新用户 Day-1 / Day-7 / Day-30 留存率。反馈提交趋势（按 Bug/需求/其他分组）。
- **内容分析**：热门歌曲 Top 50（按播放次数 / 播放时长排序）。热门歌手 Top 20。高频搜索词排行。播放源健康度：B 站视频可用率、平均加载耗时。红心/收藏热门排行。
- **用户画像**：活跃度分层（高活 / 中活 / 低活 / 沉睡用户）。听歌偏好分布（流派 / 语种 / 时段偏好）。人均使用时长、人均播放歌曲数。
- **数据导出**：看板数据导出 CSV/JSON。定时报告：周报/月报自动生成并推送。自定义时间范围筛选。

---

### 模块 3：用户管理增强

- **用户详情页**：完整播放历史（分页+搜索）、歌单及歌曲明细、反馈历史、登录历史（时间/IP/UA）、操作时间线（注册 → 首播 → 创建歌单 → 最近活跃）。
- **用户操作**：禁用/启用账号（禁用后立即踢出所有会话）。重置密码（生成临时密码或发送重置链接到绑定邮箱）。强制下线（撤销所有 JWT token）。编辑用户信息（修正用户名、清除违规头像）。
- **删除用户**：级联删除或匿名化关联数据。可选"先导出数据再删除"（合规需求）。须超级管理员二次密码验证。不可逆操作，有醒目警告。
- **批量操作**：条件筛选（注册时间范围 / 活跃状态 / 播放时长阈值 / 自定义标记），批量导出、批量禁用/启用、批量发送站内通知。
- **用户标记与备注**：管理员可为用户添加内部备注（仅管理员可见）。状态标记：正常 / 观察中 / 已警告 / 已封禁。按标记筛选用户列表。
- **列表增强**：服务端分页（避免大量用户数据撑爆前端）。可排序列（注册时间 / 播放时长 / 最近活跃）。搜索（用户名模糊匹配）。

---

### 模块 4：内容审核

- **屏蔽源管理**：屏蔽源列表（视频源 / 歌词源），按歌曲、屏蔽者、时间筛选。批量取消屏蔽。屏蔽原因分类（视频下架 / 音质差 / 内容不匹配 / 用户举报 / 其他）。自动标记规则（如同一视频被 N 个用户主动换源，自动标记「待审核」）。屏蔽源统计（原因分布饼图、增长趋势）。
- **歌单审核**（社交功能上线后适用）：公开歌单审核队列（标题/封面/简介违规检测）。举报处理工作流（接收 → 核实 → 处理 → 通知举报人）。官方精选歌单标记/置顶/推荐位管理。
- **评论/互动审核**（社交功能上线后适用）：评论审核队列。自定义敏感词库管理。用户禁言/解禁（按时间段或永久）。举报处理闭环。
- **用户头像/昵称审核**：违规头像标记 → 强制重置为默认头像。敏感昵称检测 → 强制修改。支持自动检测 + 人工复审双通道。
- **审核工作流通用能力**：待审核队列按优先级/时间排序。审核记录（谁/何时/什么操作/理由）。驳回时必填理由，通知提交者。批量审核（全选当前页通过/驳回）。

---

### 模块 5：系统配置

- **注册控制**：开放注册 / 仅邀请制 / 关闭注册（运行时切换，无需重启）。邀请码管理：批量生成、使用次数限制、有效期设置、使用记录查询。注册审批队列（管理员手动通过/拒绝，附理由）。IP 白名单/黑名单，邮箱域名限制。
- **功能开关（Feature Flags）**：运行时启用/禁用功能模块（歌词搜索、视频源切换、社交功能等）。灰度发布：按用户百分比或特定用户组逐步开启新功能。开关变更历史可追溯。
- **全局参数**：播放默认值（初始音量、淡入淡出时长）。API 请求频率限制（全局 + 单用户粒度）。缓存策略配置。推荐算法参数在线调整（多样性权重、新鲜度权重、冷门歌曲曝光率）。
- **公告管理**：创建/编辑/删除系统公告。公告展示位置：顶栏横幅 / 弹窗 / 播放页底部。定时发布 + 过期自动下架。目标用户范围：全部 / 新用户 / 活跃用户 / 自定义筛选。
- **备份与恢复**：手动触发完整数据库备份。每日自动备份（保留最近 N 份）。备份历史列表、下载备份文件。从备份恢复（须超级管理员验证，恢复前自动创建当前数据快照）。磁盘空间不足时拒绝备份并告警。

---

### 模块 6：操作审计

- **管理员操作日志**：记录关键操作及其上下文 — 操作人、目标对象、操作类型、操作时间、IP地址、变更前后对比。支持的操作类型：用户管理、内容审核、系统配置变更、权限变更、备份恢复。
- **安全事件日志**：管理员登录/登出记录。敏感操作记录。异常检测（异地 IP 登录、非常规时段操作、短时间内大量 403 请求）。API 访问异常统计。
- **日志查询与导出**：按操作人/操作类型/时间范围/目标对象多条件筛选。导出为 CSV/JSON。日志不可被管理员删除或篡改（追加写入 + 哈希链）。
- **日志保留策略**：可配置保留天数，超期自动清理。在清理前可触发归档导出。

### 模块 7：系统监控

- **服务健康**：CPU / 内存 / 磁盘使用率实时展示。Node.js 进程指标（事件循环延迟、堆内存、GC 频率）。数据库连接池状态、慢查询日志。
- **API 性能**：各端点 P50 / P95 / P99 响应时间。QPS 实时趋势图。错误率监控（4xx / 5xx 分布）。第三方 API 调用成功率（QQ 音乐、B 站、网易云）。
- **后台任务**：任务状态面板（歌词抓取、视频源搜索队列长度）。失败任务告警与重试管理。单次任务耗时统计。
- **告警规则**：可配置阈值 — API 错误率 > 阈值 / 磁盘使用率 > 阈值 / 连续 N 分钟无播放请求 / 数据库连接池耗尽。告警通知方式（站内 / 邮件 / Webhook / 企业微信机器人）。

### 核心交互流程

#### 流程一：内容审核闭环（以屏蔽源审核为例）

```
用户举报视频源
  → 系统自动创建审核项，聚合同一视频源的多条举报
  → 审核员登录后在「内容审核」看到待处理队列
  → 展开审核详情（歌曲信息、视频源信息、举报人数、举报原因分布、举报人列表）
  → 操作：
      确认屏蔽 → 视频源加入 blocked_sources → 通知举报人「已处理」
      驳回举报 → 标记为「误报」→ 通知举报人
      标记「待补充」→ 退回
  → 操作写入审计日志
```

#### 流程二：运营日常巡检

```
管理员登录 → 数据看板概览页
  → 检查核心指标是否有异常波动
  → 发现播放量骤降 → 点击进入趋势详情
    → 对比「热门歌曲 Top 50」找出播放量下降最大的歌曲
    → 检查对应视频源的可用率 → 发现高播放歌曲的视频源大面积失效
    → 进入「内容审核」模块手动触发视频源健康检查
  → 导出本周数据作为周报素材
```

#### 流程三：功能灰度发布

```
超级管理员 → 系统配置 → 功能开关
  → 选择目标功能 → 「灰度发布」
  → 配置目标人群：20% 活跃用户（注册 > 7 天 + 近 3 天有播放记录）
  → 确认启用
  → 回到数据看板，观察该功能的使用率、错误率、用户反馈
  → 无异常 → 逐步提升至 50% → 100% 全量
  → 每一步变更写入审计日志
```

#### 流程四：用户数据合规处理

```
用户请求删除个人数据
  → 管理员在用户管理中搜索该用户
  → 查看用户详情确认身份
  → 「导出用户数据」→ 生成包含播放历史、歌单、反馈的完整 JSON
  → 用户确认数据无误
  → 「删除用户及关联数据」→ 系统展示将被删除的数据量清单
  → 超级管理员二次密码验证
  → 执行删除（或匿名化保留公共贡献）
  → 写入审计日志
```

#### 流程五：备份与灾难恢复

```
日常：系统每日 03:00 自动备份 → 保留最近 7 份 → 超期自动清理

灾难恢复：
  超级管理员进入「备份与恢复」
  → 上传备份文件或选择已有备份
  → 系统校验备份文件完整性与 schema 版本兼容性
  → 弹出最终确认：「即将覆盖当前数据库，系统会先创建恢复前快照」
  → 二次密码验证
  → 执行恢复 → 服务自动重载数据
  → 写入审计日志
```

### 需确认的细节与待解决问题

**权限模型**：
- [ ] 是否需要支持「角色模板」一键套用，方便跨环境部署？
- [ ] 管理员数量是否有上限？权限扩散如何约束？
- [ ] 唯一的超级管理员账号丢失怎么办？是否需要「紧急恢复码」机制？
- [ ] 是否需要审批流（如删除用户须另一管理员确认）？

**数据看板**：
- [ ] 历史统计数据保留多久？是否需要物化视图/定时聚合以加速查询？
- [ ] 是否需要同比/环比对比能力？
- [ ] 看板数据更新方式：WebSocket 实时推送 vs HTTP 定时轮询？

**用户管理**：
- [ ] 删除用户后，其公开歌单如何处理（保留但匿名化 / 一并删除）？
- [ ] 用户数据导出格式需兼容哪些外部平台？
- [ ] 账号禁用后，已有 JWT token 立即失效还是等待过期？

**内容审核**：
- [ ] 敏感词库来源：内置预置 + 自定义 vs 对接第三方内容审核 API？
- [ ] 头像审核是否需要对接第三方图片检测（如腾讯云内容安全）？

### 关键边界情况与异常场景

| 场景 | 风险等级 | 影响 | 处理策略 |
|------|:--:|------|---------|
| 唯一超级管理员被误删 | 🔴 严重 | 系统失去最高权限控制 | 环境变量 `ADMIN_USERNAME` 始终作为兜底，不可被 Web 端降级或删除；可选「紧急恢复码」机制 |
| 数据看板查询时间范围过大 | 🟡 中等 | 查询超时、管理面板卡死 | 默认限制 30 天，更长范围需手动确认；大查询异步生成 + 缓存 |
| 备份时磁盘空间不足 | 🔴 严重 | 备份失败甚至损坏已有数据 | 备份前检查磁盘空间，不足则拒绝 + 告警；先写临时文件，完整后原子性重命名 |
| 管理员 session 被劫持 | 🔴 严重 | 用户数据泄露或恶意破坏 | IP 绑定验证；敏感操作须重新输入密码；异地 IP 变化时强制重新登录 |
| 注册策略过松导致机器人批量注册 | 🟡 中等 | 数据库膨胀、服务降级 | 注册量异常检测 + 告警；支持批量禁用 + 清理可疑账号；邀请码模式兜底 |
| 数据库 schema 升级后旧备份无法恢复 | 🔴 严重 | 灾难恢复失败 | 备份文件包含 schema 版本号；恢复前自动校验兼容性，不匹配时提示升级路径 |
| 用户列表数据量过大导致前端卡顿 | 🟡 中等 | 管理体验差 | 服务端分页（50 条/页）；虚拟滚动；服务端排序+筛选替代前端处理 |
| 反馈含敏感内容但被搜索引擎爬取 | 🟡 中等 | 隐私泄露风险 | 管理面板路由不暴露给搜索引擎（robots.txt + meta noindex）；删除后明确提示「永久删除」 |
| 多管理员并发编辑同一配置 | 🟡 中等 | 配置被覆盖 | 乐观锁（版本号校验）或「正在被 X 编辑」提示；关键配置项串行化 |
| 功能开关误关闭核心功能 | 🔴 严重 | 全站不可用 | 核心功能开关须二次确认；提供"安全模式"自动回退列表；开关变更告警通知 |
| 批量操作中途失败 | 🟡 中等 | 数据部分变更、状态不一致 | 批量操作须事务化或提供回滚能力；操作前展示受影响范围预览；失败后列出成功/失败清单 |
| 管理员新增后原用户密码泄露风险 | 🟡 中等 | 管理员权限被滥用 | 授予管理角色时强制要求该用户重新登录 + 可选的二次验证激活步骤 |

## 独立项目：监控系统

> 独立部署的可观测性平台。当前优先为 WeMusic 提供服务监控能力，未来通过标准化协议支持多系统接入。
>
> **设计原则**：独立部署（不耦合 WeMusic 业务逻辑）、标准协议（基于 OpenTelemetry / Prometheus 生态）、可扩展（多租户架构）、高可用（自身健康监控、数据持久化、故障自愈）。

### 整体架构

#### 技术选型

| 组件 | 推荐方案 | 说明 |
|------|---------|------|
| 指标采集与存储 | Prometheus + VictoriaMetrics（可选） | 社区标准，兼容 PromQL；数据量大时 VictoriaMetrics 更省资源 |
| 日志聚合 | Loki + Promtail | 轻量，与 Prometheus 统一标签体系，无全文索引开销 |
| 链路追踪 | OpenTelemetry SDK + Jaeger / Grafana Tempo | 厂商中立，避免 vendor lock-in |
| 可视化 | Grafana | 开箱即用的 Dashboard 生态，支持指标/日志/追踪联动 |
| 告警引擎 | Grafana Alerting + Alertmanager | 多通道通知、分组、抑制、静默、告警升级 |
| 配置存储 | PostgreSQL（多租户模式）/ SQLite（单租户 MVP） | 告警规则、通知渠道、租户配置 |

#### 架构分层

```

─────────────────────────────────────────────────────

─────────────────────────────────────────────────────

─────────────────────────────────────────────────────
采集层    OpenTelemetry Collector · Prometheus Exporters · Promtail
─────────────────────────────────────────────────────
被监控    WeMusic · 未来系统 A · 未来系统 B · ...
```

### 模块 1：数据采集与观测

#### 1.1 指标采集

**基础指标**（自动采集，无需业务改动）：
- 基础设施：CPU / 内存 / 磁盘 I/O / 网络吞吐 / 文件描述符（Node Exporter）
- Node.js 运行时：事件循环延迟（ELU）、堆内存使用与 GC 频率、活跃句柄数（prom-client 内置）

**WeMusic 自定义指标**（需在应用中埋点）：
- 业务指标：实时播放数、活跃用户数、搜索 QPS、歌单创建/收藏数
- API 质量：各端点 QPS / P50/P95/P99 延迟 / 错误率（4xx/5xx）
- 上游依赖：QQ 音乐/B 站/网易云 API 调用成功率与耗时
- 后台任务：歌词抓取/视频源搜索 队列深度、失败率、单次耗时
- 缓存健康：缓存命中率、过期率

#### 1.2 日志采集

- **接入方式**：Promtail 部署在被监控机器上，tail 日志文件推送至 Loki
- **采集范围**：应用日志（结构化 JSON）、访问日志（Nginx/Caddy）、错误日志、慢查询
- **日志增强**：自动注入 trace_id / span_id、主机名、服务名等标签；支持敏感信息脱敏
- **Grafana 联动**：从指标面板一键跳转至关联日志，按 trace_id 聚合查看完整请求链路日志

#### 1.3 链路追踪

- **实现方式**：OpenTelemetry SDK 在 Node.js 中自动插桩（HTTP/gRPC/SQL 调用自动追踪）
- **追踪范围**：入口（Nginx → Node.js）→ 业务逻辑 → 外部 API（QQ 音乐/B 站/网易云）→ 数据库
- **采样策略**：生产环境「尾部采样」——正常请求 10% 采样，错误/高延迟请求 100% 采样

#### 1.4 统一 Dashboard

- **服务概览大屏**：多系统实时状态一览，红绿灯健康指示，适合投屏展示
- **WeMusic 专用面板**：
  - 播放业务：实时播放数、热门歌曲 Top 20、播放源分布（B 站 vs 缓存）
  - API 质量：各端点延迟分位数、错误率趋势、上游依赖健康度
  - 后台任务：队列深度、耗时趋势、失败重试统计
- **基础设施面板**：主机资源、数据库连接池、Node.js 运行时

### 模块 2：智能告警

#### 2.1 告警模型

支持三种告警模型，可自由组合：

**阈值告警**：
- 静态阈值：API 错误率 > 5%、磁盘使用率 > 80%、内存持续 > 1GB
- 动态阈值：基于历史数据自动计算基线（过去 7 天同时段均值 ± 2 标准差），自适应波动
- 持续时间：条件满足须持续 N 分钟才触发（避免瞬时抖动误报）

**变化检测告警**：
- 环比突变：QPS 较上一小时骤降 > 50%
- 趋势预测：磁盘使用率预测 24 小时内将达 95%（线性回归）
- 同比偏差：流量与上周同期偏差 > 3 倍标准差

**组合告警**：
- AND/OR 逻辑组合多个条件
- 前置过滤（如仅业务高峰期生效）
- 依赖抑制（「QQ 音乐 API 不可用」自动抑制「歌词获取失败」告警，避免告警风暴）

#### 2.2 告警处理流程

```
异常检测 → 规则匹配 → 去重与抑制(同源合并/依赖抑制)
  → 静默检查(维护窗口跳过) → 分组(按服务/等级分组)
  → 延迟发送(等 N 分钟，恢复则取消) → 多通道通知
  → 告警升级(未确认则逐级上报，P0 可升级为电话通知)
```

#### 2.3 通知渠道

- 即时通：企业微信机器人、钉钉、飞书 Webhook
- 邮件：HTML 格式详情 + 每日汇总摘要
- Webhook：自定义回调，可对接 PagerDuty / Opsgenie / 自定义运维系统
- 站内通知：WeMusic 管理面板顶部告警横幅

#### 2.4 告警管理能力

- **规则管理**：CRUD + 启用/禁用/临时暂停，支持按时间段定时生效
- **静默窗口**：预设维护时段自动静默；支持手动一键静默指定时间
- **分组与抑制**：同服务多指标同时告警合并为一条通知；上游故障抑制下游级联告警
- **告警升级**：P0 5 分钟未确认 → 升级通知；P1 30 分钟 → 通知上级
- **告警历史**：完整记录触发/恢复时间、通知状态、确认人/时间，支持回溯分析

#### 2.5 预置告警规则（WeMusic）

| 告警名称 | 触发条件 | 等级 | 说明 |
|---------|------|:---:|------|
| 服务不可用 | 健康检查连续 3 次失败 | 🔴 P0 | 进程崩溃或端口未监听 |
| 播放功能异常 | 连续 5 分钟无成功播放事件 | 🔴 P0 | 核心功能受影响 |
| API 错误率过高 | 5 分钟内 5xx ≥ 10% | 🟠 P1 | 需排查服务或上游 |
| QQ 音乐 API 异常 | P95 延迟 > 5s 持续 10 分钟 | 🟠 P1 | 上游依赖劣化 |
| 磁盘空间不足 | 使用率 > 85% | 🟡 P2 | 需清理或扩容 |
| 内存泄漏风险 | 堆内存连续增长 1h 无 GC 回落 | 🟡 P2 | 潜在内存泄漏 |
| 后台任务积压 | 队列 > 200 持续 30 分钟 | 🟡 P2 | 处理能力不足 |
| 证书即将过期 | HTTPS 证书剩余 < 30 天 | 🟢 P3 | 提前提醒续期 |

### 模块 3：数据分析

#### 3.1 基础分析能力

- **时序分析**：任意指标时间趋势图，支持多指标叠加对比、同比/环比计算
- **聚合分析**：按服务/端点/时间维度聚合，Top N 排行
- **分布分析**：延迟分位数分布、请求体大小分布、用户活跃时长分布
- **关联分析**：两指标相关性（如 CPU 与 QPS、错误率与内存），辅助定位根因

#### 3.2 SLO / 错误预算监控

- **SLO 定义**：为目标服务设定可用性/延迟目标（如「99.9% 请求延迟 < 500ms」）
- **错误预算**：基于 SLO 计算剩余错误预算，预算耗尽前预警（燃尽图）
- **多窗口 SLO**：同时监控 7d / 30d / 90d 滚动窗口，平衡短期波动与长期趋势

#### 3.3 简易异常检测

- **统计异常**：基于 Z-Score / IQR 检测突刺和突降
- **季节性分解**：识别日周期/周周期模式，分离趋势与周期性成分
- **线性预测**：基于历史趋势预测资源耗尽时间（如「磁盘预计 7 天后满」）

#### 3.4 报表与导出

- **定时报表**：每周/每月自动生成健康报告（PDF/Markdown），含 SLA 达成、Top 告警、资源趋势、业务关键指标，推送至管理员
- **数据导出**：Grafana 原生导出（CSV/JSON/PNG）+ 开放 API 供外部系统消费

### 模块 4：多租户与可扩展性

#### 4.1 租户模型

- **租户** = 一个被监控的独立系统（WeMusic 是第一个租户）
- 每个租户独立拥有：指标命名空间、告警规则、通知渠道、Dashboard
- 租户间数据完全隔离，不可互相访问

#### 4.2 接入方式

**标准接入（推荐）**：
1. 被监控系统暴露 Prometheus `/metrics` endpoint
2. 监控系统注册新租户，提供 scrape 目标地址
3. 配置告警规则与通知渠道
4. 导入预置 Dashboard 模板（可自定义）

**轻量接入（短期任务/无法暴露 endpoint 的场景）**：
- Pushgateway 模式，主动推送指标
- OpenTelemetry SDK 集成，一键上报 traces 和 logs

#### 4.3 Dashboard 模板

- 「通用 HTTP 服务」：QPS / 延迟 / 错误率 / 状态码分布
- 「Node.js 应用」：ELU / GC / 堆内存 / 句柄数
- 「数据库」：连接池 / 慢查询 / 查询量趋势
- 管理员可分享/导入/导出 Dashboard JSON

### 21. KTV / 卡拉 OK 模式

#### 概述

结合已有视频播放和歌词功能，增加全屏视频 + 逐字高亮歌词 + 人声抑制。

- **趣味评分**：基于音高检测的简易评分系统。

#### 思考

### 22. AI 音乐助手

#### 概述

利用 LLM 增加智能交互：自然语言搜索、AI 歌单生成、AI DJ 解说。

#### 功能方向

- **自然语言搜索**："播放适合下雨天听的爵士乐"→ 自动生成播放列表。
- **AI 智能歌单生成**：给定主题（公路旅行/深夜emo/健身房），自动生成并附描述和封面。

#### 思考

### 1. 微信登录与账号绑定 ⛔ 阻塞

#### 阻塞原因（2026-07-04 确认）

以下三项硬性前提条件**均不具备**，当前无法推进：

| 前提条件 | 说明 | 状态 |
|---|---|---|
| 微信开放平台开发者账号 | 需要企业主体或个体工商户，不支持个人开发者 | ❌ 无 |
| 已备案域名 + HTTPS | OAuth 回调 `redirect_uri` 必须是线上域名，微信会校验 | ❌ 无 |
| 本地开发调试 | OAuth 需要线上回调，本地开发依赖内网穿透（ngrok/frp） | ❌ 公司安全策略禁止使用 |

微信 OAuth 2.0 协议本身决定了这三项缺一不可：没有线上回调地址，整个流程就无法完成。**此功能在获得上述前提条件前处于阻塞状态，暂时不进行实现。**

#### 替代思路（未验证）

- 如果未来有域名和服务器，可以部署一个极简的回调代理服务（如 Cloudflare Worker / Vercel Function）转发 OAuth 回调到内网服务器，绕过 ngrok/frp 的安全限制。但这仍需域名+备案。
- 微信以外是否有更轻量的第三方登录？（如 GitHub OAuth — 但面向国内用户不适用；手机验证码登录 — 需要短信服务商，也有成本）

#### 概述（原规划）

支持用户通过微信扫码登录，以及已有账号绑定微信，实现多端/多方式登录。微信是国内用户最常用的账号体系，接入微信登录可以大幅降低注册/登录门槛，为后续社交功能（评论、关注、分享）打好用户账号基础。

#### 功能方向（原规划）

- **微信扫码登录**：Web 端展示微信扫码页面，用户微信扫码授权后自动登录/注册。基于微信开放平台 OAuth 2.0。
- **绑定微信**：已注册用户可在设置中绑定微信账号，绑定后支持微信扫码登录同一账号。
- **解绑微信**：已绑定微信的用户可在设置中解绑，解绑后不影响通过其他方式登录。
- **微信用户信息**：登录后获取微信昵称、头像等基础信息，作为用户默认头像和昵称。

#### 技术考量（原规划）

- 需要微信开放平台开发者账号（需企业主体或个体工商户）。
- 需申请 `snsapi_login` 权限（微信扫码登录）。
- 服务端需新增微信 OAuth 流程：获取 code → 换取 access_token → 换取 openid/unionid → 登录/注册。
- 数据库需扩展 users 表，新增 `wechat_openid`、`wechat_unionid`、`wechat_nickname`、`wechat_avatar` 字段。
- 已有 JWT 认证体系需兼容微信登录：微信登录后同样签发 JWT token。
