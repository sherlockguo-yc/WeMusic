# 监控系统

> 独立部署的可观测性平台。当前优先为 WeMusic 提供服务监控能力，未来通过标准化协议支持多系统接入。
>
> **设计原则**：独立部署（不耦合 WeMusic 业务逻辑）、标准协议（基于 OpenTelemetry / Prometheus 生态）、可扩展（多租户架构）、高可用（自身健康监控、数据持久化、故障自愈）。

---

## 整体架构

### 技术选型

| 组件 | 推荐方案 | 说明 |
|------|---------|------|
| 指标采集与存储 | Prometheus + VictoriaMetrics（可选） | 社区标准，兼容 PromQL；数据量大时 VictoriaMetrics 更省资源 |
| 日志聚合 | Loki + Promtail | 轻量，与 Prometheus 统一标签体系，无全文索引开销 |
| 链路追踪 | OpenTelemetry SDK + Jaeger / Grafana Tempo | 厂商中立，避免 vendor lock-in |
| 可视化 | Grafana | 开箱即用的 Dashboard 生态，支持指标/日志/追踪联动 |
| 告警引擎 | Grafana Alerting + Alertmanager | 多通道通知、分组、抑制、静默、告警升级 |
| 配置存储 | PostgreSQL（多租户模式）/ SQLite（单租户 MVP） | 告警规则、通知渠道、租户配置 |

### 架构分层

```
展示层    Grafana / 自定义 Dashboard / 告警通知渠道(企微/邮件/Webhook)
─────────────────────────────────────────────────────
服务层    指标查询服务 │ 告警引擎(规则评估/去重/抑制/升级) │ 租户管理
─────────────────────────────────────────────────────
存储层    Prometheus TSDB / VictoriaMetrics · Loki · PostgreSQL
─────────────────────────────────────────────────────
采集层    OpenTelemetry Collector · Prometheus Exporters · Promtail
─────────────────────────────────────────────────────
被监控    WeMusic · 未来系统 A · 未来系统 B · ...
```

## 模块 1：数据采集与观测

### 1.1 指标采集

**基础指标**（自动采集，无需业务改动）：
- 基础设施：CPU / 内存 / 磁盘 I/O / 网络吞吐 / 文件描述符（Node Exporter）
- Node.js 运行时：事件循环延迟（ELU）、堆内存使用与 GC 频率、活跃句柄数（prom-client 内置）

**WeMusic 自定义指标**（需在应用中埋点）：
- 业务指标：实时播放数、活跃用户数、搜索 QPS、歌单创建/收藏数
- API 质量：各端点 QPS / P50/P95/P99 延迟 / 错误率（4xx/5xx）
- 上游依赖：QQ 音乐/B 站/网易云 API 调用成功率与耗时
- 后台任务：歌词抓取/视频源搜索 队列深度、失败率、单次耗时
- 缓存健康：缓存命中率、过期率

### 1.2 日志采集

- **接入方式**：Promtail 部署在被监控机器上，tail 日志文件推送至 Loki
- **采集范围**：应用日志（结构化 JSON）、访问日志（Nginx/Caddy）、错误日志、慢查询
- **日志增强**：自动注入 trace_id / span_id、主机名、服务名等标签；支持敏感信息脱敏
- **Grafana 联动**：从指标面板一键跳转至关联日志，按 trace_id 聚合查看完整请求链路日志

### 1.3 链路追踪

- **实现方式**：OpenTelemetry SDK 在 Node.js 中自动插桩（HTTP/gRPC/SQL 调用自动追踪）
- **追踪范围**：入口（Nginx → Node.js）→ 业务逻辑 → 外部 API（QQ 音乐/B 站/网易云）→ 数据库
- **采样策略**：生产环境「尾部采样」——正常请求 10% 采样，错误/高延迟请求 100% 采样

### 1.4 统一 Dashboard

- **服务概览大屏**：多系统实时状态一览，红绿灯健康指示，适合投屏展示
- **WeMusic 专用面板**：
  - 播放业务：实时播放数、热门歌曲 Top 20、播放源分布（B 站 vs 缓存）
  - API 质量：各端点延迟分位数、错误率趋势、上游依赖健康度
  - 后台任务：队列深度、耗时趋势、失败重试统计
- **基础设施面板**：主机资源、数据库连接池、Node.js 运行时

## 模块 2：智能告警

### 2.1 告警模型

支持三种告警模型，可自由组合：

**阈值告警**：
- 静态阈值：API 错误率 > 5%、磁盘使用率 > 80%、内存持续 > 1GB
- 动态阈值：基于历史数据自动计算基线（过去 7 天同时段均值 ± 2 标准差），自适应波动
- 持续时间：条件满足须持续 N 分钟才触发（避免瞬时抖动误报）

**变化检测告警**：
- 环比突变：QPS 较上一小时骤降 > 50%
- 趋势预测：磁盘使用率预测 24 小时内将达 95%（线性回归）
- 同比偏差：流量与上周同期偏差 > 3 倍标准差

**组合告警**：
- AND/OR 逻辑组合多个条件
- 前置过滤（如仅业务高峰期生效）
- 依赖抑制（「QQ 音乐 API 不可用」自动抑制「歌词获取失败」告警，避免告警风暴）

### 2.2 告警处理流程

```
异常检测 → 规则匹配 → 去重与抑制(同源合并/依赖抑制)
  → 静默检查(维护窗口跳过) → 分组(按服务/等级分组)
  → 延迟发送(等 N 分钟，恢复则取消) → 多通道通知
  → 告警升级(未确认则逐级上报，P0 可升级为电话通知)
```

### 2.3 通知渠道

- 即时通：企业微信机器人、钉钉、飞书 Webhook
- 邮件：HTML 格式详情 + 每日汇总摘要
- Webhook：自定义回调，可对接 PagerDuty / Opsgenie / 自定义运维系统
- 站内通知：WeMusic 管理面板顶部告警横幅

### 2.4 告警管理能力

- **规则管理**：CRUD + 启用/禁用/临时暂停，支持按时间段定时生效
- **静默窗口**：预设维护时段自动静默；支持手动一键静默指定时间
- **分组与抑制**：同服务多指标同时告警合并为一条通知；上游故障抑制下游级联告警
- **告警升级**：P0 5 分钟未确认 → 升级通知；P1 30 分钟 → 通知上级
- **告警历史**：完整记录触发/恢复时间、通知状态、确认人/时间，支持回溯分析

### 2.5 预置告警规则（WeMusic）

| 告警名称 | 触发条件 | 等级 | 说明 |
|---------|------|:---:|------|
| 服务不可用 | 健康检查连续 3 次失败 | 🔴 P0 | 进程崩溃或端口未监听 |
| 播放功能异常 | 连续 5 分钟无成功播放事件 | 🔴 P0 | 核心功能受影响 |
| API 错误率过高 | 5 分钟内 5xx ≥ 10% | 🟠 P1 | 需排查服务或上游 |
| QQ 音乐 API 异常 | P95 延迟 > 5s 持续 10 分钟 | 🟠 P1 | 上游依赖劣化 |
| 磁盘空间不足 | 使用率 > 85% | 🟡 P2 | 需清理或扩容 |
| 内存泄漏风险 | 堆内存连续增长 1h 无 GC 回落 | 🟡 P2 | 潜在内存泄漏 |
| 后台任务积压 | 队列 > 200 持续 30 分钟 | 🟡 P2 | 处理能力不足 |
| 证书即将过期 | HTTPS 证书剩余 < 30 天 | 🟢 P3 | 提前提醒续期 |

## 模块 3：数据分析

### 3.1 基础分析能力

- **时序分析**：任意指标时间趋势图，支持多指标叠加对比、同比/环比计算
- **聚合分析**：按服务/端点/时间维度聚合，Top N 排行
- **分布分析**：延迟分位数分布、请求体大小分布、用户活跃时长分布
- **关联分析**：两指标相关性（如 CPU 与 QPS、错误率与内存），辅助定位根因

### 3.2 SLO / 错误预算监控

- **SLO 定义**：为目标服务设定可用性/延迟目标（如「99.9% 请求延迟 < 500ms」）
- **错误预算**：基于 SLO 计算剩余错误预算，预算耗尽前预警（燃尽图）
- **多窗口 SLO**：同时监控 7d / 30d / 90d 滚动窗口，平衡短期波动与长期趋势

### 3.3 简易异常检测

- **统计异常**：基于 Z-Score / IQR 检测突刺和突降
- **季节性分解**：识别日周期/周周期模式，分离趋势与周期性成分
- **线性预测**：基于历史趋势预测资源耗尽时间（如「磁盘预计 7 天后满」）

### 3.4 报表与导出

- **定时报表**：每周/每月自动生成健康报告（PDF/Markdown），含 SLA 达成、Top 告警、资源趋势、业务关键指标，推送至管理员
- **数据导出**：Grafana 原生导出（CSV/JSON/PNG）+ 开放 API 供外部系统消费

## 模块 4：多租户与可扩展性

### 4.1 租户模型

- **租户** = 一个被监控的独立系统（WeMusic 是第一个租户）
- 每个租户独立拥有：指标命名空间、告警规则、通知渠道、Dashboard
- 租户间数据完全隔离，不可互相访问

### 4.2 接入方式

**标准接入（推荐）**：
1. 被监控系统暴露 Prometheus `/metrics` endpoint
2. 监控系统注册新租户，提供 scrape 目标地址
3. 配置告警规则与通知渠道
4. 导入预置 Dashboard 模板（可自定义）

**轻量接入（短期任务/无法暴露 endpoint 的场景）**：
- Pushgateway 模式，主动推送指标
- OpenTelemetry SDK 集成，一键上报 traces 和 logs

### 4.3 Dashboard 模板

- 「通用 HTTP 服务」：QPS / 延迟 / 错误率 / 状态码分布
- 「Node.js 应用」：ELU / GC / 堆内存 / 句柄数
- 「数据库」：连接池 / 慢查询 / 查询量趋势
- 管理员可分享/导入/导出 Dashboard JSON

## 与现有模块的关系

- **管理功能 > 系统监控**：当前内嵌在 WeMusic 管理面板的轻量监控，独立监控系统上线后，该模块逐步收敛为「监控系统快捷入口 + 简要状态视图」，深度监控由独立系统承接
- **管理功能 > 操作审计**：监控系统的告警规则变更、静默操作等应写入审计日志
- **管理功能 > 系统配置**：告警规则初期可在 WeMusic 系统配置中作子项管理，后期迁移至监控系统自有配置界面

- [ ] Prometheus 单实例 vs VictoriaMetrics：数据量 < 100 万活跃时间序列时 Prometheus 足够
- [ ] 是否需要长期存储卸载到对象存储（S3/MinIO）以支持年度对比？
- [ ] 日志存储：Loki（轻量低成本）vs Elasticsearch（功能丰富但资源消耗大）？建议从 Loki 起步

- [ ] Grafana / Prometheus 是否需要接入认证和 HTTPS？

**运维**：
- [ ] 是否需要 Infrastructure as Code（Terraform / Ansible）管理监控基础设施？
