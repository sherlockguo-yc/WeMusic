# 音量一致性

## 概述

不同 B 站视频源的音量差异巨大（UP 主录制时音量各不相同），用户连续播放多首歌时需要频繁手动调整音量。目标：自动标准化响度，让不同源之间的音量大致一致。

## 技术概念

### RMS（均方根）= 音频的平均响度

音频是一串随时间变化的数字（采样点），每个数字代表那一瞬间声波的振幅。RMS 把所有采样点的振幅平方、求平均、再开平方根，得到一个代表「整体响度」的数字。

```
RMS = 0.02  → 很轻，听不清
RMS = 0.08  → 正常
RMS = 0.30  → 震耳朵
```

### Gain（增益）= 音量倍率

Gain 是一个乘数，放在「音频源」和「耳朵」之间：

```
音频源 → [Gain 倍率] → 你听到的声音
```

- `gain = 1.0`：不改变
- `gain = 0.3`：压到原来 30%（对付太吵的源）
- `gain = 3.0`：放大 3 倍（对付太小声的源）

例：歌曲 A 的 RMS 是 0.30（太吵），歌曲 B 的 RMS 是 0.05（太小）。以 0.08 为基准：

- 歌曲 A：`gain = 0.08 / 0.30 = 0.27`（压低）
- 歌曲 B：`gain = 0.08 / 0.05 = 1.60`（放大）

### EBU R128（行业标准响度）

简单 RMS 只反映平均振幅，和人耳实际听感不完全一致。EBU R128 是广播行业的响度标准，模拟人耳对不同频率的敏感度，比简单 RMS 更准确。ffprobe 内置 EBU R128 分析能力。

## 音量层级模型

```
B站源音频 → [Gain 归一化] → [播放器音量 _bgVolume] → [OS 系统音量] → 耳朵
   ↑              ↑                  ↑                    ↑
 原始响度      新增，自动调节      用户手动调节          我们管不了
```

- **系统音量**：OS 级别，对所有应用统一缩放。管不了也看不见，但只要进入系统音量之前各首歌一样响，切歌不会有落差
- **播放器音量**（`_bgVolume`）：WeMusic 底部音量滑块，用户主动调节
- **Gain 归一化**：自动为每个视频源计算一个倍率，让不同源的响度对齐。**存储于服务端 DB**，跨用户共享

## 选定方案：服务端 fork 音频流 + ffprobe 分析

### 为什么选这个方案

- **Gain 是视频源的客观属性**，不依赖用户设置 → 应该存在服务端，跨用户共享
- **需要完整音频分析**，避免前奏太轻/太重导致的采样偏差
- **分析不能阻塞播放** → 服务端 fork 流，客户端正常播放，分析在后台并行
- **纯前台用户也能触发**：前台播放时 bgAudio preload 自动发起 stream 请求

### 已确认的设计决策

| 决策 | 选择 |
|------|------|
| 分析引擎 | ffprobe（EBU R128 标准响度），N100 上装 ffmpeg |
| 去重机制 | DB `gain_cache` 表 + `status` 字段（pending/complete/failed），按 bvid 查重 |
| 并发策略 | 串行队列，一次只分析一个，临时文件不超过 1 个 |
| 缓存 key | `bvid + cid`（服务端自动解析 cid，客户端只传 bvid） |
| Gain 基准值 | -23 LUFS（EBU R128 广播标准） |
| Gain 上下限 | clamp(0.3, 3.0) |
| 跳过标准化开关 | 需要，加一个设置项 |
| 前端 API | 独立 `GET /api/gain?bvid=xxx`，切歌前调用 |
| 分析失败重试 | 每次播放都重试（分析在后台不阻塞播放） |

#### 为什么选 bvid+cid 而非纯 bvid

B 站视频可以有多个「分 P」：同一个 bvid 下不同 cid 对应不同的音频。用 bvid 作唯一 key 会导致不同分 P 的 gain 互相覆盖。虽然多 P 视频占比不高，但用 bvid+cid 更准确。

**实现方案**：客户端对外只传 bvid，服务端在内部自动解析 cid（通过 `getVideoPages(bvid)` 取第一 P 的 cid，或从已有的 `stream.cid` 获取）。客户端无需额外改动。

### 架构

```
每次 /api/play/stream?bvid=xxx 请求到达：

1. 服务端检查 DB：这个 bvid 已有 gain？
   ├─ 有 → 正常 pipe 音频流给客户端（不启动分析）
   └─ 没有 → 启动 fork 流程：

2. 服务端 fetch B站 CDN 音频流
   ├─ 分支 A：pipe → 客户端（正常播放，零延迟）
   └─ 分支 B：write → 临时文件（服务端独立下载完整文件）

3. 临时文件完整后：
   ffprobe 测量 EBU R128 integrated loudness
   → 计算 gain
   → 写入 DB（bvid → gain）
   → 删除临时文件
```

**关键设计**：服务端从 B 站 CDN 下载是独立于客户端连接的——即使客户端提前断开（切歌/关标签页），服务端仍然继续下载完整文件并完成分析。客户端的行为不影响 gain 的计算。

### 前端 Gain 应用（后台播放）

```
bgAudio (volume 固定 1.0)
  → AnalyserNode (可选，调试用)
  → GainNode (gain = 服务端查到的归一化倍率 × _bgVolume)
  → audioCtx.destination
```

### 前后台差异

| | 前台（iframe） | 后台（bgAudio） |
|---|---|---|
| 能否触发分析？ | ✅ bgAudio preload 会触发 /api/play/stream 请求 → 服务端 fork | ✅ 正常触发 |
| 能否应用归一化 gain？ | ⚠️ 间接：调整 postMessage 音量 = `_bgVolume × cached_gain` | ✅ 直接在 GainNode 中应用 |
| 首次播放（无缓存） | gain = 1.0（听原声），分析在后台运行，下次播放有效 | 同左 |

## 边界情况讨论

### 触发时机

**Q: 用户一直在前台，会触发分析吗？**

会。前台播放时，`startVideo()` 在挂载 iframe 后 500ms 调用 `_bgPreload(bvid)`，bgAudio 做静音预缓冲。这会产生 `/api/play/stream` 请求 → 服务端 fork 流 → 触发分析。

**结果：无论用户在前台还是后台，只要播放一个视频，就会触发 gain 分析。** 纯前台用户也能贡献 gain 数据。

### 用户操作对分析的影响

**Q: 中途前后台切换，会影响 gain 计算吗？**

不影响。服务端从 B 站 CDN 的下载独立于客户端连接。前后台切换只会改变客户端侧的 bgAudio/iframe 切换，服务端 temp 文件继续正常写入。

**Q: 用户暂停了，会影响吗？**

不影响。bgAudio 暂停是客户端行为，不中断 HTTP 连接。服务端继续从 B 站 CDN 下载完整文件。

**Q: 用户调整了播放进度（seek），会影响吗？**

不影响。gain 计算只看完整音频文件的响度，和用户听到哪个位置无关。

**Q: 用户切歌了，会影响吗？**

不影响已经触发的分析。服务端正在下载的文件会继续下载完毕并完成分析。新歌的 `/api/play/stream` 请求独立处理。

### 重复与并发

**Q: 同一首歌唱第二遍，还会再分析吗？**

不会。DB 里已有 gain → 服务端跳过分析，直接正常 pipe。

**Q: 同一首歌，两个用户同时播放，会分析两次吗？**

需要去重机制。DB 中记录分析状态（pending/complete/failed），如果已有 pending 分析，第二个请求不重复启动。

**Q: N 首不同的歌同时播放，并发分析怎么办？**

N100 性能有限，建议**串行队列**：一次只分析一个文件，其他等待。临时文件也不超过 1 个。

### 数据准确性

**Q: 同一 bvid，第一次走 DASH 音频、下次走 H5 durl 回退，gain 还准确吗？**

准确。DASH 纯音频和 H5 durl 音视频合一的音频轨道内容相同，响度一致。不同码率/容器不影响解码后的 PCM 响度。

**Q: Gain 基准值（目标响度）设多少？**

固定值，建议 -23 LUFS（EBU R128 广播标准），或根据实际测试调整。不从第一首歌动态校准（如果第一首恰是极端值，基准就歪了）。

### 临时文件

**Q: 临时文件的磁盘开销？**

- DASH 纯音频：2-5MB（最常见的场景）
- H5 durl 回退（音视频合一）：50-100MB（较少触发）
- 串行分析，同时最多 1 个临时文件
- 完成后立即删除

**Q: 分析失败（ffprobe 崩溃、文件损坏），临时文件残留？**

需要清理机制：启动时检查并删除旧的 temp 文件。分析失败标记为 failed，后续同一 bvid 的播放重试分析。

### 前端数据获取

**Q: 前端如何获取 gain？**

新增 API：`GET /api/gain?bvid=xxx`。前端切歌时在播放前调用，有结果就用，没有就 gain=1.0。或者将 gain 合并到视频解析（`/api/play/resolve`）的返回值中。

### Gain 上下限

防止极端值导致失真或静音：

```
最终 gain = clamp(normalize_gain, 0.3, 3.0) × _bgVolume
```

- 下限 0.3：防止太吵的源被压到几乎静音
- 上限 3.0：防止太小的源被放大到严重失真

### 短音频跳过

时长 < 10 秒的视频不做分析（数据量不足以支撑有意义的响度计算）。

### localStorage 的 AudioContext 约束

`createMediaElementSource(bgAudio)` **只能调用一次**（浏览器限制，类似单向阀门）。必须在 APP 初始化时创建 AudioContext 管线，整个会话保持连接。换歌（改 `bgAudio.src`）不断开连接，不影响。
