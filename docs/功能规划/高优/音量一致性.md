# 音量一致性

## 概述

不同 B 站视频源的音量差异巨大（UP 主录制时音量各不相同），用户连续播放多首歌时需要频繁手动调整音量。目标：自动标准化响度，让不同源之间的音量大致一致。

## 技术概念

### RMS（均方根）= 音频的平均响度

音频是一串随时间变化的数字（采样点），每个数字代表那一瞬间声波的振幅。RMS 把所有采样点的振幅平方、求平均、再开平方根，得到一个代表「整体响度」的数字。

```
RMS = 0.02  → 很轻，听不清
RMS = 0.08  → 正常
RMS = 0.30  → 震耳朵
```

### Gain（增益）= 音量倍率

Gain 是一个乘数，放在「音频源」和「耳朵」之间：

```
音频源 → [Gain 倍率] → 你听到的声音
```

- `gain = 1.0`：不改变
- `gain = 0.3`：压到原来 30%（对付太吵的源）
- `gain = 3.0`：放大 3 倍（对付太小声的源）

例：歌曲 A 的 RMS 是 0.30（太吵），歌曲 B 的 RMS 是 0.05（太小）。以 0.08 为基准：

- 歌曲 A：`gain = 0.08 / 0.30 = 0.27`（压低）
- 歌曲 B：`gain = 0.08 / 0.05 = 1.60`（放大）

### EBU R128（行业标准响度）

简单 RMS 只反映平均振幅，和人耳实际听感不完全一致。EBU R128 是广播行业的响度标准，模拟人耳对不同频率的敏感度，比简单 RMS 更准确。ffprobe 内置 EBU R128 分析能力。

## 音量层级模型

```
B站源音频 → [Gain 归一化] → [播放器音量 _bgVolume] → [OS 系统音量] → 耳朵
   ↑              ↑                  ↑                    ↑
 原始响度      新增，自动调节      用户手动调节          我们管不了
```

- **系统音量**：OS 级别，对所有应用统一缩放。管不了也看不见，但只要进入系统音量之前各首歌一样响，切歌不会有落差
- **播放器音量**（`_bgVolume`）：WeMusic 底部音量滑块，用户主动调节
- **Gain 归一化**：自动为每个视频源计算一个倍率，让不同源的响度对齐。**存储于服务端 DB**，跨用户共享

## 已实现方案

### 一句话概括

播放一首歌时，服务端在后台偷偷下载完整音频、用 ffprobe 测出它的响度、算出一个「增益倍率」存起来。下次播放同一首歌时，浏览器直接用这个倍率把音量调准。

#### 阶段 0：APP 启动时（只执行一次）

```
1. 用户打开 WeMusic 网页
   ↓
2. initPlayer() 执行：
   ↓
3. _initAudioCtx() 被调用：
   a) 创建 new AudioContext() — 浏览器的音频控制中心
   b) 调用 audioCtx.createMediaElementSource(bgAudio)
      → 把 <audio id="audio"> 这个元素「接」到 AudioContext 的管线里
      → ⚠️ 这一步只能做一次！以后换歌只是改 bgAudio.src，管线不用重建
   c) 创建 audioCtx.createGain() — 一个「音量阀门」
   d) 连接管线：
      bgAudio → GainNode → 扬声器
      （音频从 <audio> 流出来 → 经过阀门 → 到你的耳朵）
   e) 注册点击/按键事件：一旦用户点了任何地方，就调用 audioCtx.resume()
      → 因为浏览器出于安全考虑，AudioContext 刚创建时是「暂停」状态
      → 必须有用户交互（点击/按键）才能激活
   ↓
4. _applyVol() 执行：
   a) bgAudio.volume = 1.0（原始音频不做衰减，音量完全由阀门控制）
   b) 调用 _applyNormVol()
      → 检查设置：用户关了归一化吗？
        - 关了 → 阀门开度 = 用户音量（比如 0.8）
        - 没关 → 阀门开度 = 归一化倍率(1.0，还没有缓存) × 用户音量(0.8) = 0.8
      → 设置阀门：_gainNode.gain.value = 0.8
   c) 更新音量按钮的 tooltip："后台音量 | 增益 1.00x（无缓存）"
```

#### 阶段 4：服务端 stream 端点（播放 + 触发分析）

```
19. 服务端收到 GET /api/play/stream?bvid=BV1xx...
    ↓
20. 调用 getAudioStream("BV1xx...")：
    a) 调 B站 API，拿到音频流的 CDN URL（如 https://xxx.bilivideo.com/xxx.m4a）
    b) 拿到 cid = 123（视频第一P的 ID）
    ↓
21. fetchAudio(cdnUrl)：
    → 从 B站 CDN 拉取音频流（带 Referer 头，绕过防盗链）
    → 拿到响应（upstream），body 是可读流
    ↓
22. 检查：这个视频的 gain 分析过了吗？
    SELECT status FROM gain_cache WHERE bvid='BV1xx...' AND cid=123

    ├─ status='complete' → 已有 gain，跳过分析
    ├─ status='pending'  → 正在分析中，跳过（不重复触发）
    └─ 无记录 或 status='failed' → 需要分析！
        ↓
    23. 往 DB 写一条 pending 记录：
        INSERT INTO gain_cache (bvid, cid, gain_mult, status)
        VALUES ('BV1xx...', 123, 1.0, 'pending')
        ↓
    24. enqueueAnalysis('BV1xx...', 123, cdnUrl)：
        → 把这个分析任务扔进串行队列
        → 如果队列里没有别的任务在跑 → 立刻执行
        → 如果队列里有任务 → 等前面的跑完再执行
        → 这一步不阻塞！马上就返回了
    ↓
25. pipeAudio(upstream, res)：
    → 把 B站 CDN 的音频流原样发给浏览器
    → 浏览器 bgAudio 收到数据 → 开始缓冲 → 可以播放了 ✅
    → 注意：这一步和步骤 24 是独立的两条路
      - 步骤 25：客户端 → B站流 → 浏览器，你听到声音
      - 步骤 24：服务端 → B站流 → tmp 文件 → ffprobe，和你的播放无关
```

#### 阶段 5：后台分析（在串行队列里慢慢跑）

```
26. 轮到 BV1xx 的分析任务了，runAnalysis() 执行：

    26a. 下载完整音频文件：
         fetch(cdnUrl) // 注意：这次不带 Range 头，要完整的文件！
         → 从 B站 CDN 独立下载
         → 和你的播放用的是两条不同的 HTTP 连接
         → 即使你切歌了、关网页了，这个下载照样跑完
         → 写入 /tmp/wemusic_gain_BV1xx_123_1752110400000.mp4
         ↓
    26b. 文件下载完成，比如 3.2MB
         ↓
    26c. 运行 ffprobe 分析响度：
         ffprobe -v quiet -f lavfi \
           -i "amovie=/tmp/wemusic_gain_BV1xx_123_xxx.mp4,ebur128=video=0:meter=0" \
           -show_entries frame_tags=lavfi.r128.I \
           -of csv=p=0
         
         → 输出若干行数字，如：
           -16.3
           -16.5
           -16.7
           ...
           -16.5  ← 取最后一行，这是整个文件的 integrated loudness
         ↓
    26d. 计算增益倍率：
         目标响度 = -23 LUFS（广播标准）
         实际响度 = -16.5 LUFS
         
         raw_gain = 10^((-23 - (-16.5)) / 20)
                 = 10^(-6.5 / 20)
                 = 10^(-0.325)
                 = 0.4732
         
         clamp(0.4732, 0.3, 3.0) = 0.4732（在范围内）
         ↓
    26e. 写入 DB：
         UPDATE gain_cache
         SET gain_lufs = -16.5,
             gain_mult = 0.4732,
             status = 'complete',
             updated_at = 1752110420000
         WHERE bvid = 'BV1xx...' AND cid = 123
         ↓
    26f. 删除临时文件：
         rm /tmp/wemusic_gain_BV1xx_123_xxx.mp4
         ↓
    26g. 打印日志：
         [gain] done bvid=BV1xx... cid=123 lufs=-16.5 gain=0.4732
```

## 边界情况讨论

### 触发时机

**Q: 用户一直在前台，会触发分析吗？**

会。前台播放时，`startVideo()` 在挂载 iframe 后 500ms 调用 `_bgPreload(bvid)`，bgAudio 做静音预缓冲。这会产生 `/api/play/stream` 请求 → 服务端 fork 流 → 触发分析。

**结果：无论用户在前台还是后台，只要播放一个视频，就会触发 gain 分析。** 纯前台用户也能贡献 gain 数据。

### 用户操作对分析的影响

**Q: 中途前后台切换，会影响 gain 计算吗？**

不影响。服务端从 B 站 CDN 的下载独立于客户端连接。前后台切换只会改变客户端侧的 bgAudio/iframe 切换，服务端 temp 文件继续正常写入。

**Q: 用户暂停了，会影响吗？**

不影响。bgAudio 暂停是客户端行为，不中断 HTTP 连接。服务端继续从 B 站 CDN 下载完整文件。

**Q: 用户调整了播放进度（seek），会影响吗？**

不影响。gain 计算只看完整音频文件的响度，和用户听到哪个位置无关。

**Q: 用户切歌了，会影响吗？**

不影响已经触发的分析。服务端正在下载的文件会继续下载完毕并完成分析。新歌的 `/api/play/stream` 请求独立处理。

### 重复与并发

**Q: 同一首歌唱第二遍，还会再分析吗？**

不会。DB 里已有 gain → 服务端跳过分析，直接正常 pipe。

**Q: 同一首歌，两个用户同时播放，会分析两次吗？**

需要去重机制。DB 中记录分析状态（pending/complete/failed），如果已有 pending 分析，第二个请求不重复启动。

**Q: N 首不同的歌同时播放，并发分析怎么办？**

N100 性能有限，建议**串行队列**：一次只分析一个文件，其他等待。临时文件也不超过 1 个。

### 数据准确性

**Q: 同一 bvid，第一次走 DASH 音频、下次走 H5 durl 回退，gain 还准确吗？**

准确。DASH 纯音频和 H5 durl 音视频合一的音频轨道内容相同，响度一致。不同码率/容器不影响解码后的 PCM 响度。

**Q: Gain 基准值（目标响度）设多少？**

固定值，建议 -23 LUFS（EBU R128 广播标准），或根据实际测试调整。不从第一首歌动态校准（如果第一首恰是极端值，基准就歪了）。

### 临时文件

**Q: 临时文件的磁盘开销？**

- DASH 纯音频：2-5MB（最常见的场景）
- H5 durl 回退（音视频合一）：50-100MB（较少触发）
- 串行分析，同时最多 1 个临时文件
- 完成后立即删除

**Q: 分析失败（ffprobe 崩溃、文件损坏），临时文件残留？**

需要清理机制：启动时检查并删除旧的 temp 文件。分析失败标记为 failed，后续同一 bvid 的播放重试分析。

### 前端数据获取

**Q: 前端如何获取 gain？**

`GET /api/play/gain?bvid=xxx`（fire-and-forget，不阻塞播放）

```json
{"gain": 0.47, "status": "complete"}   // 已有缓存
{"gain": null, "status": "pending"}     // 分析中
{"gain": null, "status": null}          // 无记录
```

### Gain 上下限

防止极端值导致失真或静音：

```
最终 gain = clamp(normalize_gain, 0.3, 3.0) × _bgVolume
```

- 下限 0.3：防止太吵的源被压到几乎静音
- 上限 3.0：防止太小的源被放大到严重失真

### 短音频跳过

时长 < 10 秒的视频不做分析（数据量不足以支撑有意义的响度计算）。

### localStorage 的 AudioContext 约束

`createMediaElementSource(bgAudio)` **只能调用一次**（浏览器限制，类似单向阀门）。必须在 APP 初始化时创建 AudioContext 管线，整个会话保持连接。换歌（改 `bgAudio.src`）不断开连接，不影响。

## 实现细节

### DB 建表

```sql
CREATE TABLE IF NOT EXISTS gain_cache (
  bvid        TEXT NOT NULL,
  cid         INTEGER NOT NULL DEFAULT 0,
  gain_lufs   REAL,
  gain_mult   REAL NOT NULL DEFAULT 1.0,
  status      TEXT DEFAULT 'pending',  -- pending / complete / failed
  created_at  INTEGER NOT NULL,
  updated_at  INTEGER NOT NULL,
  PRIMARY KEY (bvid, cid)
);
```

### 分析流程（服务端）

1. stream 端点检查 `gain_cache WHERE bvid=? AND cid=?`
2. 无记录或 failed → INSERT pending + 入队串行分析
3. 分析线程：fetch B站 CDN（独立连接，无 Range）→ write `/tmp/wemusic_gain_xxx.mp4` → ffprobe → calcGain → UPDATE DB → rm temp

### ffprobe 命令

```bash
ffprobe -v quiet -f lavfi \
  -i "amovie=/tmp/wemusic_gain_BVxxx_123.mp4,ebur128=video=0:meter=0" \
  -show_entries frame_tags=lavfi.r128.I \
  -of csv=p=0
```

取最后一行作为 integrated loudness（单位 LUFS）。

### Gain 计算

```js
const TARGET = -23; // LUFS（EBU R128 广播标准）
function calcGain(lufs) {
  return clamp(10 ** ((TARGET - lufs) / 20), 0.3, 3.0);
}
// 例：-16.5 LUFS → gain = 10^(-6.5/20) ≈ 0.47
```

### 串行队列

```js
let _gainQueue = Promise.resolve();
function enqueueAnalysis(bvid, cid, streamUrl) {
  _gainQueue = _gainQueue.then(() => runAnalysis(bvid, cid, streamUrl));
}
```

### 临时文件

- 路径：`/tmp/wemusic_gain_${bvid}_${cid}_${Date.now()}.mp4`
- 完成后立即 `fs.unlinkSync()`
- 启动时清理 `/tmp/wemusic_gain_*` 残留
- 串行队列保证同时最多 1 个

### 跳过标准化开关

- 存储：localStorage `wemusic_volume_normalize`（"1" / "0"），默认 "1"
- 位置：设置面板 > 字号下方，toggle-switch
- 关闭时 `_applyNormVol` 直接跳过归一化增益 → GainNode gain = `_bgVolume`

### Gain 信息可见性

音量按钮 tooltip 实时显示归一化状态：
- 无缓存：`后台音量 | 增益 1.00x（无缓存）`
- 有缓存：`后台音量 | 增益 0.47x`
- 已关闭：`后台音量 | 归一化已关闭`

设置面板开关 → 关闭/开启 → A/B 对比音量效果。
