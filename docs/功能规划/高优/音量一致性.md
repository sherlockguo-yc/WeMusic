# 音量一致性

## 概述

B 站视频源的音量差异巨大（UP 主录制时音量各不相同），用户连续播放多首歌时需要频繁手动调整音量。目标分两层：

1. **源间一致性**（已实现）：自动归一化，让不同源之间的响度大致一致
2. **UI 一致性**（待实现）：消除多个音量控制入口，统一到系统音量

---

## 一、现状问题：三个音量控制

当前存在三个独立的音量控制点，用户每次切换前后台都会感到音量不一致：

| 控制点 | 位置 | 作用范围 | 可控性 |
|--------|------|----------|--------|
| WeMusic 底部滑块 | 播放器底栏 `.volume-wrap` | 后台 bgAudio（GainNode） | ✅ 完全可控 |
| B站 iframe 音量 | B站播放器内部（cross-origin） | 前台视频音频 | ⚠️ 可 postMessage 设值，但用户能手动改 |
| 系统音量 | OS 级别 | 全局 | ✅ 用户手动 |

核心矛盾：**两条独立的音频管线各有一套音量乘数，前后台切换时必然有落差。**

```
前台：B站 CDN → iframe → [B站音量滑块] → [系统音量] → 耳朵
后台：B站 CDN → bgAudio → [GainNode(_bgVolume × safeGain)] → [系统音量] → 耳朵
```

归一会化只通过 GainNode 对后台 bgAudio 生效，前台 iframe 不受影响。

### 2.3 关键否决因素

**方案 A**（E 同理）：要求两个跨域不可控的播放器保持同步，但 B站 iframe 对我们是黑盒：
- 能发：`setVolume`（已知，在用）
- 能收：播放器就绪心跳（已知，在用）
- 不能发：seek / pause / resume（API 未公布，社区无文档）
- 不能收：currentTime / playback state（消息格式不明）
- 结构性矛盾 → 否决

**方案 F**：理论最优（一条管线、无跨域、原生 seek / 同步），但：
- 服务端视频流代理 → 带宽 10-50x 音频流
- N100/N150 家宽上传带宽不匹配 → 否决

**方案 B/C**：解决方式偏弱（仍然有前后台音量差），C 额外多了维护成本。

### 2.4 选定：方案 D-lite

D 与 D+ 在主场景（后台听歌，归一化覆盖）表现一致。差异仅在前台（次要场景）：
- D+ 的前台归一化依赖 `setVolume(safeGain*100)`，B站播放器音量算法未知，不能保证与 GainNode 线性对齐；
- D 的前台无归一化是可接受的——用户在前台看视频时听到原始音量，符合"看视频"的预期。

D-lite = D + 一个零成本锦上添花：回前台时尝试用归一化增益反推 B站音量值（`setVolume(round(clamp(safeGain, 0, 1)*100))`），设不准也无所谓（现在前台压根没归一化），设准了就白赚一致性。

## 四、音量归一化（已实现，保留）

以下为已实现的归一化系统，方案 D-lite 不改这部分。

### 4.1 技术概念

**RMS（均方根）= 音频的平均响度**

音频是一串随时间变化的数字（采样点），每个数字代表那一瞬间声波的振幅。RMS 把所有采样点的振幅平方、求平均、再开平方根，得到一个代表「整体响度」的数字。

```
RMS = 0.02  → 很轻，听不清
RMS = 0.08  → 正常
RMS = 0.30  → 震耳朵
```

**Gain（增益）= 音量倍率**

Gain 是一个乘数，放在「音频源」和「耳朵」之间：

```
音频源 → [Gain 倍率] → 你听到的声音
```

- `gain = 1.0`：不改变
- `gain = 0.3`：压到原来 30%（对付太吵的源）
- `gain = 3.0`：放大 3 倍（对付太小声的源）

例：歌曲 A 的 RMS 是 0.30（太吵），歌曲 B 的 RMS 是 0.05（太小）。以 0.08 为基准：
- 歌曲 A：`gain = 0.08 / 0.30 = 0.27`（压低）
- 歌曲 B：`gain = 0.08 / 0.05 = 1.60`（放大）

**EBU R128（行业标准响度）**

简单 RMS 只反映平均振幅，和人耳实际听感不完全一致。EBU R128 是广播行业的响度标准，模拟人耳对不同频率的敏感度，比简单 RMS 更准确。ffprobe 内置 EBU R128 分析能力。

### 4.2 音量层级模型

```
B站源音频 → [Gain 归一化] → [OS 系统音量] → 耳朵
   ↑              ↑                 ↑
 原始响度      自动调节           用户手动调
```

方案 D-lite 实施后，中间不再有 WeMusic 自维护的 `_bgVolume` 乘数。

### 4.3 详细实现流程

#### 阶段 0：APP 启动时（只执行一次）

```
1. 用户打开 WeMusic 网页
   ↓
2. initPlayer() 执行：
   ↓
3. _initAudioCtx() 被调用：
   a) 创建 new AudioContext()
   b) createMediaElementSource(bgAudio) — ⚠️ 只能调用一次
   c) 创建 createGain() — 音量阀门
   d) 连接管线：bgAudio → GainNode → 扬声器
   e) 注册点击/按键事件激活 AudioContext
   ↓
4. _applyNormVol() 执行：
   GainNode gain = skipNorm ? 1.0 : safeGain
   （不再乘 _bgVolume）
```

#### 阶段 1：用户点播放

```
5. playCurrent() → 解析 bvid → startVideo(bvid, title, dur)
```

#### 阶段 2：startVideo — 两条并行路径

```
前台：
  mountVideo(bvid, title)     → B站 iframe 播放器（画面+原生音频）
  等 500ms →
  _bgPreload(bvid)            → bgAudio 静音预缓冲（为后台切歌准备）

后台：
  _bgPreload(bvid)            → 加载音频
  _bgUnmuteAndPlay(0)         → 从 0 开始播放
```

#### 阶段 3：_bgPreload — 并行请求

```
_bgPreload(bvid):
  ① _fetchGain(bvid)          → GET /api/play/gain?bvid=xxx
     ├─ 有缓存 → _normGain = gain → _applyNormVol()
     └─ 无缓存 → _normGain = 1.0

  ② bgAudio.src = /api/play/stream?bvid=xxx&token=...
     bgAudio.muted = true      → 预缓冲
     bgAudio.volume = 1.0      → 始终 1.0
```

### 4.4 归一化分析流程

#### 服务端 stream 端点

```
GET /api/play/stream?bvid=xxx
  → getAudioStream(bvid)         调 B站 API 拿 DASH 音频 CDN URL
  → fetchAudio(cdnUrl)           从 B站 CDN 拉音频流
  → 检查 gain_cache              已有 → 跳过分析
                                无 → INSERT pending + 入队串行分析
  → pipeAudio(upstream, res)    音频流 → 浏览器
```

#### 后台分析（串行队列）

```
runAnalysis(bvid, cid, cdnUrl):
  ① fetch(cdnUrl)              完整下载音频 → /tmp/wemusic_gain_xxx.mp4
  ② ffmpeg ebur128 分析
  ③ 解析 Integrated loudness   stderr → LUFS 值
  ④ calcGain(lufs)             clamp(10^((TARGET-lufs)/20), 0.05, 10.0)
  ⑤ UPDATE gain_cache          存 gain_mult, lufs, status='complete'
  ⑥ fs.unlinkSync(tempPath)    删临时文件
```

#### ffmpeg 命令

```bash
ffmpeg -i /tmp/wemusic_gain_BVxxx.mp4 \
  -af ebur128=video=0 \
  -f null -
```

取 stderr 中 `Integrated loudness:` 后的 `I:` 值（单位 LUFS）。

#### Gain 计算

```js
const TARGET = -18; // LUFS（消费端标准）
function calcGain(lufs) {
  return clamp(10 ** ((TARGET - lufs) / 20), 0.05, 10.0);
}
// 客户端安全阀：≤ 8.0
// 例：-16.5 LUFS → gain = 10^(1.5/20) ≈ 1.19
```

#### 前端数据获取

`GET /api/play/gain?bvid=xxx`（fire-and-forget，不阻塞播放）

```json
{"gain": 0.47, "status": "complete"}
{"gain": null, "status": "pending"}
{"gain": null, "status": null}
```

### 4.5 边界情况

#### 触发时机

**Q: 用户一直在前台，会触发分析吗？**

会。前台播放时 `startVideo()` 在挂载 iframe 后 500ms 调用 `_bgPreload(bvid)` → `/api/play/stream` 请求 → 服务端 fork 流 → 触发分析。

#### 用户操作对分析的影响

| 操作 | 影响 |
|------|------|
| 前后台切换 | 不影响。服务端从 B站 CDN 的下载独立于客户端连接 |
| 暂停 | 不影响。bgAudio 暂停是客户端行为，不中断 HTTP 连接 |
| 调整进度（seek） | 不影响。gain 看完整文件响度，和播放位置无关 |
| 切歌 | 不影响已触发的分析。新歌的 stream 请求独立处理 |

#### 并发与去重

| 场景 | 处理 |
|------|------|
| 同一首歌播第二遍 | 跳过（DB 已有 status='complete'） |
| 两个用户同时播同一首 | DB pending 标记去重 |
| N 首不同歌同时触发 | 串行队列，一次分析一个 |

#### 数据准确性

| 问题 | 答案 |
|------|------|
| DASH 音频 vs H5 durl 回退，gain 还准吗？ | 准。音频轨道内容相同，不同码率/容器不影响 PCM 响度 |
| Gain 基准值设多少？ | -18 LUFS（消费端标准，Apple/YouTube 区间） |
| 短音频？ | 时长 < 10 秒跳过 |

### 4.6 DB 表结构

```sql
CREATE TABLE IF NOT EXISTS gain_cache (
  bvid        TEXT NOT NULL,
  cid         INTEGER NOT NULL DEFAULT 0,
  gain_lufs   REAL,
  gain_mult   REAL NOT NULL DEFAULT 1.0,
  title       TEXT DEFAULT '',
  status      TEXT DEFAULT 'pending',  -- pending / complete / failed
  created_at  INTEGER NOT NULL,
  updated_at  INTEGER NOT NULL,
  PRIMARY KEY (bvid, cid)
);
```
