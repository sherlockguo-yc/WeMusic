# 管理功能概览

> 面向管理员和运营人员的后台管理能力。当前实现极其单薄（仅展示用户数/反馈数/总时长 + 用户列表 + 反馈删除，权限仅靠环境变量白名单），需系统性补全。本规划按模块拆分为独立的功能体系，不涉及具体实现方案。

---

## 现状诊断

| 维度 | 现状 | 差距 |
|------|------|------|
| 权限模型 | `ADMIN_USERNAME` 环境变量白名单，管理员/用户二元区分 | 无角色分级、无操作粒度控制 |
| 数据看板 | 三个静态数字（用户数/反馈数/总时长） | 无趋势图、无用户行为分析、无可视化 |
| 用户管理 | 列表展示（用户名/注册时间/播放时长/歌单歌曲数），只读 | 无法禁用/启用、无详情页、无批量操作 |
| 内容审核 | 仅反馈列表可删除；`blocked_sources` 表有数据但无 UI 入口 | 屏蔽源不可管理、无审核工作流 |
| 系统配置 | 所有配置依赖手动改 `.env` 重启 | 无运行时配置面板、无功能开关 |
| 操作审计 | 无 | 无操作日志、安全事件黑盒 |
| 系统监控 | 无 | 服务器/API/数据库状态完全不可见 |

## 角色权限详细模型

### 角色定义

| 角色 | 核心能力 | 限制 |
|------|---------|------|
| **超级管理员**（Super Admin） | 全部模块 + 全部操作 + 管理其他管理员 + 备份恢复 | 唯一或极少，不可被 Web 端降级/删除；`ADMIN_USERNAME` 环境变量作为兜底 |
| **管理员**（Admin） | 除「权限管理」「备份恢复」「功能开关」外的全部模块；可管理用户 | 不可删除/降级超级管理员 |
| **内容审核员**（Moderator） | 仅「内容审核」模块 + 用户标记/备注（只读用户列表） | 不可禁用/删除用户 |
| **只读观察员**（Viewer） | 数据看板（只读）+ 用户列表（只读）+ 审计日志（只读） | 不可执行任何修改操作 |

### 权限矩阵（核心操作 × 角色）

| 操作 | Super Admin | Admin | Moderator | Viewer |
|------|:---:|:---:|:---:|:---:|
| 查看数据看板 | ✓ | ✓ | — | ✓ |
| 导出看板数据 | ✓ | ✓ | — | — |
| 查看用户列表/详情 | ✓ | ✓ | ✓(列表) | ✓ |
| 编辑用户信息 | ✓ | ✓ | — | — |
| 禁用/启用用户 | ✓ | ✓ | — | — |
| 删除用户及数据 | ✓ | — | — | — |
| 重置用户密码 | ✓ | ✓ | — | — |
| 管理管理员/角色 | ✓ | — | — | — |
| 管理屏蔽源 | ✓ | ✓ | ✓ | — |
| 审核歌单/评论 | ✓ | ✓ | ✓ | — |
| 修改系统配置 | ✓ | ✓ | — | — |
| 管理功能开关 | ✓ | — | — | — |
| 备份与恢复 | ✓ | — | — | — |
| 查看审计日志 | ✓ | ✓ | — | ✓ |
| 修改告警规则 | ✓ | ✓ | — | — |

**注意**：以上为建议默认权限。如果支持自定义角色，所有单元格均可通过权限配置调整。

## 核心交互流程

### 流程一：内容审核闭环（以屏蔽源审核为例）

```
用户举报视频源
  → 系统自动创建审核项，聚合同一视频源的多条举报
  → 审核员登录后在「内容审核」看到待处理队列
  → 展开审核详情（歌曲信息、视频源信息、举报人数、举报原因分布、举报人列表）
  → 操作：
      确认屏蔽 → 视频源加入 blocked_sources → 通知举报人「已处理」
      驳回举报 → 标记为「误报」→ 通知举报人
      标记「待补充」→ 退回
  → 操作写入审计日志
```

### 流程二：运营日常巡检

```
管理员登录 → 数据看板概览页
  → 检查核心指标是否有异常波动
  → 发现播放量骤降 → 点击进入趋势详情
    → 对比「热门歌曲 Top 50」找出播放量下降最大的歌曲
    → 检查对应视频源的可用率 → 发现高播放歌曲的视频源大面积失效
    → 进入「内容审核」模块手动触发视频源健康检查
  → 导出本周数据作为周报素材
```

### 流程三：功能灰度发布

```
超级管理员 → 系统配置 → 功能开关
  → 选择目标功能 → 「灰度发布」
  → 配置目标人群：20% 活跃用户（注册 > 7 天 + 近 3 天有播放记录）
  → 确认启用
  → 回到数据看板，观察该功能的使用率、错误率、用户反馈
  → 无异常 → 逐步提升至 50% → 100% 全量
  → 每一步变更写入审计日志
```

### 流程四：用户数据合规处理

```
用户请求删除个人数据
  → 管理员在用户管理中搜索该用户
  → 查看用户详情确认身份
  → 「导出用户数据」→ 生成包含播放历史、歌单、反馈的完整 JSON
  → 用户确认数据无误
  → 「删除用户及关联数据」→ 系统展示将被删除的数据量清单
  → 超级管理员二次密码验证
  → 执行删除（或匿名化保留公共贡献）
  → 写入审计日志
```

### 流程五：备份与灾难恢复

```
日常：系统每日 03:00 自动备份 → 保留最近 7 份 → 超期自动清理

灾难恢复：
  超级管理员进入「备份与恢复」
  → 上传备份文件或选择已有备份
  → 系统校验备份文件完整性与 schema 版本兼容性
  → 弹出最终确认：「即将覆盖当前数据库，系统会先创建恢复前快照」
  → 二次密码验证
  → 执行恢复 → 服务自动重载数据
  → 写入审计日志
```

## 需确认的细节与待解决问题

**权限模型**：
- [ ] 是否需要支持「角色模板」一键套用，方便跨环境部署？
- [ ] 管理员数量是否有上限？权限扩散如何约束？
- [ ] 唯一的超级管理员账号丢失怎么办？是否需要「紧急恢复码」机制？
- [ ] 是否需要审批流（如删除用户须另一管理员确认）？

**数据看板**：
- [ ] 历史统计数据保留多久？是否需要物化视图/定时聚合以加速查询？
- [ ] 是否需要同比/环比对比能力？
- [ ] 看板数据更新方式：WebSocket 实时推送 vs HTTP 定时轮询？

**用户管理**：
- [ ] 删除用户后，其公开歌单如何处理（保留但匿名化 / 一并删除）？
- [ ] 用户数据导出格式需兼容哪些外部平台？
- [ ] 账号禁用后，已有 JWT token 立即失效还是等待过期？

**内容审核**：
- [ ] 敏感词库来源：内置预置 + 自定义 vs 对接第三方内容审核 API？
- [ ] 头像审核是否需要对接第三方图片检测（如腾讯云内容安全）？
- [ ] 审核流程是单人决策还是多人投票？

**系统配置**：
- [ ] 哪些配置热更新、哪些需要重启服务？
- [ ] 配置是否需要版本管理与回滚？
- [ ] 功能开关未开启时，对应的 JS/CSS 是否仍打包？

**审计与安全**：
- [ ] 审计日志存储位置（数据库 vs 独立日志文件）？防篡改机制？
- [ ] 是否需要对接外部安全监控系统（SIEM）？
- [ ] 备份文件是否加密？密钥如何管理？

**通用**：
- [ ] 管理面板是独立路由（`/admin`）还是沿用当前 SPA 内弹窗模式？
- [ ] 移动端管理功能需支持到什么程度？
- [ ] 多管理员同时编辑同一资源时的冲突策略（乐观锁 / 最后写入胜出 / 编辑锁）？

## 关键边界情况与异常场景

| 场景 | 风险等级 | 影响 | 处理策略 |
|------|:--:|------|---------|
| 唯一超级管理员被误删 | 🔴 严重 | 系统失去最高权限控制 | 环境变量 `ADMIN_USERNAME` 始终作为兜底，不可被 Web 端降级或删除；可选「紧急恢复码」机制 |
| 数据看板查询时间范围过大 | 🟡 中等 | 查询超时、管理面板卡死 | 默认限制 30 天，更长范围需手动确认；大查询异步生成 + 缓存 |
| 备份时磁盘空间不足 | 🔴 严重 | 备份失败甚至损坏已有数据 | 备份前检查磁盘空间，不足则拒绝 + 告警；先写临时文件，完整后原子性重命名 |
| 管理员 session 被劫持 | 🔴 严重 | 用户数据泄露或恶意破坏 | IP 绑定验证；敏感操作须重新输入密码；异地 IP 变化时强制重新登录 |
| 注册策略过松导致机器人批量注册 | 🟡 中等 | 数据库膨胀、服务降级 | 注册量异常检测 + 告警；支持批量禁用 + 清理可疑账号；邀请码模式兜底 |
| 数据库 schema 升级后旧备份无法恢复 | 🔴 严重 | 灾难恢复失败 | 备份文件包含 schema 版本号；恢复前自动校验兼容性，不匹配时提示升级路径 |
| 用户列表数据量过大导致前端卡顿 | 🟡 中等 | 管理体验差 | 服务端分页（50 条/页）；虚拟滚动；服务端排序+筛选替代前端处理 |
| 反馈含敏感内容但被搜索引擎爬取 | 🟡 中等 | 隐私泄露风险 | 管理面板路由不暴露给搜索引擎（robots.txt + meta noindex）；删除后明确提示「永久删除」 |
| 多管理员并发编辑同一配置 | 🟡 中等 | 配置被覆盖 | 乐观锁（版本号校验）或「正在被 X 编辑」提示；关键配置项串行化 |
| 功能开关误关闭核心功能 | 🔴 严重 | 全站不可用 | 核心功能开关须二次确认；提供"安全模式"自动回退列表；开关变更告警通知 |
| 批量操作中途失败 | 🟡 中等 | 数据部分变更、状态不一致 | 批量操作须事务化或提供回滚能力；操作前展示受影响范围预览；失败后列出成功/失败清单 |
| 管理员新增后原用户密码泄露风险 | 🟡 中等 | 管理员权限被滥用 | 授予管理角色时强制要求该用户重新登录 + 可选的二次验证激活步骤 |
