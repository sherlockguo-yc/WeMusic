# 部署方案：GitHub Actions 构建 + N150 统一部署代理

> 创建日期：2026-07-12
> 最后更新：2026-07-20
> 状态：✅ 已上线，端到端全自动部署验证通过

## 方案概述

```
git push master → GitHub Actions (CI):
             ① npm ci（完整依赖，用于构建）
             ② npm run build（vite + codegraph）
             ③ 创建 pkg/，PUPPETEER_SKIP_DOWNLOAD=true npm ci --omit=dev
             ④ 移除 @ffmpeg-installer / @ffprobe-installer（140M 二进制，改用系统 ffmpeg）
             ⑤ 打包 server/ + shared/ + public/ + node_modules/ + restart.sh + .version → wemusic.tar.gz（约 12M）
             ⑥ 发布 GitHub Release（pre-release，tag = git SHA）
             ⑦ curl POST 通知 N150 webhook
                                       ↓
                  N150 webhook（端口 9001，秒级响应）:
                      接收 POST → 校验项目白名单 → 写入部署队列
                                       ↓
                  N150 deploy-agent cron（每 1 分钟，兜底 + 消费队列）:
                      消费队列任务 → 下载 tar.gz → 校验 .version → rsync 到部署目录
                      → restart.sh 重启服务 → 健康检查
```

**核心设计**：webhook 推送为主（秒级部署），cron 轮询为兜底（容错）。N150 不接触源码、不构建、不 npm install。所有构建由 GitHub Actions 完成，N150 仅下载 + 解压 + 重启。

**多项目支持**：`deploy-agent.sh` 是统一部署代理，通过 `~/.deploy-projects.conf` 配置管理多个项目（WeMusic、WeMonitor 等），共享同一套部署逻辑。

## 关键设计决策

| 决策点 | 最终方案 | 理由 |
|--------|---------|------|
| 部署触发 | webhook 推送（主） + cron 轮询（兜底） | webhook 秒级生效；cron 每 1 分钟兜底，容忍 CI webhook 通知失败 |
| 版本标识 | tag = git short SHA（动态） | 每次构建独立 tag，避免 `latest` 固定 tag 的 CDN 缓存滞后问题 |
| 版本查询 | `/releases?per_page=100`，手动按 `published_at` 倒序取最新 | `per_page=30` 曾导致第 33 位 Release 被漏查；需跳过 `tag_name === "latest"` 的旧 Release |
| 下载策略 | 镜像优先，直连兜底（各 3 次重试） | `ghproxy.net` 镜像加速；镜像缓存过期触发版本校验失败时自动切直连重试 |
| Puppeteer Chromium | CI `PUPPETEER_SKIP_DOWNLOAD=true` 不打包 | 减小包体积 |
| FFmpeg/FFprobe 二进制 | CI 移除 installer 包（140M），N150 装系统 `ffmpeg`，代码优先读 `FFMPEG_PATH` 环境变量 | 包体积从 65M 降到 12M |
| 仓库可见性 | public | N150 无需 token 即可匿名查 API + 下载 |
| 部署代理位置 | `~/deploy-agent.sh`（不在项目目录内） | 避免被 `rsync --delete` 清除 |
| restart.sh 位置 | 打进 CI 产物 → `~/wemusic/restart.sh` | 随包更新，与代码版本绑定 |
| 数据库 & .env | rsync 时 `--exclude 'data' --exclude '.env'` | 保护生产数据和密钥 |
| 进程隔离 | `systemd-run --user --scope` | 脱离 cron cgroup，避免被内核资源回收误杀 |
| 并发控制 | `flock` 文件锁 | 防止多个 cron 实例或 webhook 同时执行部署 |
| 健康检查 | cron 每 1 分钟检查端口存活 | 服务崩溃后 1 分钟内自动恢复，无需人工干预 |
| 频率限制 | 同一项目 webhook 60 秒内只接受一次 | 防止 CI 重试导致重复部署 |

## Webhook 推送流程（主要触发路径）

```
CI deploy.yml 构建完成
  → curl -X POST https://wemusic.sherlockguo.com/deploy
    -H "Content-Type: application/json"
    -d '{"project":"wemusic","version":"<git SHA>"}'
  → Cloudflare Tunnel → localhost:9001
  → server/webhook.js:
      ① 校验 project 是否在 ~/.deploy-projects.conf 白名单中
      ② 校验 version 格式（/^[a-f0-9]{7,40}$/）
      ③ 检查频率限制（同一 project 60s 内只接受一次）
      ④ 写入队列文件 ~/.deploy-queue/jobs/<project>.json
      ⑤ 写入状态文件 ~/.deploy-queue/states/<project>.json
  → deploy-agent cron worker（每 1 分钟）消费队列任务
```

## Cron 兜底流程（容错路径）

```
deploy-agent.sh cron 模式（每 1 分钟）:
  ① flock 获取文件锁（防止并发）
  ② 恢复中断任务（检查 ~/.deploy-queue/running/ 孤儿任务）
  ③ 队列远程更新：对比 GitHub 最新 release 与本地 .version，有新版本则入队
  ④ 消费任务（deploy_job）:
     - 从 GitHub Releases 下载 tar.gz
       镜像优先：ghproxy.net（3 次重试）
       直连兜底：GitHub Releases API（3 次重试）
     - 校验 .version 内容（匹配目标 sha）
     - 版本校验失败 → 跳过镜像，直连重试一次
     - rsync -a --delete --exclude 'data' --exclude '.env' 同步到目标目录
     - 执行 restart.sh 重启服务
  ⑤ 健康检查（无任务时）：检查各项目端口是否响应，不响应则自动 restart.sh
```

## 状态管理

`deploy-agent.sh` 使用 JSON 文件跟踪每个部署任务的状态流转：

```
queued → downloading → verifying → syncing → restarting → succeeded
                                                         → failed
```

- 队列文件：`~/.deploy-queue/jobs/<project>.json`
- 状态文件：`~/.deploy-queue/states/<project>.json`
- 运行中标记：`~/.deploy-queue/running/<project>.json`
- 事件日志：`data/deploy-events.jsonl`（项目目录内）

## 多项目配置

`~/.deploy-projects.conf` 格式（管道分隔）：

```
项目名|GitHub仓库|部署目录|端口|tag模式|产物文件名
wemusic|sherlockguo-yc/WeMusic|$HOME/wemusic|5174|sha|wemusic.tar.gz
wemonitor|sherlockguo-yc/WeMonitor|$HOME/wemonitor|18990|sha|wemonitor.tar.gz
```

新增项目只需在此文件加一行 + CI workflow 加通知步骤，无需新增 cron 条目。

## Webhook 服务

- 端口：`9001`（仅监听 `127.0.0.1`，由 Cloudflare Tunnel 反向代理）
- 端点：`POST /deploy`（入队）、`GET /health`（健康检查）
- 认证：无（依赖项目白名单校验 + 频率限制）
- 启动：`~/webhook-start.sh`（随 wemusic 部署同步重启）

## 日常使用

改代码 → `git push` → CI 构建完成 → webhook 秒级触发部署。无需任何手动操作。

如果 webhook 通知失败（CI 网络问题等），cron 在 1 分钟内自动兜底。

## 运维备忘

- 部署事件日志：`ssh weweb 'cat ~/wemusic/data/deploy-events.jsonl | tail -20'`
- 部署代理日志：`ssh weweb 'journalctl -u cron -n 20'`（cron 输出）
- 服务日志：`ssh weweb 'tail -f /tmp/wemusic.log'`
- 当前版本：`ssh weweb 'cat ~/wemusic/.version'`
- Webhook 健康：`ssh weweb 'curl -s localhost:9001/health | python3 -m json.tool'`
- 手动触发部署（仅调试用）：`ssh weweb 'bash ~/deploy-agent.sh --deploy wemusic <sha>'`
- **禁止手动重启**：服务崩溃后 cron 健康检查在 1 分钟内自动恢复，无需人工干预

## 安全模型

- 无需 GitHub Secret 或 token
- Webhook 端点只能下载已发布的公开 GitHub Release，无法注入代码
- 项目名白名单（从 `~/.deploy-projects.conf` 动态读取，新增项目自动生效）
- 频率限制：同一项目 60 秒内最多触发一次部署
- `flock` 锁防止并发部署
- 同版本自动跳过，不重复下载

## 遗留事项

- 分享海报功能依赖 Puppeteer + Chromium。`restart.sh` 启动前自动检测并安装（`npx puppeteer browsers install chrome`），首次启动会额外耗时约 30 秒（下载 ~150MB），后续版本跳过。
- 自动备份：`scripts/backup-r2.sh` → Cloudflare R2，需先在 N150 上配置 R2 凭证和 cron。
