## 详细分析

### 问题 1：第一轮方案方向错误 🔴

**过程**：
1. 用户请求"进度条水平垂直居中"
2. 我读取 CSS，看到 `.progress-center { display: flex; justify-content: center; align-items: center; }`，认为 flex 居中足以覆盖水平 + 垂直两个方向
3. 写了 flex 居中加固 + 进度填充的方案
4. 用户反馈"没生效"
5. 用 Playwright 抓取 `getBoundingClientRect()`，发现 progress 列中心 x=662，player 中心 x=720 → **偏左 58px**
6. 才发现根因是 6 列 grid 左右不对称，需要改为 3 列 `1fr auto 1fr`

**根因**：静态 CSS 分析（"这里已经 `justify-content: center` 了"）只能判断**子元素在容器内**的居中，无法判断**容器本身**在页面中的位置。居中问题需要从外到内逐层测量——应从 `getBoundingClientRect()` 拿到数值后再判断。

**教训**：修改涉及"定位/居中/对齐"的 CSS 前，必须先抓取当前实际位置。静态读代码 ≠ 理解页面渲染结果。

### 问题 2：缺少"测量先行"规则 🔴

如果有规则要求"修改居中/定位前先测量"，则第一步就是抓取 `getBoundingClientRect()`，立即发现 58px 偏移，直接跳到第三步（3 列 grid 重构），节省第一轮无效修改。

**建议新增 Rule**：在 CSS 布局规则中添加一条：

> **修改定位/居中/对齐等 CSS 属性前，先抓取目标元素及其父容器的 `getBoundingClientRect()`，用数据判断当前偏移量和根因，再决定方案。**
