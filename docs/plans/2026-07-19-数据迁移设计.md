# 数据迁移功能设计

## 概述

在 Web 管理后台新增「数据迁移」标签页，支持管理员按用户导出/导入数据，用于从本地 WeMusic 迁移用户数据到 N150 服务器。

## 决策摘要

| 决策 | 选择 |
|------|------|
| 数据粒度 | 仅用户数据，排除全局缓存（bvid_cache / gain_cache / source_completions） |
| 目标库状态 | 清空后重新导入，无需合并逻辑 |
| 触发方式 | Web 管理后台 |
| 权限 | 仅管理员（admin 及以上） |
| 导出单位 | 按用户逐个导出 |
| 导入原子性 | 事务性导入（全部成功或全部回滚） |
| 密码处理 | 完整包含 bcrypt 密码哈希 |
| 系统数据 | 单独导出 system_config（排除 webhook_url） |

## 导出的表

| 表 | 说明 |
|---|---|
| `users` | 用户账户（username, password_hash, role, status, created_at） |
| `playlists` + `songs` | 歌单及歌曲（嵌套结构） |
| `play_logs` | 播放历史（全量） |
| `likes` | 红心收藏 |
| `user_preferences` | 偏好设置（JSON） |
| `saved_albums` | 收藏专辑 |
| `blocked_sources` | 屏蔽源 |
| `feedback` | 反馈记录 |

## 不导出的表

| 表 | 原因 |
|---|---|
| `bvid_cache` | 全局视频缓存，目标服务器重新生成 |
| `gain_cache` | 全局音量缓存，目标服务器重新生成 |
| `source_completions` | 全局众包数据，目标服务器重新积累 |
| `audit_logs` | 审计日志，属于旧服务器操作记录 |
| `sensitive_words` | 敏感词库，N150 独立管理 |

## API 端点

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| GET | `/api/migration/users` | admin | 获取可导出的用户列表 |
| GET | `/api/migration/export/:userId` | admin | 导出单个用户全部数据 |
| GET | `/api/migration/export-config` | admin | 导出 system_config（排除 webhook_url） |
| POST | `/api/migration/import` | admin | 导入用户数据（事务） |
| POST | `/api/migration/import-config` | admin | 导入 system_config |

## JSON 导出格式

```json
{
  "version": 1,
  "exported_at": "2026-07-19T...",
  "source": "wemusic-data-export",
  "user": { "username": "...", "password_hash": "...", ... },
  "playlists": [{ "name": "...", "songs": [...] }],
  "likes": [...],
  "play_logs": [...],
  "preferences": { ... },
  "saved_albums": [...],
  "blocked_sources": [...],
  "feedback": [...]
}
```

## 导入流程

1. 前端读取 JSON 文件 → 发送到服务端
2. 服务端校验 JSON 格式 → 返回预览摘要
3. 管理员确认 → 开启事务 → 逐表导入（ID 重新生成） → 提交
4. 任一步失败 → 回滚
