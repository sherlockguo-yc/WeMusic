# 推荐算法（「发现」→「为你推荐」）

## 概述

`GET /stats/recommend` 端点在 `server/routes/stats.js` 中实现，基于用户播放行为生成个性化推荐列表。

## 流程

### Step 1: 计算兴趣分数（`computeSongScores`）

扫描 90 天内播放记录，每首歌聚合为一条信号，包含以下维度：

| 信号 | 分值 | 说明 |
|------|------|------|
| 完播率 ≥ 80% | +1.5 | 用户完整听完 |
| 完播率 20%~80% | +0.3 | 听了一大半 |
| 完播率 < 20% | –0.3 | 跳过了 |
| 重复播放 | +0.5/次，上限 +3.0 | 听了越多遍越喜欢 |
| 红心 | +3.0 | 直接喜欢 |
| 加入歌单 | +2.5 | 收藏到歌单 |
| 时间衰减 | ×(1–ageRatio×0.5) | 越久远越不重要（90 天线性衰减） |

额外补充：纯红心/歌单收藏（无播放记录）的歌曲默认 +3.0/+2.5。

最终 key 为 `歌名__歌手`。

### Step 2: 歌手权重（`computeArtistWeights`）

将 Step 1 的歌曲分数聚合到歌手维度：
- 独唱：全部分数归该歌手
- 合唱（含 `/`、`、`、`,`、`&`）：第一位是主唱（权重 1.0），其余客串（权重 0.4）
- 过滤：歌手总分 < 2.0 的丢弃（防止合唱客串被误判为偏好歌手）

取权重前 3 名作为「偏好歌手」。

### Step 3: 风格扩散种子

从 Step 1 中选取 score ≥ 2.0、歌名 ≥ 2 字的中文歌名，随机取最多 3 首。这些种子的歌名会用于搜索相似风格的翻唱版本。

### Step 4: 并行拉取

三种来源并行请求：

| 来源 | type | 方法 | 分值 |
|------|------|------|------|
| 偏好歌手 × 3 | `artist` | `searchSongsForRecommend()` — 拉歌手全部歌曲 | +2.0 |
| 风格扩散 × 3 | `style` | `searchSongs("歌名 翻唱")` — 搜索翻唱版本 | +1.5 |
| 热门兜底 | `chart` | `getTopList(26, 20)` — 热歌榜 | +1.0 |

### Step 5: 合并评分

所有来源的结果合并到 `candidateMap`（key = `歌名__歌手`）。每条候选：
- 基础分 = 来源分值
- 已听过 → –1.5（降低但不排除）
- 风格扩散中种子歌手 → –0.8（避免和歌手深度挖掘重复）
- 已有负分信号 或 用户不喜欢 → **跳过**

### Step 5.5: 同歌名去重（2026-07-15 新增）

**问题**：风格扩散搜索「歌名 翻唱」会拉回同一首歌的多个不同歌手版本，导致推荐列表中同一歌名（如「晴天」）出现 10+ 次，占据大量篇幅。

**方案**：
- 去掉括号内容（`晴天 (Live)` → `晴天`）归一化歌名
- 每个归一化后的歌名**最多保留 2 个版本**（按 `_candidateScore` 降序取前 2）
- 日志输出去重统计：`[recommend] name-dedup: N candidates → M (K songs had >2 versions)`

**限制值选择**：
- 2 个版本：保留原唱 + 一个优质翻唱，足够多样化
- 3 个以上：用户反馈「感觉推了很多重复的歌」，体验差

### Step 6: 分桶排序

- High bucket（score ≥ 1.5）：随机打乱后取前 30
- Mid bucket（0 ≤ score < 1.5）：随机打乱后取前 20
- 合并后总上限 50 首
- 随机打乱（`shuffle`）增加多样性，避免每次推荐顺序完全一致

### 冷启动

若 `songScores.size === 0`（新用户无任何播放记录），直接返回热歌榜 top 40。

## 修改注意事项

1. **搜索/匹配逻辑前置步骤**（参考项目 rule）：修改搜索候选逻辑前，先构造复现场景 + 分析根因 + curl 验证
2. **修改后重启**：改了 `server/routes/stats.js` 后必须 `bash restart.sh`
3. **同名去重不是全局去重**：Step 5.5 的归一化去重只在推荐管线的最后一个环节生效，不影响搜索、歌词候选等其他模块
