## 二、歌词源候选逻辑

### 2.1 搜索入口

`fetchLyrics(name, singer)`: 返回**最佳匹配**（主歌词显示）  
`searchLyricsCandidates(name, singer)`: 返回**候选列表**（歌词换源弹窗）  
`fetchLyricsById(songId)`: 直接按网易云 songId 拉取（用户手动切换）

两者共用 `pickBest()` 和相同的候选评分逻辑。

### 2.2 搜索策略（多 query 并行）

对每个搜索关键词，调用网易云 `search` 接口。query 生成顺序：

1. `{name} {singerFirst}` — 歌名 + 歌手优先
2. `{nameClean} {singerFirst}` — 去括号后的歌名 + 歌手
3. `{name}` — 仅歌名
4. `{nameClean}` — 去括号后的歌名

> `singerFirst` = `singer.split(/[\/、,，&]/)[0].trim()` — 取第一个歌手名  
> `nameClean` = `stripBrackets(name)` — 去掉括号及内容，如 `El Hombre (笑面人)` → `El Hombre`

所有 query **并行发出** (`Promise.allSettled`)。

### 2.3 候选评分（quality）

对每个搜索结果按 3 个信号打分：

| 信号 | 条件 | 分值 |
|---|---|---|
| 歌名完全匹配 | `result.name === name`（大小写不敏感）| **+4** |
| 歌手匹配 | `singerFirst` ≥ 2 字符，且在 artist 名中以**词边界**形式出现 | **+3** |
| 歌名模糊匹配 | `result.name` 包含 `name`（子串）| **+1** |

> **歌手名匹配规则（含词边界检查）：**
> - `artist.name.toLowerCase()` 包含 `singerFirst.toLowerCase()` → 匹配
> - 或满足正则 `(^|[^\w])singerFirst($|[^\w])` → 匹配
> - 单字符歌手名（如"G"）不参与歌手匹配，避免误匹配

### 2.4 候选过滤（topCandidates）

按 quality **递减排序**后，采用**三层兜底**策略，确保冷门歌曲仍有候选：

```
第 1 层：quality >= 2（至少有歌名模糊匹配或歌手匹配）
第 2 层：如果第 1 层不足 8 个，补充 quality >= 1（至少歌名模糊匹配）
第 3 层：如果仍不足 5 个，补充 quality < 1 的兜底候选
```

最终上限 20 个候选。然后**并行拉取**每个候选的歌词正文 → 过滤空内容 → 返回最多 12 个。

### 2.5 pickBest（主歌词选取）

与候选评分逻辑一致，但选出的是**一个最佳匹配**直接展示，规则：

1. 优先：`exactName && hasSinger`（歌名完全匹配 + 歌手匹配）
2. 次之：`exactName`（歌名完全匹配，不强制歌手）
3. 兜底：第一个有结果的 query 的第一个有效结果

### 2.7 QQ 音乐歌词源（双源并行，2026-07-04 实现）

#### 概述

除网易云音乐外，并行搜索 QQ 音乐歌词源，增大候选池覆盖率。网易云无版权的歌曲（如周杰伦《外婆》）可通过此源获取歌词。

#### 搜索接口

| 接口 | 用途 |
|---|---|
| `smartbox_new.fcg` | QQ 音乐联想搜索（歌名/歌手联想，前缀匹配） |
| `fcg_query_lyric_new.fcg` | QQ 音乐歌词正文（XML/CDATA 格式，内含 Base64+LRC） |

#### 双源去重策略

1. 对同一组 query 并行调用网易云 `neSearchSongs` 和 QQ 音乐 `searchQQCandidates`
2. QQ 音乐结果进行 `scoreCandidate` 评分（与网易云同权重体系）后合并
3. 按 `songid` / `mid` 去重（songid 优先），两个源同分时 QQ 音乐优先（质量更好）
4. 最终候选列表一起走 `topCandidates` 三层兜底过滤

#### 歌手置信度校验（防止 smartbox 假匹配）

smartbox 有时会误识别歌手（如"外婆"→"外婆的彭湖湾"）。校验逻辑：
- 候选歌手名**必须**出现在 `fullSearch` 或 smartbox 已搜到的歌曲的 `singer` 字段中
- 不满足 → 视为假匹配，丢弃候选歌手，直接使用关键词搜索结果

#### 代码位置

- `server/services/lyrics.js`：`qqSearchSongs()` / `qqFetchLyric()` / `searchQQCandidates()` / `scoreCandidate()`
- `server/services/qqmusic.js`：`searchSongs()` → 歌手置信度校验
- `server/routes/stats.js` → `/stats/lyrics`：`sourceId` 以 `qq:` 前缀判断分支

### 2.8 缓存 & 黑名单

- `setLyricCache` / `getLyricCache`：同名同歌手 24h 内复用（内存缓存，上限 200 首）
- 黑名单过滤：`blocked_sources` 表 → 用户在歌词换源弹窗中点 ✕ 屏蔽的 sourceId 会被排除

## 三、视频源候选逻辑

### 3.1 搜索入口

`POST /play/resolve`: 自动匹配（播放歌曲时自动调用）  
`GET /play/search`: 手动搜索（用户点"在B站搜索此歌"）

### 3.2 搜索策略（并行 Wave）

**resolve 自动匹配**：query 按歌手名逐片段生成：

对歌手名的每个 ≥2 字符片段（按 `/[\\/、,&\\s]+/` 拆分），第一段生成完整组 query，其余段各生成一条精简 query：

```
歌手片段 = singer.split(/[\\/、,&\\s]+/).filter(s => s.trim().length >= 2)
singerFirst  = 片段[0]        // 原始大小写保留，用于基础 query
singerExtra  = 片段[1:]        // 每个额外片段一条 "歌名 + 该片段" query
```

基础 query（`singerFirst` 非空时）：

1. `{name} {singerFirst}` — 歌名 + 歌手第一段
2. `{name} {singerFirst} MV` — + MV 后缀
3. `{name} {singerFirst} 官方` — + 官方后缀
4. `{name}` — 仅歌名（永远追加）
5. `{singerFirst} {name}` — 歌手+歌名逆向

额外片段 query（`singerExtra` 每个元素一条，去重后追加）：

6. `{name} {extraSegment}` — 每段单独一条（例如 `"我愿意 邓紫棋"`）

括号中文名 query（如有）：

N+1. `{bracketCN} {singerFirst}` — 括号中文名 + 歌手第一段（如 "笑面人 G.E.M."）
N+2. `{bracketCN}` — 仅括号中文名

> **示例**：`"G.E.M. 邓紫棋"` → `singerFirst = "G.E.M."`，`singerExtra = ["邓紫棋"]`。生成 6 条 query：`我愿意 G.E.M.` / `我愿意 G.E.M. MV` / `我愿意 G.E.M. 官方` / `我愿意` / `G.E.M. 我愿意` / `我愿意 邓紫棋`。
>
> **singerExtra 的设计理由**：B 站对小众/冷门翻唱视频的标题/UP 名索引权重不统一。"英文名+中文名"双名的歌手（如 G.E.M. 邓紫棋），部分视频标题仅含中文名（如「邓紫棋翻唱王菲【我愿意】」），纯英文名的 3 条 query 永远搜不到。每个额外片段都单独生成一条 query，确保覆盖到只被该片段索引到的视频。这是本项目 2026-07-17 真实漏召 bug 的根因修复。

分两波：
- **Wave 1**：所有 query × page 1 并行 → 最快 ~500ms
- **Wave 2**：如果结果 < 80 条，所有 query × page 2 并行

**手动搜索（/play/search）**：同样采用逐片段拆分策略：
1. 组合关键词 `{keyword}`（或 `{name} {singer}`）→ 2 页
2. 纯歌名 `{name}` → 2 页（当 name ≠ keyword 时追加）
3. 歌手名每个 ≥2 字符片段 `{name} {fragment}` → 各 1 页
   - 例如 `"G.E.M. 邓紫棋"` 额外生成 `"我愿意 G.E.M."` 和 `"我愿意 邓紫棋"` 各 1 页

### 3.3 排除规则（isExcluded）—— 已移除

> **2026-07-15 移除** `isExcluded()` 硬过滤。
> 原因：存在本身就是「纯音乐 / 伴奏 / instrumental / karaoke / 消音 / 人声消除」的歌曲，硬过滤会导致这类歌曲完全搜不到音频源 / 视频源。
> 现在这些标题不再被直接剔除，而是交由评分函数（§3.5）与排序（§3.6）竞争排序；用户仍可通过屏蔽源（blocked_sources）手动排除不想要的版本。

其它类别（翻唱 / cover / remix / DJ、教学 / 教程、Live / 现场 / 演唱会、Reaction / 点评、鬼畜 / 剪辑、合集 / mashup、8D / nightcore / 重制、乐器独奏版等）**并非**由 `isExcluded` 处理，而是分别走降权（`BAD_KW`）或独立逻辑（`LIVE_KW` / `isLive`），不受影响。

### 3.4 歌名片段过滤（nameSegments）

歌名 ≥ 2 字符时，要求视频标题**至少包含歌名的一个 ≥2 字符片段**：

- 中文：拆成 2-gram（如"少年与海"→"少年"/"年与"/"与海"）
- 拉丁/数字：按词语边界切分

**至少一个片段出现在标题中**，否则直接排除。避免"邓紫棋 官方 MV"的《无双的王者》出现在搜索 "Hell" 的候选中。

### 3.5 评分函数（scoreVideo / scoreBreakdown）

`scoreVideo()` 内部调用 `scoreBreakdown()`（返回评分明细），取 `Math.round(total)` 作为最终分数。评分维度：

| 维度 | 信号 | 分值 | 说明 |
|------|------|------|------|
| 歌名 | 完全匹配 | +50 | 标题包含完整歌名 |
| 歌名 | 字符级匹配 | ≤+30 | 匹配字符数 / 歌名长度 × 30 |
| 歌手 | 标题含歌手名 | +30 | 标题直接包含歌手第一段名 |
| 歌手 | UP 主名含歌手 | +20 | 降级匹配 |
| 时长 | 差 ≤ 10s | +40 | 与原曲时长差 |
| 时长 | 差 ≤ 25s | +22 | |
| 时长 | 差 ≤ 60s | +6 | |
| 时长 | 差 > 60s | -25 | 惩罚 |
| 播放量 | ≥ 1000万 | +55 | 分级加权（对数尺度） |
| 播放量 | ≥ 100万 | +40 | |
| 播放量 | ≥ 10万 | +25 | |
| 播放量 | ≥ 1万 | +12 | |
| 品质 | 高音质关键词 | +30 | 无损/母带/HiRes 等，命中 `HQ_KW` 列表中任意关键词 |
| 品质 | 官方 + MV | +42 | 标题同时含"官方"和"mv" |
| 品质 | 仅官方 | +28 | |
| 品质 | 仅 MV | +20 | |
| 品质 | 其他优质关键词 | +10/个 | 完整版/原版/audio 等（`GOOD_KW` 列表） |
| 品质 | UP 主 ≡ 歌手名 | +18 | UP 主名完全匹配歌手名（可能是官方频道） |
| 降权 | 降权词 | -14/个 | 翻唱/cover/教学/remix/DJ/AI 等（`BAD_KW` 列表） |
| 众包 | crowdBonus | 可变 | 用户之前选择过的 bvid 获得额外加分 |

### 3.6 排序（rank）

1. **歌手准确度优先**：如果提供了歌手名，先筛出标题/UP 包含该歌手的视频，只保留这部分；如果一条都没有，不触发过滤（保留全部）
2. **现场版处理**：
   - 普通歌名 → 非现场版优先（`a.live ? 1 : -1`）
   - 歌名含 live/现场 → 取消惩罚 + 现场版主动提权 +30（由 `songNameSuggestsLive()` 检测）
3. **按 score 递减**

#### 3.6.1 Live 提权：歌名暗示现场版时（2026-07-09 新增）

当歌名本身包含 live/现场类关键词（如 `"晴天 (Live)"`、`"小美满 现场版"`）：

1. **取消 live 惩罚**：不执行「非现场优先」排序，所有候选按 score 竞争
2. **主动提权 +30**：被 `isLive()` 标记为现场版的候选，score 额外 +30

检测函数：`songNameSuggestsLive(name)`，检查歌名是否包含 `live`、`现场`、`演唱会`、`现场版`（大小写不敏感）。

行为对比：
- 普通歌名（如「晴天」）→ 非现场优先 + 不做任何加分，行为不变
- 歌名含 live（如「晴天 (Live)」）→ 取消 live 惩罚 + live 版 +30 → 高分的现场版排在最前

### 3.7 黑名单过滤

`blocked_sources` 表查询当前用户 + 当前歌曲的已屏蔽 bvid 集合 → 从结果中排除。

### 3.8 Debug 面板（2026-07-16 新增）

在视频换源弹窗中新增「Debug」tab，显示 `POST /play/resolve` 的完整匹配过程。数据通过响应中的 `_debug` 字段返回。

## 五、常见问题 & 修改注意事项

### Q: 为什么某个完全无关的视频/歌词出现在候选顶部？

**视频**：检查 `nameSegments()` 是否被绕过（如搜索词参数错误导致歌手名字段混入歌名字段）  
**歌词**：检查 `hasSinger` 的词边界正则是否生效，单字符歌手名是否被排除

### Q: 为什么一首歌完全没有候选？

**视频**：检查 B 站是否风控（502），或歌名片段过滤是否过严（零片段匹配）  
**歌词**：检查 quality 三层兜底是否正常执行，或网易云 API 是否返回了任何结果

### Q: candidate 质量整体变差了？

常见误操作：
1. **改了评分权重**但没有相应调整过滤阈值
2. **加了硬质量门槛**（如 `>= N`）没有兜底逻辑 → 冷门歌曲零候选
3. **去掉了并行搜索** → 候选池缩小，质量下降
4. **改了 quality 分值**导致排序混乱（exactName +4 / hasSinger +3 是经验值）

## 六、已知陷阱 & 历史 Bug

### 6.1 nameSegments 括号 Bug（2026-07-02 修复）

**症状**：歌名含括号 `"El Hombre que Ríe (笑面人)"` → segments 变成 `"El Hombre que Ríe ("` + `"笑面"` + `"面人"`。第一个 segment 含 `(`，B 站标题不含 `(`，导致片段过滤**全灭**。

**根因**：`nameSegments` 中 `(` / `)` 既非中文也非拉丁，落入 else 分支被追加到当前 buffer。

**修复**：括号字符 `()（）` 作为 buffer 结束符，不被加入任何 segment。

### 6.2 括号中文名未搜索（2026-07-02 修复）

**症状**：`"El Hombre que Ríe (笑面人)"` 的中文译名 "笑面人" 只在歌名括号内，未独立作为搜索词。

**修复**：提取 `bracketCN`（正则 `/[(（]([^)）]+)[）)]/`），视频和歌词均新增独立的 `{bracketCN} {singerFirst}` 和 `{bracketCN}` 搜索 query。

### 6.3 屏蔽后数组下标失效（2026-07-07 修复）

**症状**：歌词/视频换源弹窗中屏蔽第 2 个候选时，实际屏蔽的是第 3 个候选。刷新后 "该出现的消失了，想屏蔽的还在"。

**根因**：`openCandModal` / `openLyricsSwitchModal` 用 `data-i`（数组下标）关联 DOM 和 `candidates[]`。屏蔽第一个候选后 `candidates.filter(...)` 让数组变短、下标重排，剩余 DOM 行的 `data-i` 已成旧下标 → `candidates[行.data-i]` 拿到错误的候选。

**修复**：改用稳定 ID 关联（`data-id`/`data-bvid`），屏蔽/选择时通过 `findById()` 查找而非数组下标。

### 6.4 主源被屏蔽后「当前」标记丢失（2026-07-07 修复）

**症状**：用户屏蔽了当前正在使用的歌词源后，候选列表中没有任何一行显示「当前」。localStorage 缓存（`wemusic_lyrics_src`）还指向已屏蔽的源，刷新后又加载到被屏蔽的源。

**根因**：
1. `fetchLyrics(name, singer)` 返回的 `main.sourceId` 可能已在黑名单中，但候选列表已过滤掉该 sourceId → 没有任何候选的 id 能匹配 `lyricsCurrentSourceId`
2. `loadLyrics(song)` 从 localStorage 读了缓存 sourceId 作为 `forceSourceId` 传给服务端 → `/stats/lyrics` 的 `sourceId` 分支不检查黑名单，直接拉取已屏蔽源的歌词

**修复**：
- `/stats/lyrics` 三个分支（sourceId 显式传参 / cache HIT / cache MISS）都检查 `main.sourceId` 是否在黑名单中，是则从候选中取最优替代并重新拉取歌词（`fetchLyricsForCandidate`）
- 服务端 `/stats/lyrics` 返回的 `sourceId` 始终是非屏蔽源
- 前端屏蔽按钮：屏蔽的是当前源时清理 localStorage 缓存 + 清空 `lyricsCurrentSourceId`
- 模态框独立拉取候选时捕获服务端返回的 `sourceId` 同步到 `lyricsCurrentSourceId`

### 6.5 屏蔽弹窗底部列表不实时刷新（2026-07-07 修复）

**症状**：用户屏蔽一个源后，「已屏蔽的源」折叠区仍是旧 HTML，需关闭弹窗再打开才显示。

**修复**：屏蔽按钮 handler 中，屏蔽成功后删除旧 `.cand-blocked-section` → 重新 `fetchBlockedList` → `insertAdjacentHTML` 追加新 HTML → 重新绑定事件。

### 6.7 离线缓存短路忽略 forceSourceId，导致切换歌词源无效（2026-07-14 修复）

**症状**：在歌词详情页点「歌词」换源，选了不同的候选源，但显示的歌词始终是最初缓存的那一版，切换无效。

**根因**：`src/lyrics.js` 的 `doLoadLyrics(song, forceSourceId)` 在 2026-07-14 的「离线缓存歌词整包落盘」提交中新增了离线缓存分支：

```js
if (song.bvid) {
  const cached = await offline.get(song.bvid);
  if (cached && cached.lyrics) {
    ...
    return true;   // ← 直接返回，forceSourceId 被整个丢弃
  }
}
```

换源点击走的是 `doLoadLyrics(song, c.id)`（即 `forceSourceId = c.id`）。只要该歌曲有 `bvid` 且离线缓存里有歌词整包，函数就在离线分支提前 `return`，永远返回最初缓存的源，用户的选择被无视。

**修复**：离线缓存仅在「未指定源」或「离线缓存的源与请求的源一致（`cached.lyrics.sourceId === forceSourceId`）」时才复用并提前返回；否则落到网络请求（`/stats/lyrics?...&sourceId=...`）以尊重用户选择。

**教训**：任何「离线/缓存优先」的短路分支，都必须先校验调用方传入的「显式意图参数」（如 `forceSourceId`），不能无条件 `return`。

### 6.6 Live 惩罚误伤含 Live 的歌名（2026-07-09 修复）

**症状**：用户听 `"晴天 (Live)"` 这类歌名含 live 关键词的歌曲时，B 站高分现场版总是被低分的 studio/MV 版压到列表底部，auto-play 选中的也是非现场版。

**根因**：`rank()` 无条件执行 `!v.live` 优先排序（`a.live ? 1 : -1`），`/play/resolve` 的 `best` 选取也是 `clean.find(v => !v.live)`。对于歌名本身就是 live 版的歌曲，现场版反而是正确答案。

**修复**：
- 新增 `songNameSuggestsLive(name)` 检测歌名是否含 `live`/`现场`/`演唱会`/`现场版`
- `rank()`：歌名暗示 live → **取消 live 惩罚** + 为 live 版 **主动提权 +30**，统一按 score 排
- `/play/resolve` `best`：歌名暗示 live → 直接取 `clean[0]`
- 普通歌名（不含 live 关键词）→ 行为完全不变（非现场优先 + 无提权）

### 6.8 singerFirst 拆分正则不一致，导致含空格歌手名 query 不完整（2026-07-16 修复）

**症状**：搜索「我愿意」+「G.E.M. 邓紫棋」时，`singerFirst` 被计算为 `"G.E.M. 邓紫棋"`（整个字符串），导致搜索 query 将「G.E.M.」和「邓紫棋」作为一个整体发送给 B 站，匹配效率低。Debug 面板中 `singerFirst` 显示为空（因为旧正则 `/[\\/、,&]/` 不包含空格分隔符，对某些编码格式会产出空串）。

**根因**：`singerFirst` 使用的正则 `/[\\/、,&]/` 与 `singerParts()` 的 `/[\\/、,&\\s]+/` 不一致。前者不处理空格分隔且结果无长度过滤。

**修复**：`singerFirstForQuery` 改用 `/[\\/、,&\\s]+/` 正则 + `>= 2` 长度过滤（与 `singerParts` 一致），但保留原始大小写用于搜索 query。例如 `"G.E.M. 邓紫棋"` → `singerFirstForQuery = "G.E.M."`，`singerParts = ["g.e.m.", "邓紫棋"]`。

### 6.9 歌手名多片段搜索漏召（2026-07-17 修复）

**症状**：搜索「我愿意 - G.E.M. 邓紫棋」时，B 站 WBI API 仅返回 2 个候选（均为现场版），另一个完全可用的视频源「邓紫棋翻唱王菲【我愿意】」从未出现在候选列表。用户通过分享链接拿到该视频的 bvid，在换源弹窗中看不到它。

**根因**：`singerFirstForQuery` 只取了歌手名的**第一段**（`"G.E.M."`），所有搜索 query 都是英文名 + 后缀的变体，从未用中文名 `"邓紫棋"` 作为 query。这个视频标题中不含 `G.E.M.`，只有 `邓紫棋`，且播放量仅 1346，在纯英文名 query 的前 40 条结果中排不进去。

B 站 WBI API（`/x/web-interface/wbi/search/type`）与 B 站网页搜索（`search.bilibili.com`）使用**不同的搜索引擎**——网页搜索基于 Elasticsearch 有实体识别（能把 "G.E.M. 邓紫棋" 识别为歌手实体），WBI API 是简单关键词匹配。同为 `order=totalrank`，两者返回结果差异很大。WBI API 对低频翻唱视频的索引权重低，必须精确命中标题中实际出现的关键词。

验证过程：实际调用 B 站 API 对比多种 query 组合：

| query | 是否命中目标视频 |
|-------|---------------|
| `我愿意 G.E.M.` | ❌ |
| `我愿意 G.E.M. MV` | ❌ |
| `我愿意 G.E.M. 官方` | ❌ |
| `我愿意 G.E.M. 邓紫棋`（完整拼接）| ❌（词多被稀释） |
| **`我愿意 邓紫棋`（仅中文名）** | **✅ 第 5 位** |

**修复**：
1. `singerFirstForQuery` 不再只用第一段 → 所有 ≥2 字符的歌手片段（`singerQueryParts`）都参与 query 生成。第一段生成完整基础组（3 条），其余段各生成一条精简查询（`{name} {extra}`）
2. 同一逻辑同步到 `/play/search` 手动搜索端点
3. 验证脚本确认：修复前 5 条 query 未命中 → 修复后 6 条 query 命中目标视频

**教训**：B 站 WBI API 和网页搜索是不同的搜索引擎，不能假定"用户手动能看到的结果 = API 也会返回"。对小众/冷门翻唱视频，必须确保 query 覆盖到视频标题中实际存在的所有关键词，多片段歌手名尤甚。

### 6.10 `_biliTitle` 在 bvid 已预设时缺失，播放栏/视频面板显示后备文本（2026-07-17 修复）

**症状**：从分享链接打开的歌曲（`song.bvid` 已由分享方预设），播放器状态栏显示 `"我愿意 - G.E.M. 邓紫棋"`（`${name} - ${singer}` 后备），而非实际的 B 站视频标题 `"邓紫棋翻唱王菲【我愿意】"`。视频面板标题也回退到 `"Bilibili 播放"`。

**根因**：`playCurrent()` 中获取 bvid / `_biliTitle` 的逻辑全部包裹在 `if (!song.bvid)` 分支内。分享链接、会话恢复等场景下 `bvid` 已被预设，整个分支被跳过，`_biliTitle` 永远为空 → 播放栏和视频面板均走 fallback。

**修复**：在 `if (!song.bvid)` 的两个分支之后新增第三个分支：`if (song.bvid && !song._biliTitle)` → 调 `/play/resolve` 获取候选列表 → 用现有 bvid 匹配对应 title → 同步写回 `song._biliTitle` / `song._biliDur` → `cacheBvid()`。

**影响入口**：分享链接、服务端/本地会话恢复、其他任何「bvid 已预设但未设 `_biliTitle`」的路径。
