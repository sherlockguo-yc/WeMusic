# 数据统计指标优化方案

> 版本：v2.0 | 日期：2026-07-05 | 状态：已实现

---

## 一、问题背景

### 1.1 旧方案的三个核心缺陷

**缺陷 1：跳过率使用绝对 30s 阈值**

```sql
-- 旧方案
SUM(CASE WHEN played_sec < 30 THEN 1 ELSE 0 END) AS skipped
```

- 一首 30 秒的 intro track 播放了 28 秒，也算"跳过"——语义错误。
- 一首 10 分钟的交响乐章播放了 25 秒，和播放 29 秒的 3 分钟流行歌，都算"跳过"——粒度太粗。
- 对所有歌曲一视同仁，忽略了歌曲时长差异。

**缺陷 2：新歌发现没有最低参与度门槛**

```sql
-- 旧方案：任何已上报的播放（≥5s）都算"发现"
SELECT COUNT(*) AS cnt FROM play_logs WHERE ... NOT IN (历史播放)
```

- 浏览式快速切歌（每首 6 秒）会虚高新歌发现数。
- 播放 5 秒不算"发现"一首歌，但旧方案会计入。
- 另有隐藏 bug：`COUNT(*)` 而非 `COUNT(DISTINCT)`，同一首新歌播放 10 次会计为 10。

**缺陷 3：平均播放时长使用全部播放做分母**

```js
// 旧方案：全部播放行数做分母
avgSec = totalSec / totalPlays;
```

- 5~30 秒的浅层试听拉低平均值。
- UI 标签"场均时长"有误导性——实际是"均首"而非"均场"。

### 1.2 现有保护机制（保留）

前端上报已有两道门槛（`src/player.js`）：

1. **3 秒延迟窗口**：`logPlay()` 中 `setTimeout(3000ms)`，播放不到 3 秒的歌曲不设为待上报。
2. **5 秒最低上报阈值**：`_flushLog()` 中 `if (sec < 5) return;`，播放不足 5 秒的歌曲不写入数据库。

这两道门槛滤掉了极端的"不到 1 秒切歌"，但 **5~30 秒区间**的播放仍在统计中造成噪音。

## 二、核心架构设计

### 2.1 设计原则

1. **统一过滤层**：定义一个"有效播放"表达式，作为所有参与度相关指标的唯一真相来源（Single Source of Truth）。
2. **退耦常量**：过滤表达式定义为独立常量，不重复出现在各查询中，未来调参只改一处。
3. **向下兼容**：保留旧字段（`plays`、`sec`），新增字段（`meaningfulPlays`、`meaningfulSec`），旧客户端不受影响。
4. **前端 fallback**：前端在 `meaningfulPlays` 不存在时，自动退回到 `plays`（兼容缓存/旧版）。

### 2.2 核心概念：有效播放（Meaningful Play）

```
有效播放 = played_sec >= 30 秒
          OR
          (duration > 0 AND played_sec / duration >= 0.4)
```

| 歌曲时长 | 满足 30s 门槛 | 需要满足 40% | 实际有效门槛 | 说明 |
|---------|-------------|------------|------------|------|
| 15s（音效） | 需 ≥ 30s | 需 ≥ 6s | **6s** | 40% 比例保护了短歌 |
| 30s（intro）| 需 ≥ 30s | 需 ≥ 12s | **12s** | 短歌也能公平纳入 |
| 1min | ≥ 30s ✓ | 需 ≥ 24s | **24s** | 30s 绝对门槛优先 |
| 3min | ≥ 30s ✓ | 需 ≥ 72s | **30s** | 30s 门槛生效 |
| 10min（交响）| ≥ 30s ✓ | 需 ≥ 240s | **30s** | 30s 足够表明参与意愿 |

**关键洞察**：30 秒绝对门槛和 40% 相对门槛是 **OR 关系**。30 秒是大多数歌曲的守护门槛，40% 比例用于保护短歌不被误伤。

### 2.3 跳过判断（独立于有效播放）

```
跳过 = (duration > 0 AND played_sec / duration < 0.25)
       OR
       ((duration IS NULL OR duration = 0) AND played_sec < 30)
```

跳过率使用比有效播放更敏感的 **25% 阈值**（vs 有效播放的 40%）。这是因为"跳过"和"是否认真听"是不同概念——你可能听了 35% 的歌曲但仍然是在认真听（不算有效播放但也不算跳过），也可能只听了 20% 就决定不喜欢这首歌（算跳过）。

| 场景 | 歌曲时长 | 播放秒数 | 进度 | 算跳过？ | 算有效播放？ |
|------|---------|---------|------|---------|------------|
| 快速切歌 | 3min | 6s | 3% | 未上报（<5s） | — |
| 浅尝即弃 | 3min | 20s | 11% | ✓ | ✗ |
| 听了一段 | 3min | 45s | 25% | ✗（=25%边界） | ✓（≥30s） |
| 听完短歌 | 30s | 12s | 40% | ✗ | ✓ |
| 不确定时长 | 未知 | 20s | — | ✓（<30s 回退） | ✗ |

### 3.1 整体架构图

```
┌──────────────┐    POST /stats/log    ┌──────────────┐
│  前端 player.js  │ ──────────────────> │  stats.js API  │
│  _flushLog()    │    {played_sec, dur} │  INSERT INTO   │

└──────┬─────────┘                      └──────┬────────┘
       │                                       │
       │ front-end call                        │ SQL queries
       ▼                                       ▼
┌──────────────┐                      ┌──────────────┐
│  report.js    │ <──── JSON ──────── │  buildReport()│
│  前端展示层    │                    │  统计计算层    │
│  avgSec,      │                    │  MEANINGFUL_  │
│  skipPct,     │                    │  COND,        │
│  persona      │                    │  SKIP_COND    │
└──────────────┘                    └──────────────┘
```

### 3.2 三层职责

| 层 | 文件 | 职责 | 关键逻辑 |
|----|------|------|---------|
| **数据采集层** | `src/player.js` | 前端播放时长上报 | 3s 延迟窗口 + 5s 最低阈值，阈值不变 |
| **统计计算层** | `server/routes/stats.js` `buildReport()` | 服务端 SQL 聚合计算 | 注入有效播放/跳过条件，进行聚合统计 |
| **展示层** | `src/report.js` `buildReportHtml()` | 前端渲染周报/月报 | 使用 meaningfulPlays/Sec 计算均时，渲染 UI |

### 3.3 数据流转时序

```
1. 用户播放歌曲 A，6 秒后切歌
   → player.js: logPlay(B, durB) 调用
   → _flushLog(elapsed=6): sec=6, ≥5 → POST /stats/log {played_sec:6, duration:180}
   → play_logs 写入一条记录 {played_sec=6, duration=180}

2. 一周后，用户打开「我的数据」→ 本周报告
   → GET /stats/weekly
   → buildReport(uid, mon, now, ...)
     → periodT:      plays=1, sec=6, meaningful_plays=0（6<30s, 6/180=3.3%<40%）
     → skipRow:      total=1, skipped=1（6/180=3.3%<25%）
     → newSongs:     cnt=0（不满足有效播放条件）
   → JSON 返回前端

3. 前端 report.js 处理
   → skipPct = 1/1 * 100 = 100%（跳过率 100%）
   → meaningfulPlays = 0, fallback to plays=1
   → avgSec = 6/1 = 6s
```

## 四、核心算法与实现

### 4.1 SQL 常量定义（退耦设计）

```javascript
// server/routes/stats.js:86-92
const MEANINGFUL_COND = "played_sec >= 30 OR (duration > 0 AND CAST(played_sec AS REAL)/duration >= 0.4)";
const SKIP_COND = "(duration > 0 AND CAST(played_sec AS REAL)/duration < 0.25) OR ((duration IS NULL OR duration = 0) AND played_sec < 30)";
```

**设计决策**：使用 JS 常量字符串 + 模板字面量 `${COND}` 注入 SQL，而非创建 SQL 视图或 CTE。
- **理由 1**：`better-sqlite3` 不支持参数化 SQL 片段，`?` 不可用于注入 SQL 关键字。
- **理由 2**：`MEANINGFUL_COND` 直接引用列名 `played_sec` 和 `duration`，与 `CREATE VIEW` 相比零运行时开销。
- **理由 3**：一处定义，多处模板注入，修改阈值（如 30→45）只需改一个数字。

### 4.2 概览查询（注入有效播放统计）

```sql
-- 在原有 SELECT 中附加两个 SUM(CASE) 聚合
SELECT COUNT(*) AS plays,
       SUM(played_sec) AS sec,
       COUNT(DISTINCT name||singer) AS unique_songs,
       COUNT(DISTINCT date(...)) AS days,
       SUM(CASE WHEN ${MEANINGFUL_COND} THEN 1 ELSE 0 END) AS meaningful_plays,  -- 新增
       SUM(CASE WHEN ${MEANINGFUL_COND} THEN played_sec ELSE 0 END) AS meaningful_sec -- 新增
FROM play_logs WHERE ...
```

**性能影响**：SQLite 引擎在已过滤的结果集上计算 `SUM(CASE)`，不产生额外 I/O。单用户一周数百行数据，额外计算时间 < 0.1ms。

### 4.3 跳过率查询

```sql
SELECT COUNT(*) AS total,
       SUM(CASE WHEN ${SKIP_COND} THEN 1 ELSE 0 END) AS skipped
FROM play_logs WHERE ...
```

**跳过率 = skipped / total**。分母 `total` 仍然是全部已上报的播放，不做有效播放过滤。这样用户可以清楚地看到"在所有播放中，真正跳过（<25%进度）的占比"。

### 4.4 新歌发现查询（含有效播放过滤 + DISTINCT 去重）

```sql
SELECT COUNT(DISTINCT name||'__'||COALESCE(singer,'')) AS cnt
FROM play_logs WHERE user_id=? AND played_at>=? AND played_at<?
AND ${MEANINGFUL_COND}                                       -- 有效播放门槛
AND (name||'__'||COALESCE(singer,'')) NOT IN (               -- 历史首次出现
  SELECT name||'__'||COALESCE(singer,'') FROM play_logs WHERE user_id=? AND played_at < ?
)
```

**三处改进**：
1. `COUNT(DISTINCT)` 代替 `COUNT(*)`：修复重复计数 bug。
2. `MEANINGFUL_COND`：5~30 秒的浅尝不计入"发现"。
3. 有效播放过滤在 NOT IN 之前执行，减少了子查询对比的基数。

### 4.5 前端平均播放时长计算

```javascript
// src/report.js:80-83
const meaningfulPlays = data.period.meaningfulPlays || data.period.plays;  // fallback
const meaningfulSec = data.period.meaningfulSec || data.period.sec;
const avgSec = meaningfulPlays ? Math.round(meaningfulSec / meaningfulPlays) : 0;
```

**`||` fallback 策略**：当 API 返回旧版结构（不含 meaningfulPlays）时，自动退回到 `plays`。这确保了：
- 浏览器缓存中的旧 JS 可以正常渲染新版 API 的返回（新版 API 始终包含 `meaningfulPlays`）。
- 新版 JS 可以正常渲染旧版 API 的返回（旧版 API 不含 `meaningfulPlays`，回退到 `plays`）。

## 五、与旧方案的对比

### 5.1 指标计算口径对比

| 指标 | 旧方案计算公式 | 新方案计算公式 | 变化说明 |
|------|-------------|-------------|---------|
| **跳过率** | `played_sec < 30 / COUNT(*)` | `ratio < 0.25 / COUNT(*)` | 绝对→相对，短歌公平 |
| **新歌发现** | `COUNT(*) of 首次出现` | `COUNT(DISTINCT) of 有效播放 + 首次出现` | 修 bug + 加门槛 |
| **平均每首时长** | `SUM(played_sec) / COUNT(*)` | `SUM(meaningful_sec) / COUNT(meaningful)` | 分母只含有效播放 |
| **播放次数** | `COUNT(*)` 全部行 | `COUNT(*)` 全部行 | 不变 |
| **总时长** | `SUM(played_sec)` | `SUM(played_sec)` | 不变 |
| **完播率分布** | `played_sec/duration` 分三档 | 不变 | 不变（本身就是相对阈值） |
| **UI 标签** | "场均时长" | "平均每首播放时长" | 文案修正 |

### 5.2 场景验证

**场景 A：正常用户（不切歌）**

用户本周听了 20 首歌，每首都听完（≥80% 进度或 ≥30s）。

| 指标 | 旧方案 | 新方案 | 差异 |
|------|--------|--------|------|
| 跳过率 | 0% | 0% | 一致 |
| 新歌发现 | 5（如果 5 首是新歌） | 5 | 一致 |
| 平均播放时长 | 180s | 180s | 一致 |

**结论**：正常用户无感知变化。

**场景 B：快速浏览用户（每首 6 秒切 50 首新歌 + 听完 10 首）**

| 指标 | 旧方案 | 新方案 | 哪个更真实 |
|------|--------|--------|-----------|
| 播放次数 | 60 | 60 | —（都不变） |
| 跳过率 | 50/60=83% | 50/60=83% | 一致（6s/180s=3.3%<25%，都算跳过） |
| 新歌发现 | 50 | 0 | **新方案**：6s 不算发现 |
| 平均每首时长 | (50×6+10×180)/60=35s | (10×180)/10=180s | **新方案**：真实反映体验 |

### 5.3 架构对比

| 维度 | 旧方案 | 新方案 |
|------|--------|--------|
| **阈值定义** | 硬编码在各 SQL 中，3 处独立值 | 2 个 JS 常量，模板注入 |
| **修改阈值成本** | 改 3 处 SQL（可能遗漏） | 改 1 个 JS 常量 |
| **新增指标** | 无 | `meaningfulPlays` / `meaningfulSec`，可延伸使用 |
| **数据库层** | 纯 SQL 查询 | 纯 SQL 查询（无额外表/视图） |
| **向后兼容** | N/A | 新增字段，旧客户端 fallback |

## 六、技术约束与依赖

### 6.1 依赖条件

| 依赖 | 版本/说明 | 影响 |
|------|----------|------|
| `better-sqlite3` | 项目已有 | `db.prepare()` 参数化查询，不支持注入 SQL 片段 |
| SQLite 3.x | 项目已有 | `CAST(... AS REAL)` 浮点运算、`NULLIF` 空值处理 |
| `idx_play_logs_user` 索引 | `(user_id, played_at)` | 所有查询依赖该索引快速定位时间范围 |
| `duration` 字段非空 | Schema: `INTEGER DEFAULT 0` | 跳过率需要 `duration IS NULL OR duration = 0` 回退逻辑 |
| 前端 `elapsed` 计时器 | 每秒 +1 | 上报精度 1 秒 |

### 6.2 边界情况处理

| 边界情况 | 处理方式 |
|---------|---------|
| `duration = 0` 或 `NULL` | 跳过判断回退到绝对 30s 阈值；有效播放中 `duration > 0` 短路为 false，只依赖 30s 条件 |
| 全量无有效播放（新用户全跳） | `meaningfulPlays` 为 0 → `|| data.period.plays` fallback → 使用全部播放 |
| 旧版 API 返回（不含 meaningfulPlays） | 前端 `||` fallback → 退回到 `plays` |
| 旧版 JS 调用新版 API | 新版 API 始终返回 `meaningfulPlays`，旧 JS 忽略该字段 |
| 歌曲时长 > 1 小时（如 podcast） | `played_sec >= 30` 即可定性为有效播放，不受时长比例影响 |

### 6.3 未改动部分

以下部分**不受影响**，保持原有行为：
- `src/player.js` 的数据采集逻辑（3s+5s 门槛）
- `POST /stats/log` 路由
- `GET /stats/overview` 路由（仍返回原始数据）
- `GET /stats/top-songs` `/top-artists` `/daily` `/hourly` 路由
- `computePersona()` 的 persona 判断逻辑
- 海报生成的 `posterData` 数据

## 八、未来扩展方向

1. **session 化**：按 30 分钟无操作间隔切分会话，引入真正的"场均时长"（每场 session 的平均时长）。
2. **有效播放权重**：将 `meaningfulPlays` 应用到 Top 歌曲/歌手排序中，降低浅尝辄止的歌曲排名。
3. **阈值可视化调参**：在设置面板中让用户自行调节跳过率/有效播放的阈值。
4. **A/B 对比展示**：周报中同时展示"全部播放"和"有效播放"两个维度的均时，让用户直观理解过滤效果。
