# 音量控制统一

## 概述

当前存在三个独立的音量控制入口（WeMusic 底部滑块、B站播放器内部滑块、系统音量），用户每次前后台切换都感到音量不一致。目标：消除 app 内的音量控制，统一到系统音量。

**状态**：✅ 已由新架构从根本上解决（2026-07-13）

> **2026-07-13**：bgAudio 单一音频源架构改造（`docs/功能规格/播放架构改造-单一音频源.md`）从根本上解决了前后台音量不一致的问题——默认模式下 bgAudio 是唯一音频源，GainNode 统一控制音量，无需任何"前后台切换"动作。本文档的方案探索过程已归档为历史参考，实际采用方案见上述规格文档。
>
> 相关需求：「[音量归一化](./音量归一化.md)」— 自动标准化不同 B站源之间的响度（已实现，全场景生效）。

## 现状问题

### 三个音量控制

| 控制点 | 位置 | 作用范围 | 可控性 |
|--------|------|----------|--------|
| WeMusic 底部滑块 | 播放器底栏 `.volume-wrap` | 后台 bgAudio（GainNode） | ✅ 完全可控 |
| B站 iframe 音量 | B站播放器内部（cross-origin） | 前台视频音频 | ⚠️ 可 postMessage 设值，但用户能手动改 |
| 系统音量 | OS 级别 | 全局 | ✅ 用户手动 |

### 核心矛盾

两条独立的音频管线各有一套音量乘数，前后台切换时必然有落差。

```
前台：B站 CDN → iframe → [B站音量滑块] → [系统音量] → 耳朵
后台：B站 CDN → bgAudio → [GainNode(_bgVolume × safeGain)] → [系统音量] → 耳朵
```

归一化（`safeGain`）只通过 GainNode 对后台 bgAudio 生效，前台 iframe 不受影响。

## 方案探索

### 各方案概述

| 方案 | 一句话 |
|------|--------|
| **现在** | 底部滑块 + B站滑块 + 系统音量，三个独立控制，前后台切换音量必跳 |
| A | iframe 静音（muted=1），bgAudio 唯一音频源，前后台都走它 |
| B | 两端都设 100%，隐藏 WeMusic 按钮，系统音量接管 |
| C | 同 B，但保留按钮改为只读指示器（显示增益信息） |
| D | 保留现有 cross-fade 切换架构，两端设最大 |
| D+ | D + 前台 iframe 也跟随归一化增益 |
| E | iframe 静音 + 不销毁（隐藏替换销毁），bgAudio 唯一音源 |
| F | 自建 `<video>` 标签 + 服务端代理 B站视频流，彻底甩掉 iframe |

### 关键否决因素

**方案 A**（E 同理）：要求两个跨域不可控的播放器保持同步，但 B站 iframe 对我们是黑盒：
- 能发：`setVolume`（已知，在用）
- 能收：播放器就绪心跳（已知，在用）
- 不能发：seek / pause / resume（API 未公布，社区无文档）
- 不能收：currentTime / playback state（消息格式不明）
- 结构性矛盾 → 否决

**方案 F**：理论最优（一条管线、无跨域、原生 seek / 同步），但：
- 服务端视频流代理 → 带宽 10-50x 音频流
- N100/N150 家宽上传带宽不匹配 → 否决

**方案 B/C**：解决方式偏弱（仍然有前后台音量差），C 额外多了维护成本。

### 选定：方案 A（实际实施）
> **2026-07-13 更新**：最终实施的方案并非文档中标注的"D-lite"，而是在后期讨论中确定并实现了"bgAudio 单一音频源 + 展开/收起替代 cross-fade"的方案（接近原始方案 A/E，但不涉及跨域同步黑盒问题——因为视频画面展开时是完全独立的模式切换，不存在"两个播放器同时跑了需要保持同步"的场景）。详见 `docs/功能规格/播放架构改造-单一音频源.md`。
