# 人机协作深度实践

> 目标：建立不依赖人类记忆力和自觉性的自动化工作流。
> 原则：每次交互获得的知识 → 编码为 Agent 可自动执行的约束/流程/模板。
>
> 最后更新：2026-07-06（v5：平台抽象层优化 + 环境检查规则）

---

## 一、架构总览

三种机制分层协作：

- **Rules**：常驻 system prompt，提供持续约束（"不能做什么"）
- **Skills**：Agent 根据 description 匹配场景，自动加载并执行结构化流程（"怎么做"）
- **Templates**：静态产出物（需求文档、规则文件），承载持久化知识

## 二、已落地方案

### 2.1 交互协议

**文件**：`.codebuddy/rules/交互协议.mdc`

定义了 Agent 与人的 5 条行为规范：

1. **不确定时 → 问，不要猜**。列出 2-3 个选项让用户选。
2. **发现风险时 → 主动提醒**，附上风险等级和影响范围。
3. **需要决策时 → 列选项，用户选**。Agent 分析，用户决策。
4. **完成任务时 → 主动总结**：做了什么、改了什么文件、有无副作用。
5. **被纠正时 → 承认、记录、预防**。第一犯询问"要创建 Rule 吗？"，重复犯主动建议规则。

### 2.2 经验沉淀

**文件**：`.codebuddy/skills/experience-capture/SKILL.md`

6 步工作流：
1. 识别关键词 → 匹配分类决策表（代码规范/历史教训→Rules，个人偏好/项目背景→Memory）
2. **重复检查**（新增步骤）：创建前扫描已有 Rules，报告是否与已有规则重叠或已被覆盖 → 让用户决定合并还是新建
3. 向用户确认分类 + 建议操作
4. 执行写入
5. **追加决策上下文（按需）**：只有当规则正文未说清"为什么这样选"时才追加注释块。规则正文已解释清楚的（如 CSS 混用禁止的背景段），不需要画蛇添足
6. 关联更新 `docs/AI协作方法论.md` 或 `docs/人机协作深度实践.md`

### 2.3 会话复盘

**文件**：`.codebuddy/skills/session-retrospective/SKILL.md`

5 步工作流：
1. 扫描会话历史 → 识别 Agent/用户/配置三端的重复修改、猜测错误、规则违反
2. 分类评分（🔴高/🟡中/🟢低）
3. 提出可执行建议（创建 Rule、修改 Rule、优化沟通）
4. 输出结构化复盘报告
5. 逐一执行行动项

**自我进化**：复盘 Skill 自带迭代检查——每次复盘结束后检查是否发现了新的问题类型，更新本 Skill 的检查清单。

### 2.4 模式规则合并

**文件**：`.codebuddy/rules/前端开发规范.mdc`（已更新）

原来的「二、CSS 变量」和「三、字号」两条实例规则合并为「二、CSS 视觉常量（模式规则）」：

> 禁止硬编码任何视觉常量。覆盖颜色、字号、间距、圆角、阴影、字体等所有视觉属性。

合并后的自查句："如果调整 `--font-size` 或切换主题，这段代码还能正常工作吗？"

### 2.5 新建文件环境检查

**文件**：`.codebuddy/rules/新建文件环境检查.mdc`

创建任何 `.js` 文件前必查两项配置：
1. `package.json` 的 `"type"` 字段（ESM / CJS）
2. `vite.config.js` 的 `root`（前端能否跨目录引用）

触发场景：2026-07-06 引入平台抽象层时，Agent 先用 CJS 写了 3 个文件，后发现项目是 ESM 全部重写，浪费 6 次文件操作。

## 三、深度讨论

### 3.1 反馈闭环：路径 A（已部署） + 路径 B（找到可行方案）

**路径 A — 会话复盘 Skill**：✅ 已部署。每次做完整功能后说"复盘"，Agent 自我审查并产生改进建议。

**路径 B — 外部程序分析跨会话历史**：✅ 可行！

CodeBuddy 提供 HTTP API（需启动 server 模式，端口 8080）：

```
# 1. 获取所有会话
GET /api/v1/sessions?cwd=/path/to/workspace

# 2. 获取某会话的所有对话记录
GET /api/v1/traces?session_id=XXX&limit=100

# 3. 获取单条对话的完整详情（含提问、回复、工具调用）
GET /api/v1/traces/:traceId
```

实现思路（约 80 行 Node.js）：
1. 调用 sessions API 获取 WeMusic 项目的历史会话列表
2. 遍历每个 session，获取 traces 列表
3. 获取每个 trace 的详情，提取完整对话内容
4. 将所有对话打包发给 LLM，带上与复盘 Skill 相同的分析 prompt
5. 输出跨会话的改进报告

优势：覆盖路径 A 看不到的跨会话模式（"过去 10 次对话中，4 次在同类问题上犯错"）。脚本可以定期运行或手动触发。

此外，CodeBuddy IDE 也支持手动导出单条对话为 `.md` 文件（History 面板 → EXPORT），可作为补充手段。

### 3.2 规则生命周期

| 规则数量 | 状态 | 动作 |
|---------|------|------|
| 1-9 条（当前：6 条） | 健康 | 创建规则时顺手检查重复（已嵌入经验沉淀 Skill 步骤 2） |
| 10-19 条 | 关注 | 扫描已有规则，报告重叠或矛盾 |
| 20+ 条 | 需管理 | 创建 `docs/rules-review.md`，3 月未被触发的规则标记候选退役 |

当前 6 条规则，什么都不用做。经验沉淀 Skill 的步骤 2 已内置重复检查。

### 3.3 实例规则 → 模式规则

已完成合并（§2.4）。原则：

- **可归类为同一根因的问题** → 合并为模式规则（如"禁止硬编码视觉常量"替代"禁止硬编码颜色"+"禁止硬编码字号"）
- **根因不同的规则** → 保持独立（如"禁止硬编码视觉常量"和"禁止外链 URL 模板字符串"虽然都叫"禁止硬编码"，但一个是主题系统，一个是构建工具 bug，不同根因）

模式规则的优势：Agent 遇到 `margin: 10px` 时自动拦住，不需要你再创建"禁止硬编码间距"的规则。

### 3.4 交互协议：Agent 怎么服务你

详见 `.codebuddy/rules/交互协议.mdc`。核心理念是：

- **你拥有的 Rules 和 Skills 是在定义 Agent 怎么跟代码打交道**
- **交互协议是在定义 Agent 怎么跟你打交道**

类比：Rules/Skills = 公司规范（怎么干活），交互协议 = 你的管理风格（怎么汇报、怎么决策、犯错怎么处理）。
