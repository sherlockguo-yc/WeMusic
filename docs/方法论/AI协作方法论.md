# 用 AI Agent 做个人项目的经验教训

> 基于 WeMusic 项目与 AI 协作数十小时的总结  
> 2026 年 7 月（v6：合并人机协作深度实践，统一方法论）

---

## 核心问题：为什么反复修 bug？

最典型的模式：

```
你: 视频标题要左右滚动
AI: 写代码 → 部署
你: 没动啊
AI: 修 → 部署
你: 位置不对
AI: 修 → 部署   （× 6 轮，花了将近一小时）
```

**根因不是 AI 写得差，是缺了三个东西：**

| 缺少的 | 导致了 |
|--------|--------|
| 实现前的设计讨论 | 空标题不处理、溢出不管、多个滚动动画互抢 —— 全是没想到的场景 |
| 跨会话的规则约束 | 每次新对话 AI 重新摸索项目结构，犯同样的错误 |
| 自动化验证 | 修 A 炸 B，改了代码没人跑测试 |

## 解法一：设计讨论替代直接编码

每次提需求前，推动双方做 5 个维度的思考：

| 维度 | 问什么 |
|------|--------|
| 数据流 | 数据从哪来、流向哪、空值怎么办 |
| 状态机 | 涉及哪些状态、切换条件、会发生冲突吗 |
| 影响面 | 改这里之后，哪个模块的行为会变 |
| 失败模式 | 每种失败用户看到什么 |
| 取舍 | 更简单的方案是什么，为什么没选 |

核心不是「问够 5 个问题」，而是**逼着自己想清楚了再动手**。大部分 bug 的根因不是实现错了，是根本没想过那个场景存在。

**典型案例：播放器底栏布局的"打地鼠"修复**

一个反复出现超过两次的布局 bug：

```
问题：底栏视频标题覆盖到「词」按钮上
第一次"修复"：把 .player-status 从 absolute 改成 grid 列 ✓
回归：视频标题又覆盖到进度条上了 ✗

为什么？因为 .progress-center（进度条）仍然是 position:absolute; left:50% 独立居中。
grid 和 absolute 两套布局算法同时在一个容器里运作，修一个坑另一个坑必然冒出来。
```

第二次修复：把所有 6 个子元素都收编为同一 grid 的 6 条轨道 ✓，但出现了新问题——6 列左右不对称（左侧内容比右侧宽），导致进度条虽然列内居中，整体却向左边偏了 58px，未处于播放条的视觉中心 ✗。

第三次（最终）修复：改用 `1fr auto 1fr` 三列 grid（左组、中组、右组），中间 `auto` 列由 grid 布局算法天然放在视觉中心。实测 600~1440px 所有宽度下进度条中心偏差 ≤ 2px ✓。

**第四次回归**（2026-07-07）：三列 grid 重构时，`.player-status` 从 grid item 变成 flex item，但 `flex: 0 0 auto`（`flex-shrink: 0`）被无脑搬运过来。在新上下文中它阻止了容器缩小 → 溢出检测失效 → 视频标题不再左右滚动。修复：`flex: 0 0 auto` → `flex: 0 1 auto` ✓。

**第五次回归**（2026-07-07，同日）：浮窗视频标题也不滚动了。原因：`setVpTitle()` 用单帧 `requestAnimationFrame` 检测溢出，而视频浮窗从 `display:none→flex` 需要两帧完成布局。底栏的 `checkMarquee()` 一直用双帧所以正常，两处溢出检测函数用了不同的 rAF 策略 → 一处间歇性失效。修复：统一为双帧 rAF ✓。

如果第一次修的时候多问两句话，就能避免后续多次返工：

| 应该问的 | 跳过的问题 |
|----------|-----------|
| "这个容器内还有其他 absolute 定位的子元素吗？" | `.progress-center` 还在用 absolute |
| "改成 grid 之后，和 absolute 的元素边界会重叠吗？" | 两套算法算出来的位置会在某个宽度下冲突 |
| "Grid 列左右对称吗？中间列是否处于视觉中心？" | 6 列不对称，进度条偏左 |
| "布局重构后，CSS 属性的语义变了吗？`flex-shrink` 在 grid item 和 flex item 上的含义完全不同" | `flex: 0 0 auto` 阻止了溢出检测 → 文本不滚动 |

**正确的修复**：把**所有**横排子元素收编为对称的 3 列 grid（`1fr auto 1fr`），彻底移除 absolute，中间列由浏览器排版引擎保证居中，不依赖任何人工像素估算。

这个教训已形式化为 workspace rule「CSS 布局混用禁止」，每次 AI 会话都会自动加载。

**这需要写进规则文件才能长期生效。** 口头约定换一个对话就没了。

## 解法二：信息沉淀体系 —— 让知识跨会话存活

### 为什么需要分层存储

AI 的承诺只在当前会话有效。跨对话持久化不能只靠一种机制——不同性质的信息需要不同的存储方式。

每次交互中获得的知识（经验、规范、事实、工作流），按下图决策：

```
这个信息是否必须在每次会话中都生效？
  ├─ 是 → Rule（永远加载到上下文）
  └─ 否 → 继续判断

这个信息是否需要执行一个完整的多步骤流程？
  ├─ 是 → Skill（触发时才加载，不浪费日常上下文）
  └─ 否 → 继续判断

这个信息是否是一个孤立的事实/偏好/教训？
  ├─ 是 → Memory（简洁，按需触发）
  └─ 否 → Doc（存储在 docs/，按需读取）
```

再判断级别：

```
这个信息是否依赖当前项目的特定技术栈/工具？
  ├─ 是 → 项目级（放在 project/.codebuddy/ 下）
  └─ 否 → 用户级（放在 ~/.codebuddy/ 下，所有项目可用）
```

### 四选一对照表

| 目的地 | 适用场景 | 加载时机 | 典型例子 |
|--------|---------|---------|---------|
| **Rule** | 永远生效的编码约束 | 每次会话自动注入 | 用 Lucide 不用 emoji、CSS 用变量、设计讨论再写代码 |
| **Skill** | 特定场景的多步骤工作流 | 匹配描述时触发加载 | 拆分模块、会话复盘、经验沉淀 |
| **Memory** | 孤立事实/偏好/历史教训 | 语义匹配时触发 | 项目用 SQLite、某 bug 因忘记 import 导致 |
| **Doc** | 设计原理/架构说明/功能规划 | 手动或代码引用时读取 | 架构概览、候选选择逻辑、功能规划 |

### 三层架构

三种机制分层协作：

- **Rules**：常驻 system prompt，提供持续约束（"不能做什么"）
- **Skills**：Agent 根据 description 匹配场景，自动加载并执行结构化流程（"怎么做"）
- **Doc / Templates**：静态产出物，承载持久化知识和设计意图

### 严禁混放

- ❌ 把长文档内容塞进 Rule（污染上下文）
- ❌ 把编码约束只记在 Memory 里（不稳定加载）
- ❌ 把多步骤流程写成 Memory（无法承载复杂度）
- ❌ 把项目特定信息放在用户级（泄漏到其他项目）

核心原则：**具体性递增，通用性递减**——越靠近项目的越具体，越靠近用户的越通用。能 User 的不要 Project（提高复用），能 Memory 的不要 Rule（节省上下文）。

### 交互协议：定义 Agent 怎么跟你打交道

Rules 和 Skills 是在定义 Agent 怎么跟代码打交道，**交互协议**是在定义 Agent 怎么跟你打交道。

类比：Rules/Skills = 公司规范（怎么干活），交互协议 = 你的管理风格（怎么汇报、怎么决策、犯错怎么处理）。

交互协议的核心条款（已写入 `~/.codebuddy/rules/AI协作规范.mdc`）：

1. **不确定时 → 问，不要猜**。列出 2-3 个选项让用户选。
2. **发现风险时 → 主动提醒**，附上风险等级和影响范围。
3. **完成任务时 → 主动总结**：做了什么、改了什么文件、有无副作用。
4. **被纠正时 → 承认、记录、预防**。第一犯询问"要创建 Rule 吗？"，重复犯主动建议规则。
5. **批量修改前 → 先列清单**，确认后一次性执行。
6. **宣告不可行前 → 至少尝试 2 种方案**。

### 实例规则 → 模式规则

规则体系会自然膨胀。维护原则：

- **可归类为同一根因的问题** → 合并为模式规则（如"禁止硬编码视觉常量"替代"禁止硬编码颜色"+"禁止硬编码字号"）
- **根因不同的规则** → 保持独立（如"禁止硬编码视觉常量"和"禁止外链 URL 模板字符串"虽然都叫"禁止硬编码"，但一个是主题系统，一个是构建工具 bug，不同根因）

模式规则的优势：Agent 遇到 `margin: 10px` 时自动拦住，不需要你再创建"禁止硬编码间距"的规则。

### 规则生命周期

| 规则数量 | 状态 | 动作 |
|---------|------|------|
| 1-9 条 | 健康 | 创建规则时顺手检查重复 |
| 10-19 条 | 关注 | 扫描已有规则，报告重叠或矛盾 |
| 20+ 条 | 需管理 | 创建 `docs/rules-review.md`，3 月未被触发的规则标记候选退役 |

### 教训：新建文件前检查环境

创建任何 `.js` 文件前必查两项配置，避免模块系统不匹配导致全部重写：

1. `package.json` 的 `"type"` 字段（ESM / CJS）
2. `vite.config.js` 的 `root`（前端能否跨目录引用）

触发场景：引入平台抽象层时，Agent 先用 CJS 写了 3 个文件，后发现项目是 ESM 全部重写，浪费 6 次文件操作。花 20 秒读两个配置文件，能省 10 分钟返工。

## 解法三：用工具替代文档

手写的架构文档会过期。改代码忘了改文档 → 文档误导 AI → AI 产生幻觉。

可以用一个小型 CodeGraph 脚本解决：用 AST 解析所有源文件，提取模块、导出、路由，生成 JSON。每次 build 自动重建，永不过期。

```js
// scripts/codegraph.js
// parse all .js files → extract exports/imports/routes → write .project-graph.json
```

```
过去: AI grep 30 次文件 → 高 Token、易遗漏
现在: AI 读 project-graph.json → 一次获得全局视图（26 模块、63 路由）
```

文档只保留**设计意图**（为什么这么做、状态机流转），结构信息全部交给代码生成。

## 解法四：测试嵌入流程而不是靠人记

三级检查，每级在「最自然的时机」自动跑：

| 层级 | 工具 | 触发时机 | 测什么 |
|------|------|----------|--------|
| 结构 | AST 依赖检查 | `npm test` / git push | 循环依赖、违规导入 |
| 逻辑 | vitest | `npm test` / git push | API 返回值、业务逻辑 |
| 视觉 | playwright E2E | `npm run deploy` | 页面渲染、按钮行为 |

**关键设计：不需要人记得跑。** 结构+逻辑挂在 `npm test` 上（CI 自动触发），视觉挂在 `npm run deploy` 上（你部署就自动跑）。零负担。

## 反馈闭环

写完代码不是终点。需要机制让每次协作的教训反哺到体系中。

### 路径 A：会话复盘（已部署）

每次做完一个功能后触发复盘 Skill，Agent 自我审查当前会话：
- 扫描对话历史，识别 Agent/用户/配置三端的重复修改、猜测错误、规则违反
- 分类评分，提出可执行建议（创建/修改 Rule、优化沟通）
- 复盘 Skill 自带迭代检查——发现新的问题类型后更新自身的检查清单

### 路径 B：跨会话批量分析（待实现）

CodeBuddy 提供 HTTP API（需启动 server 模式，端口 8080），可以编写外部脚本批量分析历史会话：

```
GET /api/v1/sessions?cwd=/path/to/workspace      # 获取所有会话
GET /api/v1/traces?session_id=XXX&limit=100       # 获取会话对话记录
GET /api/v1/traces/:traceId                        # 获取单条对话详情
```

实现思路（约 80 行 Node.js）：遍历历史会话 → 提取完整对话 → 打包发给 LLM 分析 → 输出跨会话改进报告。

优势：覆盖路径 A 看不到的跨会话模式（"过去 10 次对话中，4 次在同类问题上犯错"）。

## 局限和未解决的问题

1. **项目规模上限**：26 模块的文件级 CodeGraph 够用。项目成长到 500+ 模块时，需要正式的代码知识图谱工具
2. **AI 重构能力**：跨多文件的大重构（如拆分 900 行模块）AI 成功率不高——需要同时理解前后端契约、构建配置、调用链，信息密度太大
3. **测试覆盖策略**：E2E 只测了「核心路径没崩溃」，数据正确性目前还是靠人看
4. **规则膨胀**：规则文件多了之后，AI 可能「读了但没执行」——如何确保 AI 真正遵循规则，还没有好办法
