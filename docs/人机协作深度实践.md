# 人机协作深度实践

> 本文档探讨三个核心问题：需求表达、规范沉淀、架构决策沉淀。
> 目标是建立**不依赖人的记忆力**、**不依赖人的自觉性**的工作流。
>
> 最后更新：2026-07-05

---

## 0. 核心原则

> 把每次 Agent 交互中获得的知识，编码成**可以被 Agent 自动执行**的约束/模板/案例。
> 时间投入产生复利——规范越来越完善，Agent 产出质量越来越高，返工越来越少。

**设计哲学**：从「人记住了要做 X」变成「系统自动确保 X 被做」。每一行规则，都是用一个犯错的代价换来的自动化。

---

## 1. 需求精确表达

### 问题诊断

当前痛点：对 Agent 说一句"帮我做视频标题滚动"，Agent 就开始写代码，然后反复修 bug。根因不是实现能力差，是**人没说清楚**，Agent 也没拒绝。

### 为什么 Skill 不够

一个「需求编写 Skill」可以做两件事：
- 让 Agent 问结构化问题（好，但依赖于人主动调用）
- 生成需求文档模板（好，但模板只是格式，填什么内容才是核心）

Skill 解决不了的核心矛盾：**人会跳过思考环节直接让 Agent 写代码**。
Skill 是建议性的（"你应该先讨论设计"），不是强制性的。

### 方案：需求预检清单（人用）+ 规则门禁（Agent 强制）

**Layer 1 — 需求模板**（人在和 Agent 对话前填写）

文件位置：`docs/requirements/[feature-name].md`

```markdown
# [功能名称]

## 用户场景
- 谁、在什么情况下、想要什么结果？

## 预期行为
- 正常路径：用户操作 → 系统响应（一步步写）
- 空状态：没有数据时显示什么？
- 加载态：等待数据时显示什么？
- 错误态：网络失败/数据异常时显示什么？

## 边界情况
- 极长/极短的输入
- 并发操作（快速点击两次）
- 与其他功能的交互（会不会影响播放/搜索/队列？）

## 影响范围
- 涉及哪些文件/模块？（看 .project-graph.json）
- 可能影响哪些已有功能？

## 验收标准
- 怎么判断"做完了"？（可观测的行为，不是代码层面的）

## 备选方案
- 有没有更简单的实现？为什么没选？
```

**Layer 2 — Agent 门禁规则**

写入 `.codebuddy/rules/`，确保每次会话自动加载：

```markdown
## 需求门禁规则

收到任何功能需求时：
1. 检查是否存在 `docs/requirements/[feature-name].md`
2. 如果不存在 → 拒绝编码，引导用户填写需求模板
3. 如果存在但关键字段为空 → 指出缺失项，拒绝编码
4. 如果存在且完整 → 基于需求文档做设计讨论，然后编码

禁止：收到一句话需求后直接编码。
```

**为什么这是更好的方案？**

| 维度 | Skill 方案 | 模板 + 门禁方案 |
|------|-----------|----------------|
| 执行保障 | 人需要记得调用 | **Agent 自动检查、自动拒绝** |
| 产出物 | 会话结束即消失 | 需求文档持久化，可追溯 |
| 学习成本 | 每次都要想"该用哪个 skill" | 一次配置，永久生效 |
| 改进方式 | 改 skill 内容 | 改模板 + 改规则，两者独立迭代 |

### 实施步骤

1. 创建 `docs/requirements/` 目录和模板文件
2. 在 `.codebuddy/rules/` 中添加门禁规则
3. 下次提需求时：先用模板写 5 分钟 → 把文件给 Agent → Agent 自动走设计讨论

## 2. 项目规范的沉淀与执行

### 问题诊断

当前痛点：
- 人发现问题 → 人告诉 Agent → Agent 可能把规则放到 Memory（不稳定加载）而非 Rules（稳定加载）
- 人需要记住"看到问题 → 说出来 → 确认放对位置"这一整条链路
- 任何一个环节断裂，规范就丢失了

### 为什么 Skill 不够

一个「规范沉淀 Skill」可以引导 Agent 怎么做，但它解决不了"Agent 分不清该放 Rules 还是 Memory"这个判断问题——因为这个判断需要理解**信息类型**，而 Skill 只提供**操作流程**。

### 方案：分类决策规则（一次配置）+ 确认机制（每次触发）

**核心思路**：不是让 Agent 更聪明，而是给它一个**分类决策表**，并强制它在每次"记住"操作时执行确认流程。

**Layer 1 — 分类规则**

写入 `.codebuddy/rules/沉淀分类规范.mdc`：

```markdown
## 沉淀分类规范

当用户表达「记住 / 以后都这样 / 这是规则」意图时，按以下决策树分类：

### 决策表

| 信息类型 | 存放位置 | 示例 |
|---------|---------|------|
| 代码规范/技术约束 | .codebuddy/rules/ | CSS 变量禁止硬编码颜色 |
| 历史 bug 教训 | .codebuddy/rules/ | Grid 和 absolute 禁止混用 |
| 项目约定/命名规范 | .codebuddy/rules/ | 歌曲列表统一用 .song-row |
| 用户偏好/个人习惯 | Memory | "我喜欢用 pnpm 而不是 npm" |
| 项目背景/事实信息 | Memory | "这个项目部署在自建服务器上" |
| 临时备忘 | 不存储，直接回复确认 | "今天下午要改端口号" |

### 执行流程

1. 用户说完 → Agent 分类 → Agent 口头确认：
   "这属于 [类型]，应该存为 [Rules/Memory]，文件名建议：[xxx.mdc]，确认吗？"
2. 用户确认 → 执行存储
3. 存储后 → 检查是否需要同步更新 docs/AI协作方法论.md
```

**Layer 2 — 为什么这个方案更好**

| 旧流程（无规则） | 新流程（有分类规则） |
|---|---|
| 人发现 bug → 人告诉 Agent → Agent 猜测放哪 → 可能放错 | 人发现 bug → 人告诉 Agent → Agent 对表分类 → 显式确认 → 放对位置 |
| 依赖 Agent 的理解能力 | 依赖**预先定义的分类标准** |
| 每次交互都要博弈 | 一次定义，永久使用 |

### 一个可选增强：触发器关键词

在分类规则中增加触发词映射，减少 Agent 的歧义判断：

| 人说 | Agent 理解 |
|------|-----------|
| "记住这个坑" | → 历史教训 → Rules |
| "以后都这么写" | → 代码规范 → Rules |
| "我习惯用 XX" | → 用户偏好 → Memory |
| "这个项目用的是 XX" | → 项目背景 → Memory |

## 3. 架构决策的低成本沉淀

### 问题诊断

当前痛点：重要的架构决策（如「播放器底栏为什么用纯 Grid」）只存在于当事人的脑子里，时间久了就忘了。后续维护时不知道当初为什么这么设计，要么不敢改，要么改了出问题。

### 为什么「深化习惯」是虚的

"养成记录决策的习惯"这个建议的问题是：它把 100% 的执行负担放在人身上。人一忙、一懒、一忘，决策就丢了。需求的不是一个好习惯，而是一个**不依赖习惯的系统**。

### 方案：决策即规则注释（零额外操作）

**关键洞察**：你的架构决策大部分**已经以规则的形式存在了**。

例如：
- 「CSS 布局混用禁止」这条规则的背后，是「播放器底栏统一用纯 Grid」这个架构决策
- 「外链 URL 禁止模板字符串拼接」这条规则的背后，是「Vite 构建工具会误压缩协议相对 URL」这个架构决策

你不需要一个「架构决策日志」来记录这些——规则本身就是决策的**编码形式**。唯一缺失的是：**为什么**。

**方案：为每条规则追加决策上下文**

在规则文件末尾增加一个「决策上下文」区块：

```markdown
<!-- 
┌─ 决策上下文 ──────────────────────────────┐
│ 日期: 2026-07-03                          │
│ 触发: 播放器底栏布局 bug 反复出现 2+ 次     │
│ 方案A: Grid + absolute 混用，逐像素调整     │
│ 方案B: 全部收编为 Grid 轨道                │
│ 选择: B                                    │
│ 理由: A 方案两套算法同时运作，修一个坑必然   │
│       冒另一个。B 方案从数学上消除重叠可能。  │
│ 重评条件: 未来出现无法用 Grid 表达的复杂    │
│           布局需求时重新评估                │
└──────────────────────────────────────────┘
-->
```

**为什么这是低成本的**：

1. **创建规则时顺手写**——Agent 在创建规则的同时，填充决策上下文。不需要额外打开文件。
2. **不需要单独维护**——决策上下文附着在规则上，规则更新时决策自然同步更新。
3. **阅读规则时自然看到**——不需要记得"有个决策文档"，读到规则就看到为什么。

**实施：加入规则创建规范**

在 `.codebuddy/rules/沉淀分类规范.mdc` 中增加：

```markdown
### 规则文件格式要求

每条规则文件必须包含：
1. 规则内容（Agent 需要遵守的约束）
2. 决策上下文（HTML 注释形式，包含：日期、触发原因、备选方案、选择理由、重新评估条件）

创建规则时，Agent 在写入规则内容后，同步写入决策上下文。
```

### 对于不体现为规则的决策

有些决策不是约束性的（如"为什么选择 QQ 音乐 API 而不是网易云 API"），这类决策建议用一个**轻量的决策日志**：

`docs/decisions/README.md`：

```markdown
# 架构决策日志

| 日期 | 决策 | 选择 | 理由 |
|------|------|------|------|
| 2026-06 | 音乐 API 选择 | QQ 音乐 | 曲库更全，API 稳定性更好 |
| 2026-06 | 前端框架选择 | 原生 JS + Vite | 项目规模不需要 React，原生更轻量 |
```

这张表**不需要主动维护**。当 Agent 被问及"为什么当初选 QQ 音乐"时，Agent 能从代码、规则、对话历史中推断。表格只是索引。

## 4. 总结：三个问题的统一框架

| 问题 | 旧方案 | 新方案 | 核心转变 |
|------|--------|--------|----------|
| 需求不清晰 | 人自己想、Agent 猜 | 模板 + 门禁规则 | 从「人记得说清楚」→「系统强制你说清楚」 |
| 规范沉淀错位 | 人主动提、Agent 猜放哪 | 分类规则 + 确认机制 | 从「Agent 猜」→「Agent 对表，人确认」 |
| 决策记录丢失 | 人靠记忆、事后补 | 规则注释 + 决策上下文 | 从「额外维护」→「附着在已有产出物上」 |

**共同原则**：让 Agent 成为工作流的**执行者和执行者**，而不是一个需要人不断提醒和纠正的**协作者**。每一轮交互中发现的规律，立即编码成规则——下一次交互 Agent 就会自动遵守。
