# 用 AI Agent 做个人项目的经验教训

> 基于 WeMusic 项目与 AI 协作数十小时的总结  
> 2026 年 7 月

---

## 核心问题：为什么反复修 bug？

最典型的模式：

```
你: 视频标题要左右滚动
AI: 写代码 → 部署
你: 没动啊
AI: 修 → 部署
你: 位置不对
AI: 修 → 部署   （× 6 轮，花了将近一小时）
```

**根因不是 AI 写得差，是缺了三个东西：**

| 缺少的 | 导致了 |
|--------|--------|
| 实现前的设计讨论 | 空标题不处理、溢出不管、多个滚动动画互抢 —— 全是没想到的场景 |
| 跨会话的规则约束 | 每次新对话 AI 重新摸索项目结构，犯同样的错误 |
| 自动化验证 | 修 A 炸 B，改了代码没人跑测试 |

---

## 解法一：设计讨论替代直接编码

每次提需求前，推动双方做 5 个维度的思考：

| 维度 | 问什么 |
|------|--------|
| 数据流 | 数据从哪来、流向哪、空值怎么办 |
| 状态机 | 涉及哪些状态、切换条件、会发生冲突吗 |
| 影响面 | 改这里之后，哪个模块的行为会变 |
| 失败模式 | 每种失败用户看到什么 |
| 取舍 | 更简单的方案是什么，为什么没选 |

核心不是「问够 5 个问题」，而是**逼着自己想清楚了再动手**。大部分 bug 的根因不是实现错了，是根本没想过那个场景存在。

**这需要写进规则文件才能长期生效。** 口头约定换一个对话就没了。

---

## 解法二：规则文件固化约束

AI 的承诺只在当前会话有效。跨对话持久化有两个机制：

- **Memory**（记忆）：知识库存储，但加载不稳定，依赖 AI 判断是否相关
- **Workspace rules**（`.codebuddy/rules/`）：稳定可靠，每次新会话自动注入系统提示

**需要长期生效的约束，必须放进 rules。** 不要依赖 Memory 或口头承诺。

示例 —— `.codebuddy/rules/AI协作规范.mdc` 要求 AI 在写代码前先做设计讨论：

> "收到任何功能需求时，在写代码之前必须完成设计讨论（数据流、状态机、影响面、失败模式、取舍）和方案对比。禁止直接跳到编码。"

这句话放进 rules 之后，换多少个对话都会生效。

## 解法三：用工具替代文档

手写的架构文档会过期。改代码忘了改文档 → 文档误导 AI → AI 产生幻觉。

可以用一个小型 CodeGraph 脚本解决：用 AST 解析所有源文件，提取模块、导出、路由，生成 JSON。每次 build 自动重建，永不过期。

```js
// scripts/codegraph.js
// parse all .js files → extract exports/imports/routes → write .codebuddy/project-graph.json
```

```
过去: AI grep 30 次文件 → 高 Token、易遗漏
现在: AI 读 project-graph.json → 一次获得全局视图（26 模块、63 路由）
```

文档只保留**设计意图**（为什么这么做、状态机流转），结构信息全部交给代码生成。

## 解法四：测试嵌入流程而不是靠人记

三级检查，每级在「最自然的时机」自动跑：

| 层级 | 工具 | 触发时机 | 测什么 |
|------|------|----------|--------|
| 结构 | AST 依赖检查 | `npm test` / git push | 循环依赖、违规导入 |
| 逻辑 | vitest | `npm test` / git push | API 返回值、业务逻辑 |
| 视觉 | playwright E2E | `npm run deploy` | 页面渲染、按钮行为 |

**关键设计：不需要人记得跑。** 结构+逻辑挂在 `npm test` 上（CI 自动触发），视觉挂在 `npm run deploy` 上（你部署就自动跑）。零负担。

---

## 检查清单

| 做了什么 | 怎么做到的 |
|----------|-----------|
| 避免修 bug 循环 | 实现前做设计讨论（5 维度） |
| 跨对话约束持久化 | 写入 `.codebuddy/rules/`（非 Memory） |
| 文档不过期 | CodeGraph 每次 build 自动生成结构图 |
| 自动化验证 | 结构/逻辑/视觉三级，嵌入 `test` 和 `deploy` |

---

## 局限和未解决的问题

1. **项目规模上限**：26 模块的文件级 CodeGraph 够用。项目成长到 500+ 模块时，需要正式的代码知识图谱工具
2. **AI 重构能力**：跨多文件的大重构（如拆分 900 行模块）AI 成功率不高——需要同时理解前后端契约、构建配置、调用链，信息密度太大
3. **测试覆盖策略**：E2E 只测了「核心路径没崩溃」，数据正确性目前还是靠人看
4. **规则膨胀**：规则文件多了之后，AI 可能「读了但没执行」——如何确保 AI 真正遵循规则，还没有好办法
