# 基于完播的源排序优化

> 状态：已实现 | 日期：2026-07-06

## 背景

歌词源和视频源目前完全依赖程序化评分（歌名匹配、歌手匹配、播放量等），缺乏来自真实用户的信号。当一个源被大量用户完整播放后，这本身就是"高质量"的有力证据。

## 设计

### 完播定义

`played_sec >= duration * 90%`（播放时长 ≥ 歌曲时长的 90%）

### 统计粒度

`(source_type, source_id, song_key)` 三元组唯一标识一个统计记录。

- `source_type`：`video` | `lyrics`
- `source_id`：B 站 bvid / 网易云 songId / QQ音乐 `qq:mid`
- `song_key`：`name__singer`

按歌曲区分而非全局统计，因为同一视频对歌 A 完美不等于对歌 B 完美。

### 权重公式

对数衰减，防止少数热门源垄断：

```
视频：bonus = round(log2(completions + 1) × 5 × 100) / 100    (weight=5)
歌词：bonus = round(log2(completions + 1) × 0.5 × 100) / 100  (weight=0.5)
```

| 完播次数 | 视频加成 | 歌词加成 |
|----------|---------|---------|
| 0 | 0 | 0 |
| 1 | +5 | +0.5 |
| 3 | +10 | +1.0 |
| 7 | +15 | +1.5 |
| 15 | +20 | +2.0 |

## 数据流

```
用户播放歌曲 → 播放到 ≥90% 时长
    ↓
/src/player.js → _flushLog() 上报 played_sec + bvid + lyrics_source_id
    ↓
server/routes/stats.js → POST /stats/log
    ↓ (完播判断: played_sec >= duration * 0.9)
插入 play_logs + UPSERT source_completions (completions + 1)
    ↓
下次匹配时:
server/routes/play.js → rank() 查询 getCrowdCompletions('video', songKey) → 加分
server/services/lyrics.js → searchLyricsCandidates() 查询 getCrowdCompletions('lyrics', songKey) → 加分
```

## 冷启动

- 新源完播次数为 0 → 加成 0 → 不影响原始评分
- 累积 ≥ 1 次完播后开始生效
- 歌词源需要 ≥ 2 次完播才有 > 0 的加成（因 `log2(2+1) × 0.5 = 0.79`）
