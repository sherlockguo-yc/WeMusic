# 播放进度与音量控制说明

> 面向非技术背景的说明文档，解释 WeMusic 播放器"进度条"和"音量"分别由谁控制、
> 谁的数值是真实的、谁的数值是模拟/推算的。
> 涉及的代码全部在 `src/player.js`（前端）+ `server/routes/play.js`（后端音量分析）。
> 背景播放的状态流转细节见 `docs/specs/后台播放.md`，本文档聚焦"进度"和"音量"这两块。

---

## 〇、先认清页面里有几个"播放器"

页面上跟"播放"相关的东西，其实是 **3 个独立的角色**，很多误解都源于把它们当成了一个东西：

| 角色 | 是什么 | 你能不能看见 | 谁在用它 |
|------|--------|------------|---------|
| **① 底部播放器**（本文档统一这样称呼） | 页面最下方那一条：封面、歌名、进度条（`seekBar`）、音量滑块（`volBar`）、上一首/下一首/暂停按钮 | 能看见，是 WeMusic 自己画的 UI | 你手动操作、看进度、看音量的地方 |
| **② iframe 视频播放器** | B 站官方提供的嵌入式播放器（一个 `<iframe>`），前台（你正在看这个页面时）显示画面 | 能看见（画面） | 前台真正在放"声音+画面"的东西 |
| **③ 隐藏音频播放器**（代码里叫 `bgAudio`） | 一个没有画面、你完全看不到的 `<audio>` 标签，专门用来在你切到后台（切标签页/锁屏）时接管播放 | **看不见**，纯后台工作 | 后台（你不在看页面时）真正在放声音的东西 |

**核心原则：①"底部播放器"永远只是一个"遥控器 + 显示屏"，它自己不产生声音，也不是真实进度的来源。真正在播放声音的，前台是②，后台是③。**

下面进度和音量都按这 3 个角色分别说清楚。

---

## 一、播放进度条——主语是「①底部播放器」的进度条，不是②iframe 的进度

### 结论先说

**底部播放器上看到的进度条和时间（比如 "1:23 / 3:45"），是 WeMusic 自己用一个"每秒走一格"的计时器数出来的，不是从 ②iframe 里读出来的真实播放位置。**

### 为什么不能用 iframe 的真实进度？

②iframe 是 B 站的内容，属于"别人家的网页嵌在你的网页里"。浏览器出于安全考虑，**不允许你的网页读取 iframe 里面到底播到第几秒**。所以 WeMusic 没办法、也从来没有尝试去读取 iframe 内部的真实播放进度。

（②iframe 内部其实自己也有一条进度条/播放头，是 B 站播放器自己画的，跟①底部播放器的进度条是两条完全不同的、互不感知的进度条。）

### ①底部播放器的进度条到底是怎么走的？

1. 每首歌开始播放时，启动一个"每秒钟走一格"的计时器（不管 ②iframe 或 ③隐藏音频播放器到底播到哪，这个计时器自己按秒数往上加）。
2. ①底部播放器的进度条、当前时间文字、封面周围的进度圆环，全部由这个计时器的数字驱动更新。
3. **因此进度条不能拖动**——因为拖动进度条意味着要让 ②iframe 跳到指定时间点，但 WeMusic 没有能力命令 iframe 这么做，所以这个功能被禁用了（进度条是"只读展示"）。

### 计时器会有误差，所以有 3 次校正动作（都是"校正①的显示时间"，不是"读取②的真实进度"）：

| 校正时机 | 校正依据 | 说明 |
|---------|---------|------|
| 每首歌刚开始播放时 | ②iframe 发出的一个"我已经初始化好了"的信号 | ②iframe 从创建到真正出声中间有延迟（零点几秒到几秒），信号到达时用"信号到达时刻"倒推出这段延迟，把①的计时器往回拨一点，让显示更准 |
| 切到后台期间，每 4 秒一次 | ③隐藏音频播放器的**真实播放位置**（这是唯一有真实进度的角色） | 后台期间①的计时器可能因为浏览器省电策略走得不准，用③的真实位置纠正① |
| 切回前台的瞬间 | 同上，用③隐藏音频播放器的真实播放位置 | 切回前台前，先把①的显示时间对齐一次，再重新创建 ②iframe 并告诉它"从这个时间点开始播" |

### 什么时候判定"这首歌放完了，自动切下一首"？

- **前台时**：①的计时器数到了歌曲总时长，就认为放完了。
- **后台时**：以 ③隐藏音频播放器**真正发出"我放完了"的信号**为准（这是真实事件，比计时器准）。

## 二、音量——分成两个完全独立的音量，主语必须分清楚

**这是最容易搞混的一块：①底部播放器上只有一个音量滑块，但它背后同时控制着两个互相独立、原理完全不同的"音量"。**

### 表格先看结论

| | 音量滑块表面上显示的 | 前台实际听到的音量 | 后台实际听到的音量 |
|---|---|---|---|
| 主语 | ①底部播放器的音量滑块（唯一的手动调节入口） | **②iframe 视频播放器自己的音量** | **③隐藏音频播放器的音量（通过一个叫 GainNode 的音量放大器控制）** |
| 你拖动滑块时，谁在变 | 滑块位置、显示的百分比 | 滑块变化后，WeMusic 发一条"指令"告诉 ②iframe："帮我把你自己的音量调到 XX%" | 滑块变化后，WeMusic 直接调节 GainNode 这个音量放大器的倍数 |
| 是否受"自动音量拉平"影响 | 不影响滑块显示 | **不受影响**（iframe 只听你手动调的音量） | **受影响**（会在你手动调的音量基础上再乘一个自动倍数） |

**换句话说：①底部播放器的音量滑块，是"一个开关，控制两个不同的音量系统"——前台时实际发挥作用的是②iframe 自己的音量，后台时实际发挥作用的是③隐藏音频播放器背后的音量放大器（GainNode）。这两个音量系统互不相通，只是被同一个滑块同步遥控。**

### 为什么前台不能像后台一样用"音量放大器"直接控制？

因为②iframe 是 B 站自己的播放器，它内部的音量是 B 站播放器自己管理的，WeMusic 的网页没有权限直接伸手进去改它的音量数值，**只能像发短信一样，发一条"请把音量调到 XX%"的消息**，B 站播放器收到后自己去调整。所以前台的音量精确度、生效速度完全取决于 B 站播放器本身。

### 什么是"自动音量拉平"（只作用于③后台隐藏音频播放器）？

不同视频原始录音的响度差异很大，同样调到 50% 音量，有的歌听起来很轻，有的很响。WeMusic 加了一个自动拉平功能，**但这个功能从设计上只能作用于③（后台隐藏音频播放器），无法作用于②（前台 iframe）**，原因同上——WeMusic 管不到 iframe 内部。

工作方式：
1. 后端第一次给你播放某个视频的音频时，会在服务器后台异步下载一份完整音频，用专业工具分析这段音频"实际有多响"（这个分析不影响你正常听歌，做完一次后存起来，下次同一个视频不用再分析）。
2. 根据分析结果算出一个"倍数"：本来就轻的视频算出的倍数 > 1（放大），本来就响的视频算出的倍数 < 1（压低）。
3. 你后台听这个视频时，这个倍数会叠加在你手动调的音量上，一起作用在③的音量放大器（GainNode）上。
4. 为防止某些视频原始录音特别轻、算出的倍数过大导致炸耳朵，设置了上限（最多放大 8 倍）。
5. 这个自动拉平功能可以在设置里手动关闭，关闭后③只按你手动调的音量播放，不叠加任何自动倍数。

### 一句话记住这部分

> **音量滑块只有一个，但它是"一个方向盘，接了两套完全不同的转向系统"：前台转的是②iframe 自己的方向盘（只能远程喊话），后台转的是③隐藏音频播放器背后的独立音量放大器（可以直接控制，还会自动叠加音量拉平倍数）。**

## 三、常见疑问快问快答

**Q: 我拖动①底部播放器的进度条，能跳到指定时间点吗？**
A: 不能，进度条被禁用（`disabled`），只能看，不能拖。原因见第一节。

**Q: 前台看视频时调音量，是不是也会被"自动拉平"影响？**
A: 不会。自动拉平只作用于③（后台隐藏音频播放器），前台②iframe 的音量完全只受你手动调的滑块位置影响。

**Q: 后台听歌时，音量滑块上显示的百分比，是不是就是实际听到的音量？**
A: 不完全是。实际听到的音量 = 你滑块调的百分比 × 自动拉平倍数（如果这个功能开着的话）。滑块显示的只是"你手动设的基准值"。

**Q: 底部播放器进度条上的时间，是不是就是这首歌真正播放到的时间？**
A: 大多数时候很接近，但不是"实时读取"来的，是计时器数出来的、再定期用后台真实播放位置纠正的结果，可能有 1~2 秒的短暂误差窗口。
