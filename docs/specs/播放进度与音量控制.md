# 播放进度与音量控制说明

> 面向非技术背景的说明文档，解释 WeMusic 播放器"进度条"和"音量"分别由谁控制、
> 谁的数值是真实的、谁的数值是模拟/推算的。
> 涉及的代码全部在 `src/player.js`（前端）+ `server/routes/play.js`（后端音量分析/直链解析）。
> 架构细节见 `docs/specs/后台播放.md`，本文档聚焦"进度"和"音量"这两块。
>
> **2026-07-13 架构重写**：原「前台 iframe / 后台 bgAudio 双引擎切换」已废弃，
> 改为「bgAudio 单一音频源 + 展开视频才切 iframe」架构。本文档已按新架构重写。

## 〇、先认清页面里有几个"播放器"

页面上跟"播放"相关的东西，其实是 **2 个独立的角色**（旧版本有 3 个，现在合并为 2 个）：

| 角色 | 是什么 | 你能不能看见 | 什么时候在用它 |
|------|--------|------------|---------|
| **① 底部播放器**（本文档统一这样称呼） | 页面最下方那一条：封面、歌名、进度条（`seekBar`）、音量滑块（`volBar`）、上一首/下一首/暂停按钮 | 能看见，是 WeMusic 自己画的 UI | 你手动操作、看进度、看音量的地方 |
| **② 音频/视频引擎** | 分两种形态：**默认是隐藏的 `bgAudio`**（只出声音，你看不到它），**只有你点「展开」看视频时才换成 B 站官方 `<iframe>`**（画面+声音一起出） | `bgAudio` 看不见；`iframe` 展开时能看见画面 | `bgAudio` 是**唯一的常规声音来源**，无论你在前台还是后台/锁屏，日常听歌都是它在放。只有你主动点「展开」，才换成 `iframe` |

**核心原则：①"底部播放器"永远只是一个"遥控器 + 显示屏"，它自己不产生声音。真正在播放声音的是②，而②默认情况下、绝大多数时间里都是 `bgAudio`——不管你在不在前台看这个页面。**

只有你主动点击「展开」按钮想看视频画面时，②才会临时切换成 iframe；点击「收起」后立刻切回 `bgAudio` 继续出声。

## 一、播放进度条——两种模式，主语不同

### 结论先说

**播放进度条的数据来源取决于你有没有点开视频画面：**

| 场景 | 进度条数据来源 | 是否是"真实"进度 |
|------|--------------|----------------|
| **默认情况**（没点「展开」，`bgAudio` 在放） | 直接读取 `bgAudio` 自己的真实播放位置（`currentTime`） | ✅ **是真实进度**，每秒读取一次 |
| **点了「展开」看视频**（换成 iframe 播放） | WeMusic 自己用"每秒走一格"的计时器估算 | ❌ **是估算值**，不是真实读出来的 |

### 为什么默认情况下能拿到真实进度？

因为默认播放声音的 `bgAudio` 是 WeMusic 自己创建、自己控制的一个播放器元素，**不是跨域的第三方内容**，所以可以随时读取它"现在播到第几秒了"这个真实数据，直接显示在进度条上。这是这次架构改造带来的直接好处——旧版本前台用 B 站 iframe 播放时读不到真实进度，只能靠计时器估算再做各种校正；现在只要你没有点开视频，进度条数字就是真的。

### 为什么点开视频后又变回"估算"？

因为这时候声音+画面全部交给了 B 站官方的 `<iframe>`。iframe 是"别人家的网页嵌在你的网页里"，浏览器出于安全考虑，**不允许你的网页读取 iframe 里面到底播到第几秒**。所以只要你点开视频看画面，进度条就退化成旧版本那种"每秒走一格"的估算方式，跟点开视频前的精确进度会有细微误差——但会用 B 站播放器发出的一个初始化信号做一次性校正（修正 iframe 从创建到真正出声之间的启动延迟）。

### 进度条为什么不能拖动？

因为无论哪种模式，WeMusic 都没有能力命令播放引擎"跳到指定的某一秒"再精确接续播放（`bgAudio` 播的是流媒体直链，iframe 是别人的播放器），所以进度条从设计上就是"只读展示"，拖动被禁用。

### 什么时候判定"这首歌放完了，自动切下一首"？

- **默认情况**（`bgAudio` 在放）：以 `bgAudio` 真正发出的"我放完了"信号为准（真实事件，最准确）。
- **展开视频看画面时**：计时器估算的时间数到了歌曲总时长，就认为放完了（跟旧版本一致）。

## 二、音量——一个滑块，但背后对接的东西会随场景变化

### 表格先看结论

| | 音量滑块表面上显示的 | 默认情况下实际听到的音量 | 展开视频看画面时实际听到的音量 |
|---|---|---|---|
| 主语 | ①底部播放器的音量滑块（唯一的手动调节入口） | **`bgAudio` 背后的音量放大器（GainNode）** | **B 站 iframe 自己的音量** |
| 你拖动滑块时，谁在变 | 滑块位置、显示的百分比 | 直接调节音量放大器的倍数 | 发一条"指令"告诉 iframe："帮我把你自己的音量调到 XX%" |
| 是否受"自动音量拉平"影响 | 不影响滑块显示 | **受影响**（会在你手动调的音量基础上再乘一个自动倍数） | **不受影响**（iframe 只听你手动调的音量） |

**换句话说：默认情况下（绝大多数听歌场景），音量滑块直接、精确地控制着音量放大器，效果最好，还能叠加自动音量拉平。只有你主动点开视频看画面的这段时间，音量控制会退化成"远程喊话"B 站播放器调节，且不再有自动拉平效果。**

这跟进度条是同一个道理：**不点开视频画面，你得到的是"精确控制"；点开视频画面，画面和声音都交给了别人的播放器，WeMusic 只能做"远程喊话"式的有限控制。**

### 为什么默认情况下能做到"精确控制音量+自动拉平"？

因为默认在放声音的 `bgAudio` 是 WeMusic 自己的元素，接入了浏览器提供的音频处理能力（一个叫 GainNode 的"数字音量旋钮"），可以被精确、实时地调节，还可以在这个旋钮上叠加"自动拉平"的倍数。

### 为什么展开视频后做不到？

因为这时候出声音的是 B 站官方的 iframe 播放器，它内部的音量是 B 站自己管理的，WeMusic 的网页没有权限直接伸手进去改它的音量数值，**只能像发短信一样，发一条"请把音量调到 XX%"的消息**，B 站播放器收到后自己去调整。所以这段时间的音量精确度、生效速度完全取决于 B 站播放器本身，也无法叠加自动拉平。

### 什么是"自动音量拉平"（只在默认播放模式下生效）？

不同视频原始录音的响度差异很大，同样调到 50% 音量，有的歌听起来很轻，有的很响。WeMusic 加了一个自动拉平功能：

1. 后端第一次要播放某个视频的音频时，会在服务器后台异步下载一份完整音频，用专业工具分析这段音频"实际有多响"（这个分析不影响你正常听歌，做完一次后存起来，下次同一个视频不用再分析）。
2. 根据分析结果算出一个"倍数"：本来就轻的视频算出的倍数 > 1（放大），本来就响的视频算出的倍数 < 1（压低）。
3. 默认播放模式下，这个倍数会叠加在你手动调的音量上，一起作用在音量放大器（GainNode）上。
4. 为防止某些视频原始录音特别轻、算出的倍数过大导致炸耳朵，设置了上限（最多放大 8 倍）。
5. 这个自动拉平功能可以在设置里手动关闭，关闭后只按你手动调的音量播放，不叠加任何自动倍数。
6. **这个功能只在默认播放模式（`bgAudio` 出声）下生效**，展开视频看画面时不生效（原因见上）。

### 无论前台后台，默认模式下音量体验完全一致

因为默认情况下无论你是不是在看这个页面（切标签页、锁屏都一样），出声音的始终是同一个 `bgAudio`，音量滑块也始终对接同一个音量放大器（GainNode）。**不会再出现"前台调音量效果不一样、后台才生效"这类旧版本的问题**——这是这次架构改造的直接收益之一。

## 三、播放声音的来源：直连 B 站 vs 走自建服务器代理

这一部分跟进度/音量没有直接关系，但同属"播放链路"的一部分，一并说明：

- WeMusic **优先让浏览器直接从 B 站的 CDN 服务器下载音频**播放，不经过 WeMusic 自己的服务器转发。这样做的好处是自建服务器不需要消耗自己的网络带宽去"转发"音乐数据流。
- 极少数情况下（比如分配到的 B 站 CDN 节点策略比较严格），直连会失败，这时 WeMusic 会自动切换成"经过自己服务器转发"的方式播放，保证不会真的放不出声音，只是那首歌可能需要多等一点点时间才能开始播放。
- 这个"直连优先，失败才走自己服务器"的逻辑，跟音量/进度控制无关，全部在音频引擎内部自动处理，你不会感知到区别（除非直连失败触发了 fallback，那时会稍微多等一点加载时间）。

## 四、常见疑问快问快答

**Q: 我拖动①底部播放器的进度条，能跳到指定时间点吗？**
A: 不能，进度条被禁用（`disabled`），只能看，不能拖，无论是不是在看视频。

**Q: 默认听歌（没点展开视频）时，进度条上的时间准吗？**
A: 准，是直接读取播放引擎（`bgAudio`）的真实播放位置，不是估算的。

**Q: 我点开视频看画面后，进度条还准吗？**
A: 会有轻微估算误差（跟旧版本一样靠计时器数秒数），因为画面+声音这时候交给了跨域的 B 站播放器，WeMusic 读不到它的真实进度。

**Q: 默认听歌时调音量，是不是会被"自动拉平"影响？**
A: 会。默认模式下自动拉平和你手动调的音量会叠加生效。

**Q: 展开视频看画面时调音量，还会有自动拉平效果吗？**
A: 不会。这时候音量完全由 B 站播放器自己处理，只受你手动调的滑块位置影响，没有自动拉平。

**Q: 我切到浏览器后台/锁屏，声音会中断吗？**
A: 不会。默认模式下出声音的一直是同一个 `bgAudio`，前后台没有任何切换动作，自然不会有中断或音量突变的问题。

**Q: 如果我展开视频看画面后，把浏览器切到后台会怎样？**
A: 会自动帮你"收起"视频（因为 iframe 被浏览器限流后会没声音），改回 `bgAudio` 继续放声音，保证切到后台后依然听得到。
