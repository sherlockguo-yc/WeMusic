# 文本左右滚动

## 设计目标

底栏歌名、底栏视频标题、浮窗视频标题在文字过长时自动左右滚动。

## 实现方案

**rAF 状态机驱动**（非 CSS keyframes），统一速度 40px/s：

```
暂停 2s → 滚动 → 暂停 2s → 回滚 → 重复
```

所有滚动位置用「父级 `overflow: hidden` 裁剪 + 内层 span `translateX`」。

```js
// src/player.js
function startMarquee(inner, extra) {
  const SPEED = 40;  // px/s
  const PAUSE = 2000;
  const scrollTime = (extra / SPEED) * 1000;
  // rAF tick 累加 dt → 按阶段计算 translateX
  // hover 暂停：mouseenter 停累加，mouseleave 继续
}
```

检测溢出：`inner.scrollWidth > parent.clientWidth + 4`。

## 三处位置

| 位置 | 滚动原理 | 触发 | rAF 帧数 |
|------|----------|------|----------|
| 底栏歌名 | `.player-left` flex 子项（`flex: 1 1 auto`），inner span 在列内 translateX | `checkMarquee($('npTitle'))` | 双帧 ✅ |
| 底栏视频标题 | `.player-right` flex 子项（`flex: 0 1 auto`，允许缩小以触发溢出检测），列宽由 flex 自动计算 | `checkMarquee(statusInner)` | 双帧 ✅ |
| 浮窗视频标题 | `flex: 1`，inner span translateX | `setVpTitle(title)` | 双帧 ✅（2026-07-07 从单帧修复为双帧） |

共用一个 `startMarquee(inner, extra)`，三处速度一致。

**⚠️ 为什么必须双帧 rAF？** 视频浮窗从 `display: none` → `display: flex` 时，浏览器需要 Frame 1 计算可见性变化、Frame 2 计算 flex 布局。单帧 rAF 会读到 `clientWidth=0`，导致 `scrollWidth - 0 = 0`，溢出检测失败。底栏元素始终可见，但统一使用双帧以避免平台差异。

## 布局原则：`.player` 是 3 列居中 Grid，禁止子元素用 absolute 定位

**问题演变史（记录教训，避免好心又犯）**：
1. 第一次（~2026-06）：`.player-status` 用 `position: absolute` + 硬编码 `right: 210px` 避让 `.player-right`，按钮数量/字号一变就失效 → 文字覆盖「词」按钮。
2. 第一次"修复"：把 `.player-status` 改成了 grid 列 —— **但只改了一半**。`.progress-center`（进度条）仍然是 `position: absolute; left: 50%` 独立居中，跟 grid 是两套并行的布局算法。结果窗口/字号一变，`.progress-center` 和新的 `status` 列又重新对不上 → 视频标题覆盖到进度条时长上。
3. 第二次"修复"：把所有 6 个子元素都收编为同一 grid 的 6 条轨道，彻底消除 absolute。**但 6 列 grid 左右不对称**（左侧 cover+np+controls 比右侧 status+right 宽），导致进度条虽然列内居中，但整体偏左 58px，不处于播放条的视觉中心。
4. 第三次"修复"（2026-07-07）：6 列改为 3 列 `1fr auto 1fr`，进度条天然居中。**但是**，`.player-status` 的 `flex: 0 0 auto`（`flex-shrink: 0`）被从 6 列 grid 时代**无脑搬运**过来——当时 status 和进度条共享相邻列需要它，但在 3 列布局中 status 和进度条在不同列，这个约束不再有意义。更致命的是：在 flex 子元素的上下文中，`flex-shrink: 0` 阻止了容器缩小，导致 `parent.clientWidth` 始终 ≥ `inner.scrollWidth`，溢出检测永不触发 → **文本不再滚动**。修复：`flex: 0 0 auto` → `flex: 0 1 auto`。
5. 第五次（2026-07-07，同日）：**浮窗视频标题也不滚动了**。原因：`setVpTitle()` 只用了**单帧** `requestAnimationFrame` 检测溢出，而视频浮窗从 `display:none` 变为 `display:flex` 时，浏览器需要**两帧**才能完成 flex 布局计算——单帧读到 `clientWidth=0` → overflow=0 → 滚动永不启动。对比底栏的 `checkMarquee()` 使用双帧 rAF 一直正常工作。修复：统一为双帧 rAF + `clientWidth > 0` 守卫。
6. 第六次（2026-07-10）：**音量滑块不可见，只能静音/非静音切换**。原因：`.player`（第 4 次修复时添加的 `overflow: hidden`）+ `.player-left` + `.controls` 三层 `overflow: hidden` 链条裁剪了 `#volBar`（`position: absolute; bottom: calc(100% + 58px)` 定位在 `.volume-wrap` 上方）。JS 逻辑正常（点击按钮添加 `.open` class 使 `opacity: 1`），但用户看不到滑块。——> 第一次点击打开了滑块（被裁剪不可见），第二次点击因为 `.open` 已存在而走 else 分支切换静音。修复：移除 `.player`、`.player-left`、`.controls` 三处 `overflow: hidden`。

**根本教训 A**：**只要 `.player` 内部同时存在「grid 布局」和「absolute 布局」两套定位系统，就必然会在某个尺寸下互相打架**。修一个坑，另一个坑会在别处冒出来，永远修不完。

**根本教训 B**（2026-07-07 新增）：**布局重构时不能只搬运 CSS 属性**。从 Grid 子元素变成 Flex 子元素时，同一个属性（如 `flex`）的语义完全不同——必须逐条审查，而非直接复制旧值。`flex-shrink: 0` 在 Grid 子元素上只影响自身最小宽度，在 Flex 子元素上却阻止了整个容器缩小 → 直接导致滚动功能失效。

**根本教训 C**（2026-07-07 新增）：**溢出检测必须统一使用双帧 rAF**。`display: none` → `display: flex` 的元素需要两帧才能完成布局计算，单帧 `requestAnimationFrame` 在不同浏览器/时机下表现不一致——有时能读到正确值、有时不能。`checkMarquee()` 一直用双帧，`setVpTitle()` 只用单帧，就是 3 个滚动位置中有 1 个会间歇性失效的根因。**任何新的滚动检测代码必须使用双帧 rAF**，不允许抄 `setVpTitle` 的单帧写法。

**根本教训 D**（2026-07-10 新增）：**给容器加 `overflow: hidden` 前，必须检查其子元素是否有 `position: absolute` 需要溢出到容器之外的元素**。即使有注释说明「这是最后一道防线」，也必须确认不会误伤任何绝对定位的子元素。本案例中 `.player` 的 `overflow: hidden` 是第 4 次修复时添加的（防高倍缩放），当时没有考虑 `#volBar` 的 absolute 定位需求 —— 所有子列已有自己的 `overflow: hidden`，`.player` 层的这个是冗余且有害的。

**最终根治方案**（当前实现，`public/css/style.css` `.player` 规则块）：

把 `.player` 下所有横排内容拆为**3 个逻辑组**，用 `1fr auto 1fr` 三列 grid 让进度条天然居中：

```css
.player {
  display: grid;
  /* 3 列：左（cover+np+controls）| 中（进度条居中列）| 右（status+按钮组）
     中间 auto 宽度由 grid 居中算法天然放在视觉中心，左右 1fr 平分剩余空间 */
  grid-template-columns: minmax(0, 1fr) auto minmax(0, 1fr);
  grid-template-areas: "left center right";
  align-items: center; gap: 0 12px;
  /* 不设 overflow:hidden — 音量滑块 absolute 定位需要溢出到 .player 上方（2026-07-10） */
}
/* 左列：cover + np + controls 的 flex 容器（不设 overflow:hidden — 同上） */
.player-left { grid-area: left; display: flex; align-items: center; gap: 6px; min-width: 0; }
/* 中列：进度条精确居中，width 400px + max-width 100% */
.player .progress-center { grid-area: center; display: flex; justify-content: center; align-items: center; width: 400px; max-width: 100%; min-width: 0; overflow: hidden; }
/* 右列：status + 展开/换源/队列按钮，justify-content: flex-end */
.player-right { grid-area: right; display: flex; align-items: center; justify-content: flex-end; gap: 6px; flex-shrink: 0; min-width: 0; overflow: hidden; }
/* controls 不设 overflow:hidden — 音量滑块同上（2026-07-10） */
.player .controls { display: flex; align-items: center; gap: 10px; flex-shrink: 0; min-width: 0; }
```

Grid 的 `1fr auto 1fr` 模板**在数学上保证**中列位于播放条水平中心——这是浏览器排版引擎的基本保证，不依赖任何人工估算的像素值。经实测验证：1440px/1024px/800px/600px 所有宽度下进度条中心偏差均 ≤ 2px（亚像素舍入）。

### 高倍缩放下的溢出防护（2026-07-04 补充，2026-07-10 修订）

Grid 轨道不重叠，但**单元格内容可能溢出到相邻格**（尤其 controls/status/progress 列的文本和按钮）。防护措施：

1. **grid 子项加 `overflow: hidden`**：`.progress-center`、`.player-right`、`.progress-wrap` → 内容被裁剪在各自轨道内
2. **900px 中等宽度断点**：125%~175% 缩放场景下按钮缩小、间距收紧，提前适配而非等到 720px 移动端才切换布局
3. **⚠️ `.player`、`.player-left`、`.controls` 不再设 `overflow: hidden`**（2026-07-10）：
   - 原因：音量滑块 `#volBar` 使用 `position: absolute; bottom: calc(100% + 58px)` 定位在 `.volume-wrap` 上方，需要溢出到 `.player` 之外
   - 三层 `overflow: hidden` 链条（`.player` → `.player-left` → `.controls`）会把滑块裁剪，用户看不到也无法调整
   - 这些容器内的元素都不会水平溢出（`.np` 有自己独立的 `overflow: hidden` 处理文本截断，其他子元素都是固定尺寸按钮）
   - 不影响 `.progress-center` 和 `.player-right`——它们仍然保有 `overflow: hidden`

**铁律（新增/修改 `.player` 内任何元素前必读）**：
1. **禁止**给 `.player` 的任何直接/间接子元素设置 `position: absolute` 来做「相对整个播放条」的定位。这会让它跳出 grid 轨道计算，重新制造出两套独立算法互相打架的问题。
2. 如果确实需要视觉微调，只能通过调整 `grid-template-columns` 的比例或在轨道内部用 `justify-self`/`margin` 微调，**不能跳出所在轨道的边界**。
3. 新增一个横排元素（比如以后加个「倒计时」「音质标签」之类）时，先想它属于左/中/右哪个现有分组，还是需要新增一条 `grid-template-columns` 轨道——**不要**图省事用 `position: absolute` 糊一个位置上去。
4. 移动端 `@media (max-width: 720px)` 下 `.player` 切回 `flex; flex-wrap: wrap`（多行布局），通过 `.player-left { display: contents; }` 将其子元素升到 `.player` 的直接 flex 层级，复用原有 order/flex-basis 控制换行。不要往 flex 模式里加 grid 专用的 `grid-area` 依赖。
5. **`.player`、`.player-left`、`.controls` 不得加 `overflow: hidden`**（2026-07-10 修订）——音量滑块 `#volBar` 需要绝对定位溢出到 `.player` 上方，这些容器上的 `overflow: hidden` 会裁剪掉滑块。唯一例外：`.np`、`.progress-center`、`.player-right`、`.progress-wrap` 等需要裁剪内容的子元素仍应保留 `overflow: hidden`。

**防重复检查清单（每次改动播放器底栏后必测）**：
1. `grep -n 'position: absolute' public/css/style.css` 附近查看是否落在 `.player` 的直接子元素上——如果是，说明又在开历史倒车。
2. 用浏览器 devtools 或脚本抓取 `.progress-center` 的 `getBoundingClientRect()`，确认 `centerX` 与 `.player` 的 `centerX` 偏差 ≤ 3px，窗口宽度覆盖 720px～1920px 多档测试。
3. 测试场景：超长视频标题（如「【4K顶级修复】《外婆》—周杰伦 无损音质」）+ 字号调到最大（18px）+ 窗口压到最窄（>720px，未触发移动端布局）。
4. 构建后跑一遍上面的测试脚本或人工缩放窗口检查，而不是只看默认宽度下的截图。
5. **播放超长标题歌曲（如「【4K顶级修复】《外婆》—周杰伦 无损音质」），确认底栏视频标题自动左右滚动。**

## 相关 CSS 属性黑名单（修改时触发回归检查）

以下 CSS 属性的变更会**直接影响文本滚动的溢出检测**，修改它们时**必须**测试滚动功能：

| 属性 | 影响机制 | 安全值 |
|------|---------|--------|
| `.player-status` 的 `flex-shrink` | `flex-shrink: 0` 阻止容器缩小 → `clientWidth` 永不低于内容宽度 → 溢出不可检测 | **必须 `flex-shrink ≥ 1`** 或 `min-width: 0` + `overflow: hidden` |
| `.player-right` 的 `flex-shrink` | 同上（`.player-right` 本身不会导致问题，但其子元素会） | `flex-shrink: 0` 在此处安全（它控制 Grid 列宽，不影响内部 flex 子元素） |
| `.status-inner` 的 `display` | `display: block` 或 `min-width: max-content` 使 scrollWidth 错误 | `display: inline-block` + `white-space: nowrap` |
| `.status-inner` 的 `display` | `display: block` 或 `min-width: max-content` 使 scrollWidth 错误 | `display: inline-block` + `white-space: nowrap` |
| `.player-right` 的 `overflow` | 必须是 `hidden` 才能裁剪超出内容 | `overflow: hidden` |
| `setVpTitle` 的 rAF 帧数 | **单帧 rAF 在 display:none→flex 时读到 clientWidth=0** → 溢出永不触发 | **必须双帧 rAF**，与 `checkMarquee()` 一致 |

## 常见问题

- **不滚动**：
  - inner span 不能设 `min-width: max-content`（曾出现在 `.vp-title-inner` 上，2026-07-05 移除）；
  - 父容器必须是 flex item 且 `flex-shrink > 0`（`flex: 0 0 auto` 的 `flex-shrink: 0` 会导致 `parent.clientWidth` 始终 ≥ `inner.scrollWidth`，永不溢出——2026-07-07 修复）；
  - **浮窗视频标题必须双帧 rAF**（`display:none→flex` 元素需要两帧完成布局，单帧会读到 `clientWidth=0`——2026-07-07 修复）；
  - 确保双帧 rAF 后再检测溢出
- **文字被 `…` 截断**：父级必须移除 `text-overflow: ellipsis`（`.marquee` class 处理）
- **标题空白**：`startVideo` 的 `displayTitle` 有歌名兜底，`setVpTitle` 默认 "Bilibili 播放"
