# 文本左右滚动

## 设计目标

底栏歌名、底栏视频标题、浮窗视频标题在文字过长时自动左右滚动。

## 实现方案

**rAF 状态机驱动**（非 CSS keyframes），统一速度 40px/s：

```
暂停 2s → 滚动 → 暂停 2s → 回滚 → 重复
```

所有滚动位置用「父级 `overflow: hidden` 裁剪 + 内层 span `translateX`」。

检测溢出：`inner.scrollWidth > parent.clientWidth + 4`。

## 三处位置

| 位置 | 滚动原理 | 触发 |
|------|----------|------|
| 底栏歌名 | grid 列 60-180px，inner span 在列内 translateX | `checkMarquee($('npTitle'))` |
| 底栏视频标题 | `.player` grid 的 `status` 列（`minmax(60px, 260px)`），列宽由浏览器统一计算 | `checkMarquee(statusInner)` |
| 浮窗视频标题 | flex:1，inner span translateX | `setVpTitle(title)` |

共用一个 `startMarquee(inner, extra)`，三处速度一致。

## 布局原则：`.player` 是单一 Grid，禁止子元素用 absolute 定位

**问题演变史（记录教训，避免好心又犯）**：
1. 第一次：`.player-status` 用 `position: absolute` + 硬编码 `right: 210px` 避让 `.player-right`，按钮数量/字号一变就失效 → 文字覆盖「词」按钮。
2. 第一次"修复"：把 `.player-status` 改成了 grid 列 —— **但只改了一半**。`.progress-center`（进度条）仍然是 `position: absolute; left: 50%` 独立居中，跟 grid 是两套并行的布局算法。结果窗口/字号一变，`.progress-center` 和新的 `status` 列又重新对不上 → 视频标题覆盖到进度条时长上。

**根本教训**：**只要 `.player` 内部同时存在「grid 布局」和「absolute 布局」两套定位系统，就必然会在某个尺寸下互相打架**。修一个坑，另一个坑会在别处冒出来，永远修不完。

**最终根治方案**（当前实现，`public/css/style.css` `.player` 规则块）：

把 `.player` 下所有横排的直接子元素——封面、歌名、控制按钮、**进度条**、视频标题、右侧按钮组——全部收编为**同一个 grid 的 6 条轨道**：

```css
.player {
  display: grid;
  /* controls/status/right 全部使用 minmax + fr 弹性宽度，缩放时同比例压缩 */
  grid-template-columns: 46px minmax(60px, 150px) minmax(120px, auto) minmax(140px, 1fr) minmax(60px, 1fr) minmax(60px, auto);
  grid-template-areas: "cover np controls progress status right";
  overflow: hidden; /* 高倍缩放时防止 grid 子元素溢出容器 */
}
.player .progress-center { grid-area: progress; }   /* 不再 absolute！ */
.player .player-status   { grid-area: status; }
.player .player-right    { grid-area: right; }
```

Grid 轨道之间**在数学上不可能重叠**——这是浏览器排版引擎的基本保证，不依赖任何人工估算的像素值。只要所有子元素都在轨道内，重叠就是架构上不可能发生的事情，不需要每次靠猜数字修。

### 高倍缩放下的溢出防护（2026-07-04 补充）

Grid 轨道不重叠，但**单元格内容可能溢出到相邻格**（尤其 controls/status/progress 列的文本和按钮）。防护措施：

1. **所有 grid 子项加 `overflow: hidden`**：`.controls`、`.progress-center`、`.player-status`、`.player-right`、`.progress-wrap` → 内容被裁剪在各自轨道内
2. **900px 中等宽度断点**：125%~175% 缩放场景下按钮缩小、间距收紧，提前适配而非等到 720px 移动端才切换布局
3. **`.player` 容器加 `overflow: hidden`**：最后一道防线，防止任何元素溢出播放器整体区域

**铁律（新增/修改 `.player` 内任何元素前必读）**：
1. **禁止**给 `.player` 的任何直接/间接子元素设置 `position: absolute` 来做「相对整个播放条」的定位。这会让它跳出 grid 轨道计算，重新制造出两套独立算法互相打架的问题。
2. 如果确实需要视觉微调，只能通过调整 `grid-template-columns` 的 `minmax()`/`fr` 比例，或在轨道内部用 `justify-self`/`margin` 微调，**不能跳出所在轨道的边界**。
3. 新增一个横排元素（比如以后加个「倒计时」「音质标签」之类）时，先想它属于哪条现有轨道，还是需要新增一条 `grid-template-columns` 轨道——**不要**图省事用 `position: absolute` 糊一个位置上去。
4. 移动端 `@media (max-width: 720px)` 下 `.player` 切回 `flex; flex-wrap: wrap`（两行布局），这是有意为之的独立布局模式，不要往 flex 模式里加 grid 专用的 `grid-area` 依赖。
5. **所有 grid 子元素必须保留 `overflow: hidden`**，这是防止高倍缩放时内容溢出到相邻轨道的唯一手段。新增 grid 子元素时检查是否遗漏。

**防重复检查清单（每次改动播放器底栏后必测）**：
1. `grep -n 'position: absolute' public/css/style.css` 附近查看是否落在 `.player` 的直接子元素上——如果是，说明又在开历史倒车。
2. 用浏览器 devtools 或脚本抓取 `.progress-center` / `.player-status` / `.player-right` 三者的 `getBoundingClientRect()`，确认两两之间 `right <= left`（无重叠），窗口宽度覆盖 720px～1920px 多档测试。
3. 测试场景：超长视频标题（如「【4K顶级修复】《外婆》—周杰伦 无损音质」）+ 字号调到最大（18px）+ 窗口压到最窄（>720px，未触发移动端布局）。
4. 构建后跑一遍上面的测试脚本或人工缩放窗口检查，而不是只看默认宽度下的截图。

## 常见问题

- **不滚动**：inner span 不能设 `min-width: max-content`；确保双帧 rAF 后再检测溢出
- **文字被 `…` 截断**：父级必须移除 `text-overflow: ellipsis`（`.marquee` class 处理）
- **标题空白**：`startVideo` 的 `displayTitle` 有歌名兜底，`setVpTitle` 默认 "Bilibili 播放"
