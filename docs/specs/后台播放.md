# 播放架构（单一音频源）

> 描述 WeMusic 播放架构、状态流转、已知问题及修复历史。
> 修改 `src/player.js` 中播放相关代码前**必须阅读本文档**。
>
> **2026-07-13 架构重写**：原「前台 iframe / 后台 bgAudio 双引擎切换」架构已废弃，
> 改为「bgAudio 单一音频源 + 展开视频才切 iframe」架构。原因、方案对比见
> `docs/requirements/播放架构改造-单一音频源.md`。

## 一、架构概览

WeMusic 只有一个真正会发声的「音频源」：`bgAudio`（一个隐藏的 `<audio id="audio">` 元素）。
无论用户在浏览器前台还是后台/锁屏，日常听歌全部由 `bgAudio` 播放，因此**不存在**「前后台切换音频引擎」这件事，也不需要交叉过渡、健康检查、双引擎并存等复杂逻辑。

唯一的例外：用户主动点击底部播放器的「展开」按钮，想看视频画面时，才切换为 B 站官方 `<iframe>`（画面+声音一起播）。点击「收起」/关闭视频面板后，立即销毁 iframe，改回 `bgAudio` 从记录的进度继续播放声音。

| 状态（`mode` 变量） | 载体 | 说明 |
|------|------|------|
| `'bg'`（默认，唯一常规状态） | `bgAudio` | 无论前台/后台，唯一的声音来源 |
| `'iframe'`（仅展开视频时） | B 站 `<iframe>` | 画面 + 声音一起由 iframe 输出；`bgAudio` 静默预载但不播放 |

切换入口：`expandVideo()` / `collapseVideo()`（由 `videoBtn`/`vpHide` 按钮触发），**不再监听 `visibilitychange` 做音频引擎切换**。`visibilitychange` 仅保留一个用途：用户展开视频后把标签页切到后台时，自动触发 `collapseVideo()`（因为 iframe 会被浏览器限流/静音，必须交还给 bgAudio 才能保证后台仍有声音）。

## 二、音频直链：直连优先，代理兜底

为减少自建服务器带宽消耗（服务器不再代理转发每一次播放的完整音频流量），播放地址解析分两层：

```
GET /api/play/direct-url?bvid=xxx
  → 服务端内部仍走 getAudioStream()（WBI 签名 + 风控规避）解析出 B 站 CDN 真实直链
  → 直接把直链 { url, backup, cid, mime, expiresAt } 返回给前端
  → 前端 bgAudio.src = url（crossorigin="anonymous"，直接跨域播放，不经过自建服务器）
```

**实测结论**（2026-07-13）：B 站 CDN 音频直链本身带 `access-control-allow-origin: *`，用真实浏览器 UA 请求即可跨域播放成功，且能被 `AudioContext.createMediaElementSource()` + `GainNode` 正常处理（不会被标记为 tainted）。但不同网络环境可能分配到不同 CDN 厂商节点，风控策略可能不完全一致，因此保留兜底：

```
bgAudio 触发 error 事件
  → 若尚未 fallback 过 → src 切换为 /api/play/stream?bvid=xxx（服务端代理，现有接口）
  → 若已经 fallback 过仍失败 → 判定该曲目播放失败，自动切下一首
```

直链有效期约 114 分钟（`deadline` 查询参数），足够单曲播放，暂停超长时间后继续播放可能过期，届时会触发 error/stalled 走恢复流程重新取一次直链。

## 三、核心状态变量

```js
let mode = 'bg';               // 'bg' | 'iframe'
const bgAudio = document.getElementById('audio'); // <audio id="audio" crossorigin="anonymous">

let _usedFallback = false;     // 当前曲目是否已从「直连」降级为「代理」
let _healthCheckId = null;     // play() 成功后 5s 健康检查 setTimeout ID
let _stalledTimer = null;      // stalled 事件 3s 兜底切歌 setTimeout ID
let _volume = 0.8;             // 用户手动设置的音量 0~1（前后台/展开态统一生效）

let autoTimer = null;          // 每秒 tick 的 setInterval ID
let elapsed = 0;               // 当前播放进度（秒）
let totalDur = 0;              // 歌曲总时长（秒）
let timerPaused = false;       // 是否暂停自动连播
```

## 四、播放流程

### 4.1 正常播放一首歌（`playCurrent` → `startVideo`）

```
playCurrent()
  ↓
检查当前歌曲是否被标记为「不喜欢」→ 是 → playNext(true) 跳过
  ↓ 否
解析出 bvid
  ↓
startVideo(bvid, title, dur)
  ↓
if (mode === 'iframe')
    mountVideoAt(bvid, title, 0)     // 保持展开状态，画面切到新歌
    _loadBgTrack(bvid)                // 静默预载 bgAudio，供随时收起
else
    _loadAndPlayBgTrack(bvid, 0)      // 请求 direct-url → bgAudio.src → play()
  ↓
startTimer(dur) → _restartTick()      // 启动每秒 tick
```

### 4.2 展开视频（`expandVideo`，由「展开」按钮触发）

```
用户点击「展开」
  ↓
记录 bgAudio.currentTime 为起播秒数
  ↓
bgAudio.pause()
mode = 'iframe'
  ↓
mountVideoAt(bvid, title, 起播秒数)   // 创建 iframe，从该秒数起播
  ↓
_restartTick()                        // tick 切换为「计时器估算」模式
```

### 4.3 收起视频（`collapseVideo`，由「收起」按钮/关闭视频面板/标签页切后台触发）

```
用户点击「收起」（或标签页切到后台）
  ↓
记录 elapsed（展开期间计时器估算的进度）
  ↓
销毁 iframe
mode = 'bg'
  ↓
_playBgAudioWithRetry(记录的进度)      // bgAudio seek 到该进度 + play()（失败重试 10 次）
  ↓
_restartTick()                        // tick 切回「读取 bgAudio.currentTime」模式
```

### 4.4 自动切歌判定（`autoAdvance` 触发条件）

- `mode === 'bg'`：`bgAudio` 的 `ended` 事件（真实播完）触发。
- `mode === 'iframe'`：计时器估算的 `elapsed >= totalDur + 1` 触发（因为 iframe 跨域无法读取真实进度）。
- 两种模式都有的兜底：`bgAudio` 触发 `stalled`/`error` 且已用尽 fallback → 判定播放失败，自动切歌。
- `bg` 模式下的**停滞检测**：`_tick()` 中连续 5 次（约 5 秒）`bgAudio.currentTime` 无变化 → 判定流已死，自动切歌。

## 五、三层健壮性防护（针对 `bgAudio`，`mode === 'bg'` 时生效）

### 第一层：`_playBgAudioWithRetry` 的 `play()` 重试

- 最多 10 次，间隔递增 500ms → 3s。
- 需要 seek 到指定进度时，等待 `seeked` 事件再 `play()`（超时 1.5s 兜底直接 play）。
- 新曲目还没加载完（`readyState < 2`）时，等待 `canplay` 事件再 `play()`。

### 第二层：play() 成功后 5 秒健康检查

- `_healthCheckId` setTimeout：如果 `mode === 'bg' && !timerPaused && bgAudio.paused`，重新 `doPlay()`。

### 第三层：`_tick()` 每秒巡检（停滞检测 + 意外暂停恢复）

- 若 `bgAudio.paused`（且已经成功播放过一次，排除加载中的正常暂停态）→ 重新 `play()`。
- 若连续 5 次 `currentTime` 无变化 → 判定流已死，自动切歌。

### 第四层：`error` / `stalled` 事件恢复

- `error` 事件：先尝试直连→代理降级（仅一次），仍失败则自动切歌。
- `stalled` 事件：等 3 秒仍未恢复（未触发 `playing` 事件）则自动切歌。
- `playing` 事件触发时清除 `stalled` 定时器（避免短暂网络波动误切）。

## 六、不喜欢歌曲自动跳过（2026-07-13 新增）

用户标记为「不喜欢」（`source_type='song'`，存储在 `blocked_sources` 表）的歌曲会在多处被拦截：
- **`playCurrent()`**：如果当前歌曲被标记为不喜欢 → 调用 `stopPlayback()` + `playNext(true)` 自动跳过
- **`computeNextIndex()`**：计算下一首索引时跳过不喜欢的歌曲（顺序/随机模式均生效）
- **`playPrev()`**：返回上一首时同样跳过不喜欢的歌曲
- **`enqueueNext()`**（手动「下一首播放」）：如果目标是已标记不喜欢的歌曲 → toast 提示并拒绝添加

**死循环保护**：当队列中所有歌曲都被标记为不喜欢时，`computeNextIndex()` 返回 `-1`，`playNext()`/`playPrev()` 会 toast 提示「队列中的歌曲均已标记为不喜欢」并停止播放。

## 七、进度条

`mode === 'bg'` 时，进度条显示的时间 = `bgAudio.currentTime` 的真实播放位置（每秒读取一次），这是**真实进度**，不再需要多路校准逻辑（旧架构里 postMessage 心跳校正、bgSync 定时校准等均已删除，因为进度来源统一且真实）。

`mode === 'iframe'` 时（仅展开视频期间），因 iframe 跨域无法读取真实播放位置，仍沿用「每秒 +1」的计时器估算，并保留 B 站 postMessage 心跳一次性校正（`_onBiliMessage`，修正 iframe 从创建到真正出声之间的延迟）。

**拖拽跳转（2026-07-13 新增）**：`mode === 'bg'` 时 `seekBar`/`lpSeekBar` 可拖动，松手触发 `seekTo(sec)` → 直接设置 `bgAudio.currentTime`（因为 `bgAudio` 是本地元素，可精确 seek）。`mode === 'iframe'` 时 `seekBar` 保持 `disabled`（B 站官方未公开支持 postMessage 精确 seek），切换 mode 时由 `_updateSeekInteractivity()` 同步更新 disabled 状态。同时接入 Media Session 的 `seekto`/`seekbackward`/`seekforward`，支持锁屏/耳机快进快退（仅 `mode === 'bg'` 生效），并通过 `setPositionState()` 让系统进度条与 `elapsed`/`totalDur` 保持同步。

详细的「进度条主语是谁」「音量到底控制什么」说明见 `docs/specs/播放进度与音量控制.md`。

## 八、已废弃的旧机制（供历史排查参考）

以下机制在 2026-07-13 重构中已删除，如果在旧代码/日志中看到相关字样，说明是重构前的行为：

- `visibilitychange` 驱动的前后台音频引擎切换（`_bgPreload`/`_bgUnmuteAndPlay`/交叉过渡等）
- `_startBgSync`/`_stopBgSync`：后台每 4 秒用 `bgAudio.currentTime` 校准计时器的定时器（现在 `bg` 模式下进度本身就直接读 `bgAudio.currentTime`，不需要"校准"一个独立计时器）
- 前台 iframe 静音预缓冲 `bgAudio`、切后台交叉过渡等待 iframe canplay 才停 `bgAudio` 的逻辑
- `paneVisible`/`applyPaneVisibility`/`setPaneVisible`：视频面板显示状态改由 `mode` 变量直接决定，不再需要独立的"面板可见性"状态
