# 需求文档：播放架构改造——bgAudio 作为唯一音频源

> 状态：已完成编码与验证，含拖拽进度条跳转的扩展实现
> 关联规格文档：`docs/specs/后台播放.md`、`docs/specs/播放进度与音量控制.md`
> 本文档记录需求讨论过程与最终实现结果。

---

## 一、背景与动机

### 现状问题

当前播放器采用"前台 iframe + 后台 bgAudio"双引擎切换架构（详见 `docs/specs/后台播放.md`）：

- 前台播放时，声音由 B 站官方 `<iframe>` 播放器输出，音量只能通过 `postMessage` 远程喊话调节，无法叠加服务端音量归一化。
- 后台播放时，切换为隐藏的 `<audio>`（`bgAudio`）输出，通过 `GainNode` 精确控制音量，且支持归一化。
- 两套引擎之间需要复杂的交叉过渡、健康检查、stalled/error 恢复等逻辑（`visibilitychange` 触发），是当前代码中最复杂、最容易出 bug 的部分。
- 音量体验不一致：前台/后台切换时实际听感可能有差异（归一化仅后台生效）。
- 服务器目前对所有音频请求做代理转发（`/api/play/stream`），带宽全部消耗在自建服务器上行链路。

### 用户核心场景

**用户绝大部分时候是"听歌"，不需要看视频画面**。只有主动点击"展开视频"时才需要画面。因此没必要为了极少数"看视频"场景，让所有播放都背负双引擎切换的复杂度和体验风险。

---

## 二、步骤 1：用户场景（一句话）

> 用户日常听歌时，无论浏览器在前台还是切到后台/锁屏，声音都应该始终来自同一个播放引擎（`bgAudio`），音量调节、播放进度、归一化效果三者在任何前后台状态下保持完全一致；只有当用户主动点击"展开"想看视频画面时，才切换为 B 站 iframe 播放（画面+声音），点击"收起"后立即无缝切回 `bgAudio` 继续播放声音。

## 三、步骤 2：数据流

### 3.1 音频直链获取（新增）

```
用户播放某首歌
  → 前端调用 /api/play/resolve（不变，仍返回候选 + best 匹配）
  → 新增：前端调用新接口获取"可直连播放的音频地址"
       GET /api/play/direct-url?bvid=xxx[&cid=xxx]
       → 服务端内部仍走现有 getAudioStream()（WBI 签名 + 风控规避）解析出 CDN 真实直链
       → 返回 { url, backup, cid, expiresAt }  ← 直接把直链交给前端，不做代理转发
  → bgAudio.crossOrigin = 'anonymous'; bgAudio.src = url
  → bgAudio.play()
```

### 3.2 直连失败兜底（新增）

```
bgAudio 触发 error 事件
  → 判断：是否已经 fallback 过一次？
     否 → 将 src 切换为服务端代理地址 /api/play/stream?bvid=xxx（现有接口，不变）
          重新 load() + play()
     是 → 判定该曲目播放失败，toast 提示 + 自动切下一首
```

### 3.3 音量归一化 gain 查询（不变）

```
GET /api/play/gain?bvid=xxx → { gain, status }（现有接口，逻辑不变）
仍然只作用于 bgAudio 的 GainNode，因为现在 bgAudio 是唯一音频源，
归一化效果变为"全场景生效"（不再区分前后台）。
```

### 3.4 展开/收起视频（触发源替代 visibilitychange）

```
用户点击「展开」（videoBtn）
  → 记录 bgAudio.currentTime（当前真实播放进度）
  → bgAudio.pause()
  → 创建 iframe（不静音），用记录的进度作为起播秒数
  → 画面+声音全部由 iframe 输出

用户点击「收起」（vpHide）或关闭视频面板（vpClose）
  → 销毁 iframe
  → bgAudio.currentTime = 收起前记录/推算的进度
  → bgAudio.play()（无缝续播声音）
```

### 3.5 空值/异常处理

| 场景 | 处理 |
|------|------|
| `/api/play/direct-url` 请求失败（网络错误/接口异常） | 直接 fallback 到 `/api/play/stream` 代理地址，不额外提示用户 |
| CDN 直链跨域被拒绝（无 CORS 头/403） | 触发 `error` 事件 → 走 3.2 的 fallback 流程 |
| CDN 直链超过有效期（约 114 分钟，实测 `deadline` 参数）仍在播放 | 播放到一半失效会触发 `error` 或 `stalled`，走现有的 stalled/error 恢复逻辑，恢复时重新请求一次新直链 |
| 展开视频时 `bgAudio` 还没加载出真实进度（刚切歌） | 用当前维护的 `elapsed`（计时器值）兜底 |
| 用户在视频展开状态下切到浏览器后台标签页 | 自动触发"收起"逻辑，销毁 iframe，切回 bgAudio 播放（避免 iframe 被浏览器后台限流导致声音中断） |

## 四、步骤 3：状态机

### 4.1 音频源状态（替代原来的"前台/后台"状态）

```
[bgAudio 播放中] ──点击展开──> [iframe 播放中（画面+声音）]
       ^                              │
       │                          点击收起 / vpClose /
       │                          切到浏览器后台标签页
       └──────────────────────────────┘
```

- **不再存在** "前台 iframe 静音预加载 bgAudio" 这种双引擎并存状态。
- 任意时刻只有一个引擎在真正输出声音。
- `visibilitychange` 事件的职责大幅收窄：仅用于"视频展开时切到后台 → 自动收起"这一种情况，不再负责音频引擎切换。

### 4.2 直连/代理状态（新增，音频源内部的子状态）

```
[尝试直连 CDN] ──error──> [已 fallback，走服务器代理] ──error──> [判定播放失败，自动切下一首]
       │
    canplay/播放成功
       │
       v
[直连播放中]
```

- 每首新歌开始播放时，重置为"尝试直连"状态。
- fallback 只允许发生一次（避免代理也失败时无限重试卡死）。

### 4.3 并发/互斥说明

- 展开视频的过程中（bgAudio → iframe 切换的一瞬间），需要 `playSeq` 类似的序列号机制防止用户快速点击"展开/收起"导致状态错乱（复用现有 `playSeq` 思路，不必新增变量）。
- 切歌（上一首/下一首/自动连播）如果发生在视频展开状态下：直接销毁旧 iframe，用新歌的 bvid 重新创建 iframe（保持展开状态跟随播放的是"当前歌曲"，不因切歌自动收起）。

## 六、步骤 5：边界与失败场景

| 场景 | 表现 |
|------|------|
| CDN 直链被某些边缘节点拒绝跨域（403/无 CORS 头） | 自动 fallback 到代理地址，用户感知为"稍微多等一点点"，不会看到错误提示 |
| 直连和代理都失败（该 bvid 完全不可用） | 走现有"匹配失败"提示逻辑，toast 提示 + 可点"换源" |
| 直链有效期耗尽（约 114 分钟）导致中途播放失败 | 触发现有的 error/stalled 恢复机制，恢复时重新请求一次新的直链（而不是用旧的过期直链重试） |
| 用户展开视频后网络波动，iframe 加载慢 | 展开是用户主动操作，允许有加载等待态（沿用现有 iframe loading 体验），不做特殊处理 |
| 用户在视频展开状态下快速点击"收起→展开→收起" | 用序列号机制保证只有最后一次操作生效，避免出现多个 iframe 或声音重叠 |
| 服务器带宽/直连成功率的长期监控 | 建议在 `/api/play/direct-url` 或 `error` fallback 处打日志，记录 fallback 触发率，便于后续评估 CDN 直连的实际稳定性（不同网络环境的用户可能拿到不同 CDN 厂商节点） |

## 七、步骤 6：备选方案对比

| 方案 | 说明 | 取舍结论 |
|------|------|---------|
| **A. 服务器仍代理所有音频流**（现状不变） | 最稳定，但服务器带宽压力大，且无法获得"单一音频源"带来的架构简化收益 | 不选：与用户目标（减少家用带宽压力、简化架构）相悖 |
| **B. 服务器只发直链，客户端直连，无 fallback** | 实现最简单 | 不选：已实测发现直链有 CORS 依赖于具体 CDN 节点的 UA 校验，虽然目前测试宽松，但存在少数边缘节点策略不同的风险，无 fallback 会导致部分用户完全放不出声音 |
| **C. 直连优先 + 代理兜底**（本文档采用） | 兼顾带宽节省与稳定性，实现复杂度可接受 | **采用** |
| **D. 保留前后台双引擎，只是把"展开视频"从 visibilitychange 触发改成按钮触发** | 改动范围小 | 不选：治标不治本，双引擎切换的复杂度和音量不一致问题依然存在，没有解决用户提出的核心痛点 |

## 八、步骤 7：验收标准

1. **用户听歌，全程不点开视频面板**：无论浏览器标签页在前台还是切到后台/锁屏，声音不间断、不重叠、不产生额外延迟或卡顿。
2. **用户拖动音量滑块**：无论前后台状态如何，音量调节立即生效且效果一致（不会出现"前台调音量没反应，后台才生效"的情况）。
3. **用户开启"音量归一化"设置**：无论前后台，同一首歌的实际音量倍数保持一致（现状是仅后台生效，改造后全场景生效）。
4. **用户点击"展开"按钮**：视频画面出现，声音从点击前 `bgAudio` 播放到的进度秒数继续，无明显跳跃或重复播放的片段。
5. **用户点击"收起"按钮**：视频画面消失，声音从展开状态最后播放到的进度秒数继续（用 `bgAudio` 恢复），无声音中断感。
6. **播放进度条**：默认模式（`mode === 'bg'`）下可拖拽跳转播放位置，拖拽中实时预览时间文字、松手才真正 seek `bgAudio.currentTime`。展开视频模式（`mode === 'iframe'`）下进度条保持禁用（B 站官方未公开支持 postMessage 精确 seek），并显示提示"展开视频时请在视频内拖动进度条"。
7. **模拟直连失败场景**（如故意让接口返回错误直链）：播放能自动降级到代理地址正常播放，用户最多感知到轻微加载延迟，不会看到播放失败的错误提示。
8. **服务器带宽验证**：用浏览器开发者工具 Network 面板确认，正常播放场景下音频数据是从 B 站 CDN 域名直接下载，而不是从自建服务器域名下载（代理 fallback 触发时才是服务器域名）。
